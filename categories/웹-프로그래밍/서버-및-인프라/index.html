L4 스위치의 로드 밸런싱 과 웹 서버의 로드 밸런싱 차이

개요

  • 리버스 프록시의 로드 밸런싱과 L4 스위치를 통한 로드 밸런싱은 서로 다른 계층에서 동작하며, 각각 장단점이 있음
  • L4 스위치는 네트워크 계층에서 트래픽을 분산하는 반면,
  • 리버스 프록시는 애플리케이션 계층에서 트래픽을 분산
  • 그러면 왜 웹 서버를 통한 로드 밸런싱을 사용하는지, 그 이유를 비교하며 살펴보자

1. L4 스위치 로드 밸런싱 (전송 계층, 4계층)

  • L4 스위치는 OSI 모델의 4계층(전송 계층)에서 IP 주소와 포트 정보를 기반으로 트래픽을 분산
  • 이 방식은 TCP/UDP 연결 정보를 바탕으로 트래픽을 전달하기 때문에, 애플리케이션의 동작 방식에 대한 이해 없이 순수하게 네트워크 정보만을 가지고 로드 밸런싱을 수행

장점

  • 성능
    • L4 스위치는 네트워크 계층에서 동작하기 때문에 트래픽 처리 속도가 매우 빠름
    • 애플리케이션 계층까지 파악하지 않기 때문에 부하가 적음
  • 프로토콜 독립성: TCP/UDP와 같은 전송 계층 프로토콜만 처리하므로, HTTP 외에도 다양한 프로토콜을 지원
  • 안정성: 네트워크 장비의 특성상 안정적으로 트래픽을 분산할 수 있음

단점

  • 세부적인 트래픽 제어가 불가능
    • L4 스위치는 HTTP 요청의 헤더나 쿠키 등의 세부 정보에 접근할 수 없기 때문에, 애플리케이션 레벨에서 필요한 세밀한 로드 밸런싱(예: 세션 스티키네스, 특정 경로 기반 로드 밸런싱 등)이 불가능
  • 상태 기반 로드 밸런싱 불가능
    • L4 스위치에서는 사용자 세션이나 쿠키 같은 애플리케이션 상태를 인식할 수 없으므로, 세션 기반의 분산 처리가 어려움

2. 웹 서버를 통한 리버스 프록시 로드 밸런싱 (애플리케이션 계층, 7계층)

  • Nginx나 Apache 같은 웹 서버는 리버스 프록시로서 **애플리케이션 계층(7계층)**에서 트래픽을 분산함
  • HTTP 및 HTTPS 요청을 직접 처리하고, HTTP 헤더나 쿠키와 같은 애플리케이션 관련 데이터를 기반으로 로드 밸런싱을 수행할 수 있음

장점

  • 세부적인 트래픽 제어 가능

    • 웹 서버는 HTTP 요청의 세부적인 정보(예: URL 경로, 헤더, 쿠키, 파라미터)를 분석하여, 요청을 세밀하게 분산할 수 있음
    • 이를 통해 경로 기반(path-based) 또는 도메인 기반 로드 밸런싱이 가능합니다.

  • 세션 관리

    • 웹 서버는 세션 스티키네스(Session Stickiness)를 설정하여, 사용자의 요청을 같은 서버로 보내는 방식을 지원
    • 이를 통해 로그인 상태나 장바구니 같은 사용자 상태를 유지할 수 있음

  • SSL/TLS 처리

    • 웹 서버는 SSL/TLS 암호화를 처리할 수 있음
    • 웹 서버가 SSL 종료를 처리하고, 내부로는 HTTP 요청을 전달할 수 있음

  • 다양한 부하 분산 방식 라운드 로빈, 최소 연결 방식, IP 해싱 등 다양한 로드 밸런싱 전략을 적용할 수 있음

단점

  • 성능

    • L4 스위치보다 **처리 속도가 느릴 수 있음
    • 특히 대규모 트래픽 환경에서는 L4 스위치만큼의 성능을 보장하기 어려울 수 있음

  • 복잡성

    • L4 스위치에 비해 웹 서버는 더 많은 구성 요소와 설정을 요구
    • SSL 처리, 상태 관리 등의 기능을 위해 설정이 복잡할 수 있음

3. 웹 서버를 통한 로드 밸런싱을 사용하는 이유

1) 애플리케이션 레벨의 세밀한 제어

  • L4 스위치에서는 네트워크 트래픽만을 기반으로 단순하게 분산하지만, 웹 서버를 통한 리버스 프록시에서는 HTTP 헤더, 쿠키, URL 경로와 같은 정보를 기반으로 세밀한 트래픽 관리가 가능
  • 예를 들어, 웹 서버는 사용자의 세션을 유지하거나, 특정 API 경로에 따라 다른 서버로 트래픽을 분산하는 등의 작업을 수행할 수 있음

2) SSL/TLS 처리

  • 웹 서버는 SSL 종료를 처리할 수 있음
  • L4 스위치는 단순히 트래픽을 분산하지만, Nginx/Apache와 같은 웹 서버는 SSL/TLS 암호화를 처리한 후, 내부 네트워크에서는 HTTP로 통신할 수 있음
  • 이 방식은 보안과 성능을 모두 고려한 설계입니다.

3) 복잡한 로드 밸런싱 전략 지원

  • 웹 서버는 세션 기반 로드 밸런싱, 라운드 로빈이나 최소 연결 방식 등의 다양한 로드 밸런싱 알고리즘을 지원
  • 이를 통해 웹 애플리케이션에서 요구하는 다양한 로드 분산 정책을 구현할 수 있음

4) 유연성

  • 웹 서버는 모니터링 및 로깅 기능도 함께 제공하여, 애플리케이션 상태를 추적하고 트래픽 분산을 유연하게 조정할 수 있음
  • 또한, 새로운 서버 추가 시 더 간단하게 확장성을 제공할 수 있음

4. 결론

  • L4 스위치는 성능과 네트워크 기반의 단순한 트래픽 분산에 강점이 있음
  • 반면에 **리버스 프록시 웹 서버(Nginx, Apache)**는 애플리케이션 계층에서 HTTP 요청을 세밀하게 처리하고, SSL 종료와 세션 관리를 지원할 수 있는 강력한 기능을 제공
  • 따라서, 단순한 네트워크 트래픽 분산이 필요한 경우에는 L4 스위치를, HTTP 요청 기반의 세밀한 제어와 SSL 처리가 필요한 경우에는 리버스 프록시 웹 서버를 사용하는 것이 일반적

리버스 프록시

리버스 프록시(Reverse Proxy)란? 쉽게 이해하는 개념과 활용

  • 인터넷에서 웹 서버나 애플리케이션 서버의 트래픽을 효율적으로 관리하고 보안을 강화하는 중요한 기술 중 하나가 리버스 프록시(Reverse Proxy)
  • 복잡한 웹 시스템에서 자주 사용되는 핵심 기술이지만, 처음 접하는 사람에게는 다소 생소할 수 있기에, 이 포스팅에서는 리버스 프록시가 무엇인지, 어떻게 동작하는지, 그리고 이를 활용하는 주요 이점들을 쉽게 설명

1. 리버스 프록시(Reverse Proxy)란?

  • 클라이언트(사용자)가 요청한 트래픽을 여러 서버로 분배하거나 필터링하는 중간 역할을 하는 서버

  • 쉽게 말해, 클라이언트가 웹사이트에 요청을 보내면, 리버스 프록시가 이 요청을 대신 받아 내부 서버(웹 서버, 애플리케이션 서버)에 전달하고, 응답을 다시 클라이언트에게 돌려줌

  • 리버스 프록시는 서버 앞에 위치하여 클라이언트와 서버 간의 모든 트래픽을 중재하는 역할을 하며, 서버의 IP 주소나 내부 구조를 클라이언트에게 노출하지 않고 보호할 수 있음

리버스 프록시의 동작 구조

  1. 클라이언트가 특정 웹사이트에 접속하려고 요청을 보냄
  2. 요청은 리버스 프록시 서버로 전달
  3. 리버스 프록시 서버는 내부 서버 중 하나(웹 서버나 애플리케이션 서버)로 요청을 전달
  4. 내부 서버는 요청을 처리하고, 그 결과를 리버스 프록시 서버에 돌려줌
  5. 리버스 프록시 서버는 클라이언트에게 응답을 보냄

예시

  • 클라이언트가 www.example.com에 접속하려고 할 때, 클라이언트는 실제 서버의 IP 주소나 구조를 모르고, 리버스 프록시 서버가 요청을 대신 받아 처리한 후 클라이언트에게 결과를 전달

2. 리버스 프록시의 주요 역할

리버스 프록시는 단순히 요청을 전달하는 역할뿐만 아니라, 다양한 기능을 통해 웹 서비스의 성능을 향상시키고, 보안 및 확장성을 개선할 수 있음. 여기서 리버스 프록시의 주요 기능과 장점을 살펴보자

1) 로드 밸런싱 (Load Balancing)

  • 리버스 프록시는 로드 밸런서의 역할을 할 수 있음
  • 여러 대의 서버에 클라이언트 요청을 분배함으로써 트래픽을 균등하게 나누고, 한 서버에 과부하가 걸리지 않도록 도와줌
  • 예시
    • 리버스 프록시 서버가 3개의 웹 서버(A, B, C) 중 한 서버에만 트래픽이 몰리지 않게 분산하여 트래픽을 처리
    • 이로 인해 서버의 부하를 줄이고, 안정적인 서비스 제공이 가능

2) 보안 강화

  • 리버스 프록시는 서버의 IP 주소와 내부 구조를 감추는 역할을 하기 때문에, 외부 공격자가 내부 서버에 직접 접근하기 어려움
  • 또한 SSL 인증서를 리버스 프록시 서버에 설치하여, HTTPS 요청을 처리하고 내부 서버와의 통신은 HTTP로 설정하여 보안을 강화할 수 있음
  • 방화벽 역할: 리버스 프록시는 방화벽처럼 동작하여, 원치 않는 요청이나 악의적인 공격을 차단할 수 있습니다.

3) 캐싱(Cache)

  • 리버스 프록시는 정적 콘텐츠(이미지, CSS, JavaScript 파일 등)를 캐시하여 클라이언트가 동일한 요청을 보낼 때 더 빠르게 응답할 수 있음
  • 이를 통해 서버 부하를 줄이고, 응답 시간을 개선할 수 있음
  • 예시
    • 클라이언트가 자주 요청하는 이미지 파일이나 CSS 파일을 리버스 프록시 서버가 캐시하여, 서버에 재요청하지 않고 캐시된 데이터를 클라이언트에 반환
    • 이렇게 하면 속도가 빨라지고 서버 리소스가 절약됨

4) SSL 종료(SSL Termination)

  • 리버스 프록시는 SSL 종료 기능을 통해 HTTPS 요청을 처리할 수 있음
  • 리버스 프록시가 클라이언트로부터 HTTPS 요청을 받아 SSL/TLS 암호화를 해제한 후, 내부 서버에는 HTTP 요청으로 전달할 수 있음, 이 방법을 통해 내부 서버는 복잡한 SSL 처리를 하지 않고도 안전하게 운영될 수 있음
  • 예시
    • Nginx 리버스 프록시 서버가 HTTPS 요청을 처리하고, 내부 서버는 HTTP로만 통신하게 하여 부하를 줄이면서 보안성을 유지

5) 다중 서버 통합 (Server Aggregation)

  • 리버스 프록시는 여러 서버의 응답을 통합하여 클라이언트에게 전달할 수 있음
  • 예를 들어, 사용자가 서로 다른 서비스에 대한 요청을 보낼 때, 리버스 프록시는 각 요청을 적절한 서버로 분배하고, 그 결과를 하나의 응답으로 통합해 반환할 수 있음
  • 예시
    • 리버스 프록시 서버가 API 서버, 이미지 서버, 데이터베이스 서버 등 여러 서버의 응답을 통합해 클라이언트에게 하나의 웹 페이지로 반환

3. 리버스 프록시의 활용 예시

리버스 프록시는 여러 환경에서 유용하게 사용될 수 있으며, 그 중 몇 가지 대표적인 예시는 다음과 같음

1) Nginx를 사용한 리버스 프록시 설정

  • Nginx는 가장 널리 사용되는 리버스 프록시 서버 중 하나로, 정적 파일 제공, 로드 밸런싱, SSL 처리 등 다양한 기능을 제공

위 설정은 Nginx가 클라이언트 요청을 받아 localhost:5000에서 실행 중인 내부 서버로 요청을 전달하는 리버스 프록시 역할을 하게됨

2) 클라우드 환경에서의 리버스 프록시 사용

  • 리버스 프록시는 클라우드 환경에서 다중 서버 관리와 확장성을 위해 자주 사용됨
  • 특히 AWS, GCP와 같은 클라우드 플랫폼에서는 로드 밸런싱, 보안 및 자동 확장을 위해 리버스 프록시를 사용하여 트래픽을 효율적으로 관리

4. 리버스 프록시와 포워드 프록시(Forward Proxy)의 차이점

리버스 프록시와 포워드 프록시(Forward Proxy)는 개념이 비슷하지만, 역할과 동작 방식이 다름

  • 리버스 프록시는 클라이언트의 요청을 받아 내부 서버로 전달하고, 서버의 응답을 다시 클라이언트에게 돌려주는 역할을 함, 주로 서버 보호 및 트래픽 관리에 사용됩니다.
  • 포워드 프록시는 클라이언트가 직접 요청할 수 없는 외부 서버로의 요청을 대신 보내주는 역할을 함, 주로 사용자의 익명성을 보호하거나 차단된 콘텐츠에 접근할 때 사용됨
구분 리버스 프록시(Reverse Proxy) 포워드 프록시(Forward Proxy)
동작 대상 서버 보호 및 내부 서버로의 요청 중계 클라이언트 보호 및 외부 서버로의 요청 중계
목적 서버의 로드 밸런싱, 보안 강화, 캐싱 처리 클라이언트의 익명성 보호, 네트워크 차단 우회
사용 예시 Nginx, Apache, AWS ELB VPN, 회사나 학교에서 특정 웹사이트 차단 우회

5. 리버스 프록시의 장점 요약

  • 보안 강화: 서버 IP 및 내부 구조 보호, SSL 종료 제공
  • 로드 밸런싱: 트래픽 분산으로 성능 및 안정성 향상
  • 캐싱 제공: 정적 파일 캐시를 통해 빠른 응답 속도 제공
  • SSL 처리: HTTPS 요청을 처리하여 보안성 강화
  • 서버 확장성: 여러 서버를 통합하고 분산하여 더 많은 트래픽 처리 가능

결론

  • 리버스 프록시(Reverse Proxy)는 웹 서비스의 성능과 보안을 크게 향상시킬 수 있는 강력한 기술
  • 특히, 트래픽을 관리하고 보안을 강화하는 데 탁월한 역할을 함
  • Nginx와 같은 리버스 프록시 서버를 올바르게 설정하면, 서비스의 확장성과 안정성을 확보하면서도, 더 나은 사용자 경험을 제공할 수 있음
  • 리버스 프록시는 로드 밸런싱, 캐싱, 보안, SSL 처리 등 다양한 기능을 제공하므로, 대규모 웹 서비스나 클라우드 환경에서 필수적인 도구로 활용되고 있음

웹서버와 WAS 차이

1. 웹 서버 (Web Server)

  • 기본 역할

    • 웹 서버는 정적 콘텐츠(HTML, CSS, JavaScript, 이미지 파일 등)를 클라이언트(주로 브라우저)에 제공하는 역할을 함
    • 정적 콘텐츠: 변경되지 않는 파일들을 그대로 클라이언트에 전송
    • 대표적인 웹 서버로는 Nginx, Apache HTTP Server가 있음

  • 동작 방식

    • 클라이언트(브라우저)가 요청을 보내면, 웹 서버는 요청된 리소스(파일)를 찾아서 클라이언트에게 전달
    • 예를 들어, 사용자가 웹 페이지를 요청하면, 웹 서버는 해당 페이지의 HTML 파일을 클라이언트에 전송

  • 동적 콘텐츠 처리

    • 웹 서버는 동적 콘텐츠(사용자 요청에 따라 실시간으로 생성되는 데이터)는 직접 처리하지 못함
    • 동적 요청은 WAS나 애플리케이션 서버로 전달되어야 함
    • 동적 콘텐츠는 데이터베이스와 상호작용하거나 사용자 입력에 따라 변하는 데이터를 생성하는 작업을 의미

  • 주요 특징

    • 정적 파일(HTML, CSS, 이미지) 제공
    • HTTP 요청 및 응답을 처리
    • 동적 요청은 직접 처리하지 않고, WAS로 전달

2. WAS (Web Application Server)

  • 기본 역할

    • WAS는 동적 콘텐츠를 처리하는 서버
    • 웹 서버가 처리하지 못하는 동적 요청(데이터베이스 연결, 비즈니스 로직 처리 등)을 처리하고, 동적 웹 애플리케이션을 실행하는 역할을 함
    • 동적 콘텐츠: 사용자의 요청에 따라 실시간으로 생성되는 콘텐츠(예: 데이터베이스 조회, 사용자 세션 처리, 트랜잭션 등)를 생성하여 클라이언트에게 전달

  • 동작 방식

    • 클라이언트 요청 중에서 동적인 웹 애플리케이션 처리 요청이 있을 경우, WAS는 이를 처리하고 그 결과를 웹 서버를 통해 클라이언트에게 전달
    • 예를 들어, 사용자가 로그인 요청을 하면, WAS는 데이터베이스와 통신하여 해당 사용자의 정보를 확인하고 그 결과를 클라이언트에게 전달

  • 예시

    • Tomcat, JBoss, WebLogic 등이 대표적인 WAS
    • 이러한 WAS는 서블릿 컨테이너를 사용하여 자바 기반의 웹 애플리케이션을 구동하거나, PHP, Python, Ruby와 같은 언어로 작성된 웹 애플리케이션을 처리

  • 주요 특징

    • 비즈니스 로직을 처리하는 역할
    • 동적 웹 애플리케이션 실행
    • 세션 관리, 트랜잭션 처리, 데이터베이스와 상호작용
    • WAS는 자체적으로 웹 서버 기능을 포함할 수 있으므로, 정적 콘텐츠도 처리 가능하지만, 보통은 웹 서버(Nginx, Apache)와 함께 사용

3. 웹 서버와 WAS의 차이

구분 웹 서버 (Web Server) WAS (Web Application Server)
기본 역할 정적 콘텐츠(HTML, CSS, 이미지 등)를 제공 동적 콘텐츠(데이터베이스, 비즈니스 로직)를 처리
정적/동적 처리 정적 파일을 처리 (동적 파일은 WAS로 전달) 동적 웹 애플리케이션 실행 및 비즈니스 로직 처리
예시 Nginx, Apache HTTP Server Tomcat, JBoss, WebLogic
기술 스택 HTTP 프로토콜 처리, 정적 파일 서빙 서블릿, JSP, Spring 등 웹 애플리케이션 프레임워크 구동
주요 사용 사례 정적 웹 사이트, 파일 서버 동적 웹 애플리케이션 서버 (로그인 처리, 데이터베이스 접근 등)
부가 기능 로드 밸런싱, 캐싱 등 트랜잭션 관리, 세션 관리, 데이터베이스 연결

4. 웹 서버와 WAS의 협업

  • 일반적으로 웹 서버(Nginx, Apache)와 WAS(Tomcat 등)는 함께 사용됨

  • 웹 서버는 클라이언트로부터 모든 요청을 받아서, 정적 콘텐츠는 직접 처리하고, 동적 콘텐츠 요청은 WAS에 전달

  • 이와 같은 분리된 구조는 성능과 보안 면에서 더 효율적

    • 웹 서버는 정적 리소스를 빠르게 제공하고,

  • WAS는 복잡한 비즈니스 로직을 처리하며 데이터베이스와 상호작용

부록 1. 궁금했던 점

Spring Boot와 같은 현대적인 웹 프레임워크에서는 정적 페이지와 동적 페이지가 모두 함께 작성되며, 웹 서버를 별도로 사용하지 않고도 서버(Tomcat WAS 등)에서 처리할 수 있는데 웹 서버가 왜 필요할까?

Spring Boot와 WAS (Tomcat 등)의 기본 동작

  • Spring Boot는 기본적으로 내장형 WAS(예: Tomcat, Jetty)를 포함하고 있음

  • 따라서 별도의 외부 WAS를 설정하지 않고도 정적 페이지(HTML, CSS, 이미지 등)와 동적 페이지(Java 기반의 서버 로직)를 모두 처리할 수 있음

    • 정적 콘텐츠: Spring Boot 애플리케이션 내에서 src/main/resources/static 디렉터리에 정적 파일을 두면, Spring Boot 내장 WAS가 이를 처리

    • 동적 콘텐츠: 동적 요청은 Java 코드로 작성된 비즈니스 로직에 따라 처리되며, Spring MVC 패턴으로 요청-응답 흐름을 처리

따라서, Spring Boot 애플리케이션 하나만으로도 정적/동적 리소스 처리 및 배포가 모두 가능

그렇다면, 왜 웹 서버(Nginx, Apache)가 필요할까?

일반적으로 Spring Boot와 같은 애플리케이션 서버만으로도 배포가 가능하지만, 웹 서버를 추가로 사용하면 몇 가지 장점이 있기 때문에 대규모 환경이나 프로덕션에서는 여전히 웹 서버를 사용하는 경우가 많음

1) 정적 파일 제공의 효율성

  • Nginx나 Apache 같은 웹 서버는 정적 파일(HTML, CSS, 이미지 등)을 제공하는 데 매우 최적화되어 있음
  • 웹 서버는 정적 파일을 매우 빠르게 처리할 수 있으며, 정적 파일에 대해 캐싱, 압축 등의 기능을 제공하여 성능을 향상시킬 수 있는 반면
  • Spring Boot 내장 WAS는 이러한 정적 파일 서빙에 최적화되어 있지 않기 때문에, 대규모 트래픽 상황에서 웹 서버가 정적 콘텐츠를 처리하면 부하를 줄일 수 있음

2) 로드 밸런싱 및 리버스 프록시

  • Nginx나 Apache는 리버스 프록시로 설정하여, 여러 Spring Boot 애플리케이션 인스턴스 간에 로드 밸런싱을 할 수 있음

    • 이렇게 하면 트래픽을 균등하게 분배할 수 있어 서버 과부하를 방지할 수 있음

  • 예를 들어, Nginx가 클라이언트 요청을 받아 여러 WAS 인스턴스(Spring Boot)로 요청을 분산시키는 구조로 만들면, 하나의 서버가 과부하에 걸리는 것을 방지할 수 있음

3) 보안 (SSL/TLS 처리)

  • SSL 인증서를 설정하여 HTTPS를 적용할 때는 Nginx나 Apache와 같은 웹 서버를 사용하는 것이 일반적
  • 웹 서버는 리버스 프록시 역할을 하며, SSL/TLS를 통한 암호화된 통신(HTTPS)을 처리하는 데 있어 효율적이고 편리한 설정을 제공
  • 리버스 프록시 역할을 하는 웹 서버는 클라이언트로부터 받은 HTTPS 요청을 처리하고, 내부 네트워크에서는 HTTP로 WAS에 요청을 전달
  • 이렇게 구성하면 외부 통신에서는 SSL/TLS 암호화를 통해 보안성을 유지하면서도, 내부 통신은 복잡한 암호화 작업 없이 효율적으로 처리할 수 있음

4) 확장성 및 장애 대응

  • 웹 서버는 요청을 다수의 애플리케이션 서버(WAS)로 분산시킬 수 있음

    • 여러 Spring Boot 인스턴스를 실행하여 마치 하나의 서버처럼 운영할 수 있기 때문에, 확장성을 확보할 수 있음

  • 예를 들어, Nginx가 3개의 Spring Boot 서버에 트래픽을 균등하게 나눠주면, 한 서버에 문제가 생겨도 다른 서버에서 요청을 처리할 수 있어 고가용성을 유지할 수 있음

5) 캐싱 및 성능 최적화

  • Nginx나 Apache는 정적 콘텐츠에 대해 캐싱을 제공하여 성능을 향상시킬 수 있음

    • Spring Boot는 기본적으로 캐싱 기능이 없으므로, 이러한 웹 서버를 추가하면 페이지 로딩 속도를 개선할 수 있음

  • 웹 서버가 캐시된 정적 리소스를 제공하면, 애플리케이션 서버(WAS)의 부하가 줄어들어 더 많은 동적 요청을 처리할 수 있음

웹 서버와 WAS가 함께 사용되는 아키텍처

프로덕션 환경에서는 일반적으로 **Nginx/Apache(웹 서버)**가 리버스 프록시로 동작하고, **Spring Boot(내장 WAS)**가 실제 애플리케이션 로직을 처리하는 구조로 구성

  • 웹 서버(Nginx): 정적 파일 서빙, 로드 밸런싱, SSL 처리, 리버스 프록시 역할
  • WAS(Spring Boot 내장 Tomcat): 동적 요청 처리, 데이터베이스 연동, 비즈니스 로직 처리

이와 같은 분리된 아키텍처를 통해 성능과 확장성을 극대화할 수 있음
