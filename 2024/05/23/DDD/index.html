DDD 를 위한 아키텍처 소개

1. Layered Architecture

Presentation Layer(표현 계층)

  • 사용자의 요청을 받아 응용 영역에 전달하고 응용 영역의 처리 결과를 다시 사용자에게 보여주는 역할
  • 사용자는 소프트웨어를 사용하는 사람일 뿐만 아니라 REST API 를 호출하는 외부 시스템일 수 있음
  • 스프링 MVC 프레임워크가 표현 영역을 위한 기술
  • HTTP 요청을 응용 영역이 필요로 하는 형식으로 변환해서 응용 영역에 전달하고 응용 영역의 응답을 HTTP 응답으로 변환하여 전송
    • ex) 웹 브라우저가 HTTP 요청 파라미터로 전송한 데이터를 응용 서비스가 요구하는 형식의 객체 타입으로 변환해서 전달하고, 응용 서비스가 리턴한 결과를 JSON 형식으로 변환해서 HTTP 응답으로 웹 브라우저에 전송

Application Layer(응용 계층)

  • 사용자가 요청한 기능을 실행
  • 업무 로직을 직접 구현하지 않으며 도메인 계층을 조합해서 기능을 실행
    • 즉, 응용 계층은 도메인 모델을 이용해서 사용자에게 제공할 기능을 구현하며 실제 도메인 로직 구현은 도메인 모델에 위임

Domain Layer(모델 계층)

  • 시스템이 제공할 도메인 규칙을 구현
  • 도메인 모델을 구현
  • 도메인의 핵심 로직을 구현
    • 주문 도메인의 경우 배송지 변경, 결제완료, 주문 총액 계산 과 같은 핵심 로직을 도메인 모델에서 구현

Infrastructure Layer(인프라 계층)

  • DB나 메세징 시스템과 같은 외부 시스템과 연동을 처리
    • RDBMS 연동 처리
    • Kafka 등 메시징 큐에 메세지를 전송하거나 수신하는 기능을 구현
    • MongoDB나 Redsis와 데이터 연동을 처리
    • HTTP 클라이언트를 이용해서 REST API를 호출하는 것도 처리
    • SMTP를 이용한 메일 발송 기능을 구현
  • 인프라스트럭쳐 영역은 논리적인 개념을 표현하기보다는 실제 구현을 다룸

도메인 주도 설계란?

  • 도메인이란

    • 소프트웨어로 해결하고자하는 문제 영역

      • ex) 온라인 서점 소프트웨어 (책을 판매하기 위한 상품 조회, 구매, 결제, 배송 추적 등의 기능)

  • 하위 도메인이란

    • 하나의 도메인은 다시 하위 도메인(들)으로 나눌 수 있음
    • ex) 온라인 서점 소프트웨어의 책을 판매하기 위한 상품 조회, 구매, 결제, 배송 추적 등의 기능

  • 하위 도메인 유형

    • 핵심(Core)
      • 회사가 경쟁업체와 다르게 수행하고 있는 것
      • 복잡성 높지만 BIZ 경쟁력 제공
      • 우리 회사의 서비스의 핵심 기술
      • 구글의 검색 순위 알고리즘, 우버의 방향에 따른 손님 매칭 서비스 등
    • 일반(Generic)
      • 모든 회사가 같은 방식으로 수행하는 비지니스 활동
      • 복잡하고 구현하기 어려우나 경쟁력을 제공하지는 않음. 알려진 영역
      • 인증, 권한 부여
      • Spring Security
    • 지원(Supporting)
      • 회사 비지니스 지원 활동
      • 기능 간단, 어떠한 경쟁 우위 제공하지 않음
      • CRUD, ETL 위주의 기능
      • 게시판, 공지사항 등

    https://github.com/shchoice/shchoice.github.io/assets/100276387/7a21a8d0-60da-4d88-8a2b-9194e406fd2d

  • 하위 도메인 유형에 따른 구현 방법

    • 핵심(Core)
      • 사내에서 구현, 조직의 핵심 숙련 인재 할당
      • 자주 지속적으로 변경 예상되므로 가장 진보된 엔지니어링 기술 적용
    • 일반(Generic)
      • 이미 만들어진 제품, 오픈소스 솔루션
    • 지원(Supporting)
      • 사내에서 구현하지 않음
      • 정교한 디자인 패턴이나 고급 엔지니어링 기술이 필요 않음
      • 새로운 이재 양성을 위한 연습 기회 제공(외주나 신입사원)

    https://github.com/shchoice/shchoice.github.io/assets/100276387/2848f46b-7c02-4671-8935-944136839dcf

  • 멘탈모델과 도메인 지식

    • 도메인 전문가가 문제를 생각하는 방식, 즉 멘탈 모델
    • 문제를 해결하기 위해서는 문제를 이해해야 함
    • 기존의 과정
      • OOAD(Object-Oriented Analysis Design), UML 을 가지고 만듬
    • 도메인 주도 설계에서의 과정
      • 유비쿼터스 언어 : 도메인 지식을 변환하는 대신 도메인을 설명하기 위한 단일화된 체계인

  • 유비쿼터스 언어(Ubiquitous Language)

    • 정확하고 일관성 있는 용어를 사용, 모호한 용어 사용X
    • 동의어 X
    • 예제
      • SMTP 이런 단어를 사용하면 안됨
      • 정책(X) → 보험 계약 정책 등 구체적으로 사용을 해야함
      • 사용자(X) → 방문자, 계정

  • 도메인 모델

    • 효과적인 모델은 그 목적을 달성하는 데 필요한 세부사항만 포함
    • 실 세계의 복사본이 아니라 문제를 해결하려는 의도가 있으며, 그 목적에 필요한 정보만 제공해야 함

  • 도메인 모델링

    • 유비쿼터스 언어로 사실상 비지니스 도메인 모델을 구축하는 것
    • 명사와 행위 파악

  • Bounded Context

    • 모델의 경계
    • Bounded Context는 유비쿼터스 일관성이 유지되는 경계
    • 유비쿼터스 언어의 용어, 원칙, 비지니스 규칙은 해당 바운디드 컨텍스트 내에서만 일관성이 있음

  • Context Mapping

    • BC는 서로 독립적으로 발전할 수 있지만 상호작용해야함, 즉 BC는 접점이 있으며 이를 contract라 함
