파이썬 데이터 모델

파이썬 데이터 모델이란 무엇인가

  • 파이썬 객체가 가지는 내부 동작 및 상호작용을 정의하는 일련의 프로토콜과 규칙
  • 객체 간 상호작용을 관리하며, 파이썬이 지원하는 많은 기능과 특징을 가능하게 함
  • 일종의 프레임워크

파이썬 데이터 모델은 다음과 같은 프로토콜로 구성됨

  • 시퀀스(sequence) 프로토콜 : 시퀀스 형식의 객체(리스트, 튜플, 문자열) 에 대한 연산을 정의

    • len() : 시퀀스의 길이를 반환
    • getitem(): 시퀀스의 인덱스를 기반으로 요소(index)를 반환

  • 매핑(mapping) 프로토콜: 매핑 형식의 객체(딕셔너리) 등에 대한 연산을 정의

    • getitem() : 매핑 객체의 키를 기반으로 값을 반환
    • setitem() : 매핑 객체의 키를 기반으로 값을 설정
    • delitem() : 매핑 객체의 키를 기반으로 값을 삭제
    • len() : 매핑 객체의 길이를 반환
    • keys, values, items 메서드는 각각 매핑 객체의 키, 값, 키-값 쌍에 대한 리스트를 반환

  • 반복(순회) 가능한(iterable) 프로토콜 : 객체를 반복할 수 있는(iterable) 객체에 대한 연산을 정의

    • iter() : iterator 객체인 iter(self.items)를 반환

  • 반복자(nextable) 프로토콜 : 객체를 반복(iteration)할 수 있는 객체에 대한 연산을 정의

    • next() : 요소를 하나씩 반환하다가, 모든 요소를 반환하면 StopEteration 예외를 발생시켜 반복을 멈춤

  • 호출 가능한(callable) 프로토콜 : 객체를 함수처럼 호출(callable) 할 수 있는 객체에 대한 연산을 정의

    • call() : 객체를 함수처럼 호출할 때 실행됨

  • 속성(attribute) 프로토콜 : 객체의 속성(attribute)을 접근할 수 있는 객체에 대한 연산을 정의

    • getattr() : 객체의 속성을 가져오려고 할 때 실행됨

  • 컨텍스트 관리(Context management protocol) 프로토콜 : with 문과 함께 사용되는 콘텍스트 관리 객체에 대한 연산을 정의

    • enter() : 컨텍스트 매니저 객체가 생성될 때 호출되어 컨텍스트 진입 시 처리를 수행하는 메서드

    • exit

      ()

      • 컨텍스트에서 빠져나올 때 호출되어 컨텍스트 종료 시 처리를 수행하는 메서드

      • 컨텍스트 내에서 예외가 발생하거나, 정상적으로 종료되었을 때 모두 호출됨

      • exit

        ()는 세개의 인자를 받음

        • exc_type : 발생한 예외의 타입, 예외가 발생하지 않은 경우 None이 전달됨
        • exc_value : 발생한 예외의 인스턴스, 예외가 발생하지 않은 경우 None이 전달됨
        • traceback : 발생한 예외의 추적 정보, 예외가 발생하지 않은 경우 None이 전달됨

      • 예외를 처리하고 예외를 처리한 경우 True를 반환, 예외를 처리하지 않은 경우 False를 반환, 반환 값은 예외 처리 이후의 실행 흐름을 결정

  • 문자열 표현(String representation) 프로토콜 : 객체를 문자열로 표현하는 방법을 정의

    • str() : str함수로 호출되고, 일반적으로 객체의 간단한 문자열을 반환
    • repr() : repr함수로 표출되고, 공식적인 문자열을 표현을 반환

  • 복제 가능한(copyable) 프로토콜 : 객체를 복제할 수 있는 메서드를 정의

    • copy()

      • 얕은 복사(shallow copy)를 수행하는 메서드
      • 객체의 참조값을 복사하므로, 원본 객체와 복사본 객체는 동일한 객체를 참조
      • 따라서 원본 객체의 속성 값이 변경되면 복사본 객체의 속성 값도 변경됨

    • deepcopy()

      • 깊은 복사(deep copy)를 수행하는 메서드
      • 객체의 값을 복사하므로, 원본 객체와 복사본 객체는 서로 독립된 객체
      • 따라서 원본 객체의 속성 값이 변경되어도 복사본 객체의 속성 값은 변경되지 않음

  • 비교 가능한(comparable) 프로토콜 : 객체를 비교할 수 있는 메서드

    • eq(self**,** other) : self == other 를 구현, 같은 값인 경우 True를 반환
    • ne(self**,** other) : self != other 를 구현, 같은 값이 아닌 경우 True를 반환
    • lt(self**,** other) : self < other 를 구현, self가 other보다 작은 경우 True를 반환
    • le(self**,** other) self <= other 를 구현, self가 other보다 작거나 같은 경우 True를 반환
    • gt(self**,** other) : self > other 를 구현, self가 other보다 큰 경우 True를 반환
    • ge(self**,** other) : self >= other 를 구현, self가 other보다 크거나 같은 경우 True를 반환

  • 해시 가능한(hashable) 프로토콜

    • 객체를 해시 테이블의 키로 사용할 수 있는지 여부를 판단하는 프로토콜
    • 이 프로토콜을 구현하려면 객체의 해시값(hash value)을 반환하는 hash() 메서드와 객체의 동등성(equality)을 비교하는 eq() 메서드를 구현해야함
    • 해시 가능한 객체는 불변성(immutable)을 갖으며 같은 값의 객체는 항상 같은 해시값을 갖도록 구현되어야 함
    • 해시 가능한 객체는 dictionary와 set에서 키로 사용될 수 있음
    • hash() : 객체의 해시값(hash value)을 반환
    • eq() : 메서드와 객체의 동등성(equality)을 비교

  • 해시 가능한(hashable) 프로토콜

    • 시퀀스 프로토콜과 같은 기능을 수행하지만, 인덱스를 통해 객체에 접근하는 방식을 강제
    • getitem() : 인덱스로 객체에 접근할 수 있도록 함

  • 슬라이싱 가능한(sliceable) 프로토콜 : 객체를 슬라이싱(slicing)할 수 있는 메서드(시퀀스 객체의 일부를 추출할 때 사용됨)

    • getitem

      ()

      • 이 메소드는 슬라이싱 연산자에 의해 호출됨
      • 시작 인덱스(start index), 끝 인덱스(end index), 그리고 간격(step)을 인자로 받아 시퀀스 객체의 일부를 추출하여 반환

  • empty 프로토콜 : 객체가 비어 있는지 여부를 알 수 있음

    • bool() 또는 len() 메소드를 구현해야함
    • bool() : 객체가 비어 있으면 False를 반환하고, 그렇지 않으면 True를 반환
    • len() : 객체의 길이(크기)를 반환, 객체가 비어 있으면 0을 반환 , 그렇지 않으면 객체의 길이를 반환

  • 길이(length) 프로토콜 : 객체의 길이를 반환

    • len() : 리스트의 길이를 반환

  • 존재성(test for membership) 프로토콜 : in 연산자를 사용하여 객체 내에 특정 값이 존재하는지 확인하는 데 사용됨

    • contains() : 인자로 전달된 값이 객체 내에 존재하는지 여부를 불리언 값으로 반환

  • 산술 연산 프로토콜 : 산술 연산(덧셈, 뺄셈, 곱셈, 나눗셈 등)할 수 있도록 정의하는 프로토콜

    • add()
    • sub()
    • mul()
    • truediv()

  • 비트 연산 프로토콜 : 객체 간 비트 연산을 정의하는 프로토콜

    • 비트 연산자(&, |, ^, ~, <<, >>)를 사용하여 비트 연산을 수행

매직 메서드

매직 메서드란

  • 파이썬 데이터 모델에서 프로토콜을 구현하기 위한 특별한 메서드
  • 앞서 보았던 프로토콜로 포함이 되는 __로 시작하고 __로 끝나는 이름을 갖음(ex, len)
  • 객체 간의 상호작용을 정의하고 객체를 기능적으로 확장하는 데 사용됨

매직 메서드는 어떻게 사용하나

  • my_object.len() 으로 직접 사용하지 않고 len(my_object) 형태로 호출
  • list, str, bytearray 등과 같은 내장 자료형의 경우 파이썬 인터프리터는 손 쉬운 방법을 선택
    • CPython의 경우 len() 메서드는 메모리에 있는 모든 가변 크기 내장 객체를 나타내는 PyVarObject C 구조체의 ob_size 필드의 값을 반환(이 방법이 메서드를 호출하는 방법보다 빠름)
  • 일반적으로 사용자 코드에서 특별 메서드를 직접 호출하는 경우는 그리 많지 않음
    • 사용자 코드에서 특별 메서드를 자주 호출하는 경우는 init() 메서드가 유일
  • 특별 메서드를 호출해야 하는 경우에는 일반적으로 len(), iter(), str() 등 관련된 내장 함수를 호출하는 것이 좋음 →내장 데이터형의 경우 특별 메서드를 호출하지 않는 경우도 있으며 메서드 호출보다 빠름

매직 메서드의 장점

  • 파이썬에서 매우 유용하게 사용되며 객체 지향 프로그래밍의 일관성을 유지하는 데 도움이 됨

  • 내장 함수나 연산자가 사용되는 상황에서 클래스 객체에 대한 사용자 정의 동작을 정의할 수 있음

  • 매직 메서드를 구현하면 이를 활용한 기능 구현을 라이브러리 형태로 공유할 수 있음

    • len() : len
    • for 루프 등 : iter, next

  • 파이썬 표준 라이브러리에서 제공하는 풍부한 기능을 별도로 구현할 필요 없이 바로 사용

    • random의 choice(), reversed(), sorted() 함수 등

    • 리스트와 같은 시퀀스 자료형

내장 함수(Built-in function)

  • 파이썬에서 이미 정의되어 있는 함수로, 별도의 정의 없이 바로 사용 가능

  • print(), len(), type(), range(), sum(), min(), max() 등이 내장 함수

  • 매직 메서드와 다름

매직 메소드와 내장 함수의 다른점

  • 매직 메소드는 클래스에서 정의된 특별한 이름을 가진 메서드fh 해당 클래스의 객체에 대해 호출될 때 자동으로 호출되어 객체의 동작을 정의
  • 매직 메소드는 클래스의 일부이며, 해당 클래스에서 정의된 메서드이기에 내장 함수와는 별개의 개념

결론

  • 파이썬 데이터 모델은 프로토콜을 구현함으로써 객체 간의 상호 작용을 쉽게 구현할 수 있게 함
  • 프로토콜은 인터페이스를 정의하는데 사용되는 개념이며, 매직 메서드는 이러한 인터페이스를 구현하는 데 사용되는 메서드
  • 리스트와 튜플은 시퀀스 프로토콜을 구현하므로, 인덱싱, 슬라이싱, 정렬 등과 같은 연산을 쉽게 수행할 수 있음
  • 객체의 속성을 직접 접근할 수 있는 속성 접근 프로토콜을 구현함으로써, 객체 간의 데이터 공유 및 상호작용을 구현할 수 있음

파이썬 데이터 모델

  • 일종의 프레임워크로써, 파이썬을 설명하는 것
  • 시퀀스, 반복자, 함수, 클래스, 콘텍스트 관리자 등 언어 자체의 구성단위에 대한 인터페이스를 공식적으로 정의

  • 매직 메소드는 custom 객체를 구현할 때 다음과 같은 기본적인 언어 구조체를 구현하고 지원하고 함께 사용할 수 있게 해줌

    • 함수 및 메소드 호출
    • 객체 생성 및 제거
    • 문자열 표현 및 포맷
    • 블록 등 콘텍스트 관리
  • 스페셜 메소드의 장점

      • 연속 데이터를 저장하는 Sequence 자료형에서 요소마다 같은 기능을 적용할 때 사용

      • 최근에는 lambda() 나 map() 함수를 프로그램 개발에 사용하는 것을 권장하지 않음

함수형 프로그래밍

업무를 수행하면서 함수형 프로그래밍이 많이 대두되고 있으며, 새로 함께하는 팀장님께서는 함수형 프로그래밍으로 코드를 구현하는 것에 대해서 많이 말씀을 하고 계셨다. 학교에서 배웠던 것에 익숙한 나에게 함수형 프로그래밍의 개념은 처음에 너무 이질적이었고 배움이 더뎠었다. 하지만, 업무를 하면서 내가 커밋한 코드를 리뷰할 때 팀장님께서는 함수형으로 작성하면 코드도 간결하고 속도 및 메모리 면에서 더 이득을 볼 것이라며 예시를 들어주셨다. 이렇게 조금씩 함수형 코드를 보게 되면서 함수형 프로그래밍이라면 1줄 혹은 몇 줄 안되게 작성할 수 있는 것을 명령형 프로그래밍을 사용하면 코드가 10 여줄 가까워지며, 가독성이 좋은 코드가 아님을 여실히 느끼게 되었다. 따라서 철저히 학습 필요성을 느끼게 되어 진지하게 함수형 프로그래밍에 대해 공부한 내용을 정리해보게 되었다

프로그래밍 패러다임

명령형 프로그래밍과 함수형 프로그래밍으로 나눌 수가 있으며, 이 둘을 구분하는 중요한 특징 하나는 상태에 대한 개념

  • 명령형(imperative) 프로그래밍
    • 순차적(procedual) 언어
    • 객체지향(object-oriented) 언어
  • 함수형(functional) 프로그래밍

파이썬 언어는 위 세 가지 범주에 속하는 특성을 모두 포함. 비록 파이썬은 순수한 함수형 언어는 아니지만 상당히 다양한 함수형 프로그래밍을 할 수 있음

파이썬 함수형 프로그래밍의 장점과 단점

파이썬 함수형 프로그래밍의 장점

  • 함수형 프로그래밍 언어의여러 디자인 패턴과 설계 기법을 파이썬 프로그래밍에 적용할 수 있음
  • 간결하고 우아한 프로그래밍이 가능
  • 파이썬의 제너레이터 식을 활용하면 메모리에 큰 데이터 구조를 만드는 일을 피할 수 있어 자원을 덜 사용하면서 더 빠르게 동작할 수 있는 프로그램을 작성

파이썬 함수형 프로그래밍의 단점

  • 쉽고 순수한 함수형 프로그램을 작성할 수 없음
  • 파이썬에는 순후 함수형 프로그래밍에 필요한 몇 가지 기능이 부족
    • 재귀호출의 깊이가 한정
    • 모든 식에 대한 지연 계산(Lazy Evaluation)을 지원하지 않음
    • 최적화된 컴파일러가 없음

파이썬이 제공하는 함수형 프로그래밍 기능

  • 함수가 일급 계층 객체(first class object)
    • 함수가 함수를 인자로 받거나 결과로 반환할 수 있음
    • 함수는 소스 코드 상의 구성 요소로만 존재하고 실행 시점의 데이터 구조로는 존재하지 않음
  • 여러가지 고차 함수를 제공
    • map()
    • filter()
    • functools.reduce()
    • sorted(), min(), max() 등도 있음
  • 함수형 프로그래밍
    • 불변적인 데이터 구조를 자주 활용 → 상태가 없는 개체를 강조하면 최적화를 유연하게 수행할 수 있음
    • 파이썬은 tuple과 namedtuple을 불변 객체로 제공

모델 최적화

모델 최적화: 훈련 성능과 일반화 성능의 균형

개요

  • 모델 최적화는 훈련 성능을 극대화하고, 과대적합을 방지하여 일반화 성능을 유지하는 두 가지 방향에서 이루어져야함
  • 과대적합은 훈련 데이터에 지나치게 맞추는 현상이며, 반대로 일반화는 새로운 데이터에서도 좋은 성능을 유지하는 것을 의미합
  • 이 글에서는 훈련 성능을 향상시키면서도 과대적합을 방지하는 방법들을 설명

1. 과대적합(Overfitting)과 일반화(Generalization)

과대적합(Overfitting)

  • 과대적합은 모델이 훈련 데이터의 세부적인 패턴에 지나치게 맞춰지면서, 훈련 데이터에서는 매우 좋은 성능을 내지만, 새로운 데이터에서는 성능이 저하되는 현상을 의미
  • 과대적합된 모델은 학습한 데이터에만 적합하고, 현실 세계의 데이터에 대한 예측 능력이 떨어짐

일반화(Generalization)

  • 일반화는 모델이 훈련 데이터 이외의 데이터에서 잘 작동하는 능력을 의미
  • 딥러닝 모델의 성능을 제대로 평가하려면 훈련 데이터와는 다른 검증 데이터나 테스트 데이터에서 모델이 얼마나 잘 작동하는지 살펴보는 것이 중요
  • 일반화를 달성하기 위해서는 훈련 중 과대적합을 방지하는 다양한 기법이 필요

2. 훈련 성능 향상 (과대적합 가능성 증가)

  • 훈련 성능을 향상시키는 것은 모델이 훈련 데이터에서 좋은 성능을 내도록 최적화하는 과정
  • 하지만 이 과정에서 과대적합의 위험이 증가할 수 있으므로 주의가 필요

1) 경사하강법의 핵심 파라미터 튜닝

  • 옵티마이저 선택

    • 딥러닝 모델을 훈련할 때 Adam이나 SGD(Stochastic Gradient Descent) 같은 옵티마이저를 선택할 수 있음(Adam은 적응형 학습률을 사용하여 빠르고 안정적인 학습을 제공)

  • 학습률(Learning rate)

    • 학습률이 너무 높으면 훈련이 불안정해지고, 너무 낮으면 학습 속도가 느려짐
    • 적절한 학습률을 선택하는 것이 훈련 성능을 향상시키는 중요한 요소
    • 학습률 스케줄링(Learning rate scheduling)을 적용하면, 학습 도중에 학습률을 점진적으로 줄여 과대적합을 방지할 수 있음

  • 배치 크기(Batch size)

    • 배치 크기가 클수록 훈련의 안정성이 증가하지만, 작은 배치 크기는 더 빠른 수렴과 일반화 성능 향상에 도움이 될 수 있음
    • 프랑소와 숄레는 작은 배치 크기를 사용할 경우, 노이즈를 통해 모델이 더 나은 일반화 성능을 가질 수 있다고 언급

2) 모델 구조 조정

  • 적합한 모델 선택

    • 모델 구조는 해결하려는 문제에 맞게 설계되어야 함
    • 간단한 문제를 해결할 때는 작은 모델로도 충분하지만, 복잡한 문제에서는 더 큰 네트워크가 필요
    • 프랑소와 숄레는 모델 선택 시 베이스라인 모델을 먼저 구현하고, 이를 바탕으로 더 복잡한 모델을 설계하는 방법을 권장

  • 층을 늘리기

    • 딥러닝 모델의 깊이를 늘리는 것은 복잡한 패턴을 학습할 수 있도록 도와줌
    • 하지만, 너무 많은 층을 사용하면 과대적합의 위험이 커질 수 있으므로 주의해야 함

3) 모델 용량 증가

  • 노드 수 증가
    • 각 층의 **뉴런 수(노드 수)**를 늘리면 모델의 용량이 커져 더 복잡한 패턴을 학습할 수 있음
    • 하지만, 너무 큰 모델은 훈련 데이터에 과도하게 맞춰지면서 일반화 성능이 저하될 수 있음

3. 일반화 성능 향상 (과대적합 방지)

  • 과대적합을 방지하고 모델의 일반화 성능을 높이기 위해서는 다양한 정규화 기법과 모델 조정 방법을 적용할 수 있음
  • 프랑소와 숄레는 딥러닝 모델이 데이터에 너무 맞춰지지 않도록 하기 위해 아래와 같은 기법들을 권장

1) 데이터셋 큐레이션

  • 더 많은 데이터 확보

    • 더 많은 훈련 데이터를 확보하면 모델이 더 다양한 패턴을 학습할 수 있어 일반화 성능이 향상됨
    • 데이터를 증가시키는 데이터 증강(data augmentation) 기법도 매우 유용

  • 정확한 레이블 할당

    • 잘못된 레이블을 포함한 데이터는 모델의 성능을 저하시키고 과대적합을 유발할 수 있음
    • 레이블 오류를 최소화하는 것이 중요

2) 더 나은 특성(feature) 개발

  • 특성 공학은 모델이 학습할 수 있는 의미 있는 특성을 만들어내는 과정
  • 프랑소와 숄레는 딥러닝에서 자동으로 특성을 추출할 수 있는 장점이 있지만, 여전히 좋은 특성 설계가 모델 성능에 중요하다고 강조

3) 네트워크 용량 줄이기

  • 간소한 모델 설계
    • 모델의 복잡성을 줄여 과대적합을 방지
    • 층의 수나 노드 수가 너무 많으면 훈련 데이터에 지나치게 맞춰지기 때문에, 문제에 적합한 적당한 크기의 모델을 사용하는 것이 중요

4) 가중치 규제(Regularization)

  • L1, L2 규제: 모델의 가중치 크기를 제한하여 너무 큰 가중치 값을 방지
  • 이를 통해 모델이 훈련 데이터에 너무 맞춰지지 않도록 하고, 더 일반화된 모델을 만들 수 있음

5) 드롭아웃(Dropout)

  • 드롭아웃은 훈련 중에 무작위로 일부 뉴런을 비활성화하여 모델이 특정 뉴런에 의존하지 않도록 하는 정규화 기법입니다.
  • 이를 통해 과대적합을 방지하고 더 나은 일반화 성능을 얻을 수 있음

6) 교차 검증(Cross-validation)

  • 교차 검증은 데이터를 여러 번 나누어 학습하고 검증하여 모델이 훈련 데이터에 과적합되지 않도록 방지하는 방법입니다. 특히 소규모 데이터셋에서 유용하며, 데이터 분할에 따른 성능 차이를 줄일 수 있습니다.

7) 조기 종료(Early Stopping)

  • 조기 종료는 검증 데이터에서 성능이 더 이상 개선되지 않을 때 훈련을 멈추는 기법
  • 과대적합이 일어나는 것을 방지하기 위해 **검증 손실(validation loss)**이 일정 기간 동안 향상되지 않으면 훈련을 중단할 수 있음

결론

결론

  • 딥러닝 모델의 최적화는 훈련 성능을 극대화하는 것과 동시에 과대적합을 방지하여 일반화 성능을 높이는 것이 핵심
  • 프랑소와 숄레는 이 두 가지 목표를 균형 있게 달성하는 것이 중요하다고 강조하며, 모델이 현실 세계에서도 높은 성능을 유지하도록 다양한 정규화 기법과 훈련 기법을 적용해야 한다고 설명
  • 훈련 성능을 향상시키면서도 과대적합을 방지하는 전략을 적절히 사용하면, 모델은 더 나은 일반화 성능을 유지하며, 새로운 데이터에 대해 높은 예측력을 가질 수 있음

부록 1. 일반화 성능을 최적화하는 전략: 학습 그래프 기반 접근

  • 모델 학습 과정에서 **훈련 곡선(Training curve)**과 **검증 곡선(Validation curve)**을 주의 깊게 관찰하는 것은 과대적합과 과소적합을 방지하고, 모델이 일반화 성능을 갖출 수 있도록 돕는 강력한 방법
  • 이를 기반으로 모델의 네트워크 크기나 하이퍼파라미터를 점진적으로 조정하여 최적의 성능을 찾는 것이 핵심

1. 먼저 모델을 과대적합시키기

  • 초기에는 충분한 복잡성을 가진 네트워크로 시작하여 모델이 훈련 데이터에 잘 맞도록 해야함
  • 이때 훈련 데이터에서 성능이 높아지고, 검증 데이터에서 성능이 떨어지는 과대적합이 발생하는 시점을 확인.
  • 과대적합의 징후는 훈련 세트에서는 손실이 계속해서 낮아지는데, 검증 세트에서는 손실이 낮아지다가 다시 증가하는 패턴을 보임

2. 과대적합 발생 시 네트워크 크기 줄이기

  • 모델이 과대적합되기 시작하면, 네트워크의 복잡도를 줄이기 위해 층의 수나 노드 수를 줄임
  • 이를 통해 모델이 더 단순해지고, 과대적합을 방지할 수 있음
    • 층 수 또는 뉴런 수를 줄여서 모델이 과도하게 복잡하지 않도록 하되, 훈련 데이터에 대한 성능이 여전히 유지되는지 확인

3. 과소적합 발견 시 네트워크 크기 증가

  • 네트워크를 단순화한 후 과소적합이 발생할 수 있음 (과소적합은 훈련 데이터에서의 성능이 충분히 올라가지 않는 경우)
    • 훈련 곡선과 검증 곡선 모두가 손실이 계속 높은 상태에 머무르는 경우 과소적합을 의미합
    • 이때는 네트워크 크기를 다시 키워야 함
  • 층의 수나 노드 수를 다시 늘려 모델이 충분한 표현력을 가질 수 있도록 조정

4. 최적의 일반화 성능을 찾기

  • 과대적합과 과소적합 사이의 균형점을 찾기 위해, 네트워크의 복잡성을 조정하면서 훈련 데이터와 검증 데이터에서 성능이 모두 향상되는 지점을 찾아야 함
  • 훈련 손실은 꾸준히 낮아지고, 검증 손실은 더 이상 증가하지 않는 상태가 최적의 일반화 성능을 의미
  • 이를 확인하기 위해서는 훈련과 검증 곡선이 비슷하게 수렴하는 지점을 찾는 것이 중요

학습 곡선을 통해 모델 성능 조정하기

훈련 및 검증 손실 그래프

  • 훈련 중 **훈련 손실(Training Loss)**과 **검증 손실(Validation Loss)**의 그래프를 주의 깊게 살펴보는 것이 중요

  • 각 손실 값의 변화 추이를 통해 과대적합 또는 과소적합 여부를 판별할 수 있음

    • 과대적합 발생 시: 훈련 손실은 계속해서 감소하지만, 검증 손실은 증가하기 시작

    • 과소적합 발생 시: 훈련 손실과 검증 손실 모두 높은 상태로 남아있음

    • 일반화가 잘 이루어질 때: 훈련 손실과 검증 손실이 함께 낮아지며, 검증 손실이 더 이상 증가하지 않을 때가 최적의 상태

훈련 및 검증 정확도 그래프

  • 정확도 그래프(Accuracy Curve) 또한 학습 곡선에서 중요한 지표

  • 훈련 정확도와 검증 정확도 사이에 큰 차이가 생기면 과대적합의 가능성이 있음

    • 과대적합: 훈련 정확도는 계속해서 높아지지만, 검증 정확도는 일정 수준에서 멈추거나 떨어짐

    • 과소적합: 훈련 정확도 자체가 낮고, 검증 정확도도 낮은 상태가 지속

    • 일반화: 훈련과 검증 정확도가 비슷한 수준에서 안정되면 모델이 일반화가 잘 되고 있다는 신호

데이터 최적화(데이터 누수, 데이터 전처리, 데이터 분할 전략)

개요

  • 데이터 최적화는 머신러닝 모델 성능을 극대화하기 위한 필수 단계
  • 모델을 학습시키기 위해서는 적절한 데이터 처리와 전처리 과정이 필요하며, 이를 통해 모델이 과대적합을 방지하고 일반화 성능을 높일 수 있음
  • 데이터 누수 방지와 함께 데이터 전처리 및 데이터 분할 전략을 중심으로 효율적인 데이터 준비 방법에 대해 알아봄

1. 데이터 누수(Data Leakage) 방지

  • 데이터 누수는 모델이 학습하는 동안 미래 정보를 포함하거나 평가 과정에서 테스트 데이터의 일부를 사용하는 경우 발생
  • 이는 모델 성능이 부풀려질 수 있지만, 실제 환경에서의 성능은 크게 떨어지게 됨
  • 데이터 누수는 훈련 데이터에 부정확한 정보를 제공하여 모델이 잘못 학습하게 만드는 중요한 문제

데이터 누수 방지 방법

1) 미래 정보 포함 여부 확인

  • 미래의 데이터를 학습에 포함시키는 것은 누수의 가장 흔한 사례
  • 시계열 데이터나 시간 의존적인 데이터에서는 과거 데이터만 사용해야 하며, 미래 데이터를 절대 포함하지 않도록 주의해야 함
  • 금융 시장에서 주식 가격 예측 모델을 만들 때, 미래의 주가 데이터가 학습에 포함되지 않도록 해야함 → 이를 방지하기 위해 시간순으로 데이터를 분할하여 과거 데이터를 학습에 사용하고, 미래 데이터를 검증용으로 사용함

2) 중복 샘플 방지

  • 같은 샘플이 훈련과 테스트 세트에 중복되어 포함되면, 모델은 이미 학습한 데이터를 기반으로 평가되어 성능이 과대평가될 수 있음
  • 데이터 분할 시에는 샘플 중복을 방지하여 데이터 누수가 발생하지 않도록 해야함
  • 예시: 고객 데이터를 기반으로 머신러닝 모델을 학습할 때, 같은 고객이 훈련과 테스트 세트에 중복으로 포함되지 않도록 주의해야 함

3) 전처리 단계에서의 주의

  • 전처리 과정에서 데이터를 잘못 처리하면 데이터 누수가 발생할 수 있음
  • 특히 훈련 세트와 테스트 세트를 구분하지 않고 전체 데이터에 대한 통계값을 사용하여 전처리를 하면 테스트 데이터의 정보가 학습 과정에 누출될 수 있음
  • 예시: 결측값 처리 시 전체 데이터의 평균값이나 중앙값을 사용하는 대신, 훈련 데이터에서만 통계치를 계산하고 테스트 데이터에는 그 값을 적용해야 함

4) 특징 공학(Feature Engineering)에서의 주의

  • 예측할 타깃 변수와 강하게 연관된 변수를 모델 학습에 포함하면 미래 정보가 누출되어 모델이 과대평가될 수 있음
  • 예시: 의료 데이터를 활용한 예측 모델에서, 환자의 퇴원 여부를 예측하는 경우, 이미 퇴원 여부가 포함된 변수를 학습에 포함시키면 안됨

5) 레이블 누수 확인

  • 모델이 학습 과정에서 타깃 변수와 관련된 정보를 직접적으로 사용하는 것은 데이터 누수의 전형적인 사례입
  • 타깃 변수와 직접적인 상관관계가 있는 변수를 학습에 포함시키지 않도록 해야 함
  • 예시: 대출 승인 여부를 예측할 때, 이미 승인 여부가 기록된 변수를 모델이 학습하도록 하면 안됨

2. 데이터 분할 전략

  • 데이터 분할은 모델의 성능을 제대로 평가하기 위해 필수적인 과정
  • 데이터를 적절히 분할함으로써 모델의 일반화 성능을 평가하고, 과대적합을 방지할 수 있음
  • 프랑소와 숄레는 데이터 분할의 중요성을 강조하며, 올바른 분할이 성능 평가에 핵심적인 역할을 한다고 언급

데이터 분할 방법

1) 훈련/검증/테스트 데이터 분할

  • 일반적으로 데이터는 세 개의 세트로 나누어 사용

    • 훈련 세트(Train): 모델을 학습시키는 데 사용됨

    • 검증 세트(Validation): 하이퍼파라미터 튜닝 및 모델 선택에 사용됨

    • 테스트 세트(Test): 최종적으로 모델 성능을 평가하는 데 사용됨 테스트 세트는 한 번만 사용하고, 이후에는 모델 학습에 포함되지 않아야함

2) 데이터 섞기(Shuffle)

  • 데이터는 무작위로 섞는 것이 중요, 그렇지 않으면 데이터 순서에 따라 모델이 편향된 학습을 할 수 있음
  • 특히 지역별이나 시간순으로 정렬된 데이터는 무작위로 섞어야 모델이 다양한 패턴을 학습할 수 있음
  • 단, 시계열 데이터의 경우에는 데이터를 섞으면 안 되고, 시간순으로 유지해야 함, 그렇지 않으면 미래 데이터를 학습에 사용하게 되어 데이터 누수가 발생할 수 있음

3) 층화 샘플링(Stratified Sampling)

  • 데이터셋이 불균형한 경우, 데이터 분할 시 층화 샘플링을 사용하여 클래스 비율을 유지해야함
  • 이를 통해 소수 클래스가 훈련 및 검증 세트에서 충분히 포함될 수 있도록 보장
  • 예시: train_test_split 함수에서 stratify 옵션을 사용하여 불균형 데이터의 클래스 비율을 유지

3. 데이터 전처리(Data Preprocessing)

  • 데이터 전처리는 모델 학습에 있어서 중요한 단계
  • 프랑소와 숄레는 데이터를 적절히 전처리해야만 모델이 일반화 성능을 높일 수 있다고 설명
  • 이 과정에서 잘못된 전처리는 모델 성능을 저하시킬 수 있으며, 데이터 누수의 원인이 될 수도 있음

전처리 방법

1) 결측값 처리(Missing Data Handling)

  • 결측값을 처리하지 않고 모델에 포함하면 성능 저하가 발생할 수 있음
  • 결측값은 평균값, 중앙값 등으로 대체할 수 있으며, 때로는 특정 모델을 사용하여 결측값을 예측할 수 있음
  • 중요한 점은, 훈련 세트의 통계값만 사용하여 결측값을 처리해야 한다는 것, 테스트 데이터에는 훈련 세트에서 계산된 통계값을 사용해야 데이터 누수를 방지할 수 있음

2) 범주형 변수 인코딩(Categorical Encoding)

  • 머신러닝 모델은 숫자형 데이터만 처리할 수 있기 때문에 범주형 변수는 숫자로 변환해야함

    • One-Hot Encoding: 범주형 변수를 이진 변수로 변환하여 처리하는 방법

    • Label Encoding: 범주를 숫자로 변환하는 방식

3) 스케일링(Scaling)

  • 변수 간 스케일이 크게 다른 경우, 정규화나 표준화를 통해 스케일을 맞춰줘야함
  • 이는 특히 거리 기반 알고리즘에서 매우 중요함
  • 이때도 스케일링은 훈련 세트로부터 계산된 값을 사용하여 테스트 세트에 적용해야함

4. 데이터 증강(Data Augmentation)

  • 데이터가 충분하지 않거나 특정 클래스의 데이터가 부족한 경우, 데이터 증강을 통해 데이터를 인위적으로 늘릴 수 있음
  • 특히 딥러닝 모델에서는 데이터 증강이 중요한 역할을 함

데이터 증강 방법

1) 이미지 데이터 증강

  • 이미지 데이터의 경우 회전, 확대, 축소, 뒤집기 등을 통해 다양한 훈련 데이터를 생성할 수 있음
  • 이는 과대적합을 방지하고, 모델이 다양한 패턴을 학습할 수 있도록 도와줌

2) 텍스트 데이터 증강

  • 텍스트 데이터에서는 문장의 순서를 변경하거나 동의어로 대체하는 방식으로 데이터를 증강할 수 있음
  • 이를 통해 텍스트 데이터를 더욱 다양하게 만들 수 있음

3) 합성 데이터 생성

  • **GAN(생성적 적대 신경망)**이나 SMOTE 기법을 통해 소수 클래스 데이터를 생성할 수 있음
  • 이 방법은 데이터가 부족하거나 불균형 데이터셋에서 사용될 수 있음

5. 데이터 품질 유지 및 관리

  • 데이터 최적화의 마지막 단계는 데이터 품질 관리
  • 프랑소와 숄레는 데이터 일관성과 버전 관리가 중요하다고 강조하며, 데이터가 변경될 경우 모델 성능에 미치는 영향을 추적하는 것이 필수적이라고 설명

데이터 품질 관리 방법

  • 데이터 버전 관리: 데이터 수집 및 처리 과정에서 버전 관리를 명시하여 추적합니다. 이를 통해 데이터 변경 시 모델에 미치는 영향을 추적할 수 있음
  • 데이터 소스의 일관성 유지: 여러 소스에서 데이터를 수집할 때, 스키마와 포맷을 일관되게 유지해야함
  • 데이터 품질 모니터링: 정기적으로 데이터 품질을 모니터링하여 이상값이나 패턴을 확인하고, 잘못된 데이터가 모델에 학습되지 않도록 방지해야함

1. 우리는 올바르게 학습을 하고 있는가?

필드에서 일하는 우리에게 학습의 모습은 어떠해야할까??

어떠한 프로젝트 일감이 들어왔는데, 그 프로젝트는 내가 기존에 사용해본 적이 없는 심지어는 단순한/단편적인 지식만 있다고 하자. 이 때 어떻게 반응을 하는가?

나의 경우의 학습의 문제점(반성할 점 😰)

나의 경우에는 많은 경우 아래와 같은 습관이 있다.

  • 습관
    • 주제에 맞는 책을 구매하고 A-Z까지 정독을 하고, 책에 있는 예제를 따라하며 정리한다.
    • 해당 내용을 동료들과 공유하기 위해 PPT를 작성()
    • 배운 것들을 바탕으로 구현을 시작하나 얼마가지 않아 문제점에 봉착함
      • 업무를 하다보면 책에 없는 내용들이 많음
    • 문제점을 해결하기 위해 블로그/오픈채팅방 및 또다른 전문서적 등을 찾아보며 점진적인 구현
  • 단점
    • 학습에 너무 많은 시간을 사용하다보니 구현에 많은 시간을 쏟지를 못함
      • 과할 경우, 코드를 치는 것보다 읽고 정리하는 시간이 더 길음
    • 최악의 경우, 읽은 책이 실제 코드에 활용되는 부분이 별로 없음

위와 같이 반응하는 이유는 책에서 말하는 것처럼 학교 학습에서 효과적이었던 방법들을 그대로 야생 학습에 가져와서 적용하려 하기 때문이다. 학교에서의 습관과 전략을 여전히 필드에서 사용하기에 거기에서부터 문제가 발생한다.

야생학습과 학교학습

야생학습(😀) 학교학습(😰)
대부분 협력적 대부분 개별적
대부분 비순차적 공부 순서가 정해져있음
대부분 자료에 한정이 없다 대부분 교과서, 교재, 시험 범위 등 정해져 있음
명확한 평가가 없다 대부분 시험이라는 명확한 평가 기준이 있음
대부분 정답이 없음 무엇이 정답인지 명확
대부분 목표가 불분명하고 바뀜 대부분 합격, 자격증 같은 목표가 분명

우리가 속한 필드는 거의 대부분이 야생학습의 환경이다. 그래서 우리가 속한 곳은 야생학습이라는 패러다임을 갖고 나아가야 한다.

2. 무엇을 배워야 보다 유리해질수 있을까

인공지능 시대에 대비하려면 배우기 힘든 것에 집중하는 것이 더 좋다. 왜냐하면 정형화할 수 있거나 일정 수준의 규칙이 있는 것에 대해서는 인공지능의 능력이 사람의 능력보다 뛰어나기에, 우리는 인공지능이 배우기 힘든 것에 대해서 배워야 보다 미래에 유리해질 것이다.

학습에 불리한 조건(인공지능이 잘하는 것)(😀) 학습에 유리한 조건(인공지능에 대체대기 어려운 것)
목표가 분명/객관적/정적 목표가 모호/주관적/동적
매 순간 선택할 수 있는 행동/선택의 종류가 유한함 매 순간 선택할 수 있는 행동/선택의 종류가 불확실
매 순간 자신이 목표에 얼마나 근접했는지 알 수 있음(선택의 피드백을 빨리 받을 수 있음) 매 순간 자신의 목표에 얼마나 근접했는지 알기 어렵다(선택의 피드백을 빨리 받기 어려움)
닫힌 시스템(즉, 예상 못한 외부 요소가 갑자기 들어오지 않는) 속에서 일함 열린 시스템(예상 못한 외붕 요소가 갑자기 들어오는 경우가 흔한) 속에서 일함
과거의 선택과 결과에 대한 구조화된 데이터가 많음 선택과 결과에 대한 구조화된 데이터가 별로 없음

즉 암묵지, 직관으로 작동하는 회색 영역, 다시말해 자신이 왜 이런 선택을 했는지 쉽게 설명할 수 없는 것들을 선택하는 것이 보다 좋을 것이다.

    • 내가 한 행동에 대한 피드백을 10분후, 1시간, 1일, 1주일 후 등 여러 주기를 통해 지속적으로 얻어야 함
      • (그렇지 못한다면) 행동/결정한 문제에 대한 피드백을 몇 달후(테스트 단계)에 받음 - 결정 내렸던 근거들에 깊이감이 없어지며며 심지어는 결정의 이유조차 가물해짐
    • 저지른 실수는 바로 다음 주기에서 교정
    • 상사, 고객 혹은 내가 개발하는 프로그램에서 직접 피드백을 적극적으로 구해야함
  • 타당성
    • 변수를 제한하고 실험을 하면서 규칙성과 인과관계를 찾으려는 노력을 해야함
  • 적절한 난이도
    • 나의 실력과 작업의 난이도가 비슷해야 함

    • 크라센의 입력 가설(Input Theory) : i+1 이론이라고 하며, 학습자의 수준을 i라고 할 때 딱 한 단계 높은 i+1 수준의 입력이 주어질 때에만 언어 능력이 유의미하게 진전한다는 이론

포트 중복 사용에 따른 포트번호 확인 및 프로세스 종료

Intro

개발을 수행하면서 부득이하게 프로세스를 종료시켰다고 생각했지만, 윈도우 백그라운드에서 프로그램이 동작 중일 경우가 발생한다. 그래서 프로세스를 기동하려고 하면, 포트가 중복되어 에러를 반환함을 보게 된다. 또한 IntelliJ에서 tomcat을 통해 개발 중인데 tomcat JMX port(1099) 등이 다른 프로세스가 선점해서 포트 중복으로 기동이 안되는 경우도 마주치게 된다.

이러한 경우를 위해 컴퓨터를 재기동하여 해결할 수도 있겠지만(😭) Windows CMD를 통해 프로세스를 안전하게 종료시켜보자!

핵심 키워드

  • netstat -nao : 현재 Windows에서 등록되어 있는 포트를 검색함
  • netstat -nao | findstr 포트번호 : 현재 Windows에서 등록되어 있는 포트 중에서 입력한 포트번호에 대해서만 정보를 가져옴
  • tasklist /svc /FI “PID eq 프로세스아이디” : 특정 PID가 어떠한 프로세스(서비스)인지 확인
  • taskkill /f /pid 프로세스아이디 : 특정 PID에 대해서만 프로세스 종료

Contents

윈도우에서 특정 포트(port)를 사용하는 프로세스를 찾기 위해서는 netstat, findstr 명령어를 사용해야함

  • netstan -nao를 사용하면 현재 Windows에 등록되어 있는 포트 정보가 출력됨

    • netstat -nao | findstr

  • 특정 포트번호만 확인하기 위해서는 netstat -nao | findstr 포트번호를 입력

  • 마지막의 값이 PID 즉, 프로세스 ID이므로 해당 프로세스를 강제종료

    • 하지만 이에 앞서 해당 PID가 어떤 서비스인지 확인을 해보자

ElasticSearch를 통해 본 REST API 설계 사례 연구

  • ElasticSearch는 현대 분산형 검색 엔진에서 널리 사용되는 도구로, 그 RESTful API 설계는 매우 직관적이며 확장성이 뛰어남
  • 이번 글에서는 ElasticSearch의 RESTful 아키텍처 설계 방식을 사례로, 내가 개발하는 소프트웨어에서 REST API 설계를 어떻게 하면 좋을지 알아보고자 함
  • 그 과정에서 ElasticSearch가 RESTful 아키텍처 원칙을 어떻게 구현했는지를 살펴보고, 이를 기반으로 내가 만들고자 하는 API 설계를 구체화해보자

1. 리소스 기반 설계 (Resource-Oriented Design)

ElasticSearch의 리소스 설계

  • ElasticSearch는 데이터를 리소스 단위로 관리
    • 인덱스는 하나의 데이터베이스처럼 사용되며, 각 인덱스는 여러 도큐먼트(Document)를 포함
    • 도큐먼트는 JSON 형태로 저장되며, 인덱스와 도큐먼트 ID를 통해 고유하게 식별 가능
    • GET /index/_doc/{id}로 특정 도큐먼트를 조회하고, POST /index/_doc/{id}로 도큐먼트를 추가 또는 수정하며, DELETE /index/_doc/{id}로 도큐먼트를 삭제하는 방식
  • 이처럼 리소스 기반 설계는 각 리소스를 명확하게 URI로 표현하며, URI를 보면 어떤 작업이 이루어지는지 직관적으로 이해 가능

내 소프트웨어에 적용하기

  • 내가 설계하는 API에서도

    리소스를 기반으로 한 URI 설계

    를 참고할 수 있음

    • 예를 들어, 사용자 데이터를 관리하는 API라면, /users/{id}로 사용자를 조회하고 /users/{id}로 사용자 데이터를 수정하는 방식으로 설계 가능
    • 중요한 것은 리소스(사용자, 제품, 주문 등)를 명확히 정의하고, 이를 URI로 직관적으로 나타내는 것

2. HTTP 메서드의 일관된 사용

ElasticSearch의 HTTP 메서드 사용

  • ElasticSearch는 REST의 HTTP 메서드를 매우 일관성 있게 사용
    • GET: 데이터를 조회
    • POST: 새로운 리소스를 생성하거나 기존 리소스를 업데이트
    • DELETE: 리소스를 삭제
    • 각 메서드는 특정한 작업을 수행하며, 그 목적에 맞게 사용됨
  • 이를 통해 HTTP 표준을 따르면서도, API 사용자는 어떤 메서드를 사용할지 쉽게 예측 가능

내 소프트웨어에 적용하기

  • API를 설계할 때,

    HTTP 메서드를 일관성 있게 사용하는 것

    이 매우 중요

    • 데이터 조회에는 GET, 생성에는 POST, 삭제에는 DELETE 등 명확한 규칙을 정하여 사용자에게 일관된 경험을 제공
    • HTTP 메서드와 리소스의 역할을 일관되게 매핑하여 혼란을 최소화할 수 있음

3. Stateless 설계 (비상태성)

ElasticSearch의 Stateless 설계

  • ElasticSearch는 각 요청이 독립적임
    • 클라이언트 상태를 서버에서 기억하지 않으며, 모든 요청은 필요한 정보를 모두 포함해야 함
    • 예를 들어, 검색 작업을 할 때는 매번 쿼리와 필요한 파라미터를 URI나 본문에 포함시켜서 요청
  • 이러한 비상태성 설계는 서버 확장성과 성능에 유리하며, 특히 분산 시스템에서 효과적임

내 소프트웨어에 적용하기

  • 내가 설계하는 API에서도

    Stateless 설계

    를 적용할 수 있음

    • 서버는 클라이언트의 상태를 기억하지 않고, 각 요청은 필요한 정보를 포함해야 함
    • 이를 통해 분산 시스템에서의 확장성을 높이고, 서버 리소스를 효율적으로 사용할 수 있음

4. URL의 일관성

ElasticSearch의 URL 설계

  • ElasticSearch는 일관된 URI 패턴을 통해 다양한 작업을 수행
    • 인덱스 관리: GET /index로 인덱스 조회
    • 도큐먼트 관리: GET /index/_doc/{id}로 도큐먼트 조회
    • 검색 작업: POST /index/_search로 검색 수행
    • 이처럼 모든 작업에 대해 일관성 있게 URI가 설계되어 있어, API 사용자 입장에서 학습 곡선이 낮음

내 소프트웨어에 적용하기

  • 내가 설계하는 API에서도

    일관된 URI 패턴

    을 사용하는 것이 중요

    • 같은 리소스에 대해 유사한 패턴을 유지함으로써, API 사용자들이 새로운 엔드포인트를 쉽게 예측하고 사용할 수 있도록 해야 함
    • 예를 들어, /products로 제품 목록을 조회하고, /products/{id}로 특정 제품을 조회하는 방식으로 일관성을 유지하는 것이 좋음

5. 확장성과 유연성을 고려한 파라미터화

ElasticSearch의 파라미터 활용

  • ElasticSearch는 다양한 URI 파라미터를 사용하여 기능을 확장하고 유연성을 제공
    • pretty: 응답을 보기 좋게 포맷팅
    • q: 검색 쿼리를 간단히 수행
    • size: 반환할 도큐먼트 수를 제한
    • 이러한 파라미터들은 클라이언트가 세부적으로 요청을 제어할 수 있게 해주며, 다양한 요구 사항을 충족

내 소프트웨어에 적용하기

  • API 설계 시,

    파라미터를 통한 유연성 제공

    이 중요함

    • 예를 들어, 데이터 조회 API에서 limit, offset 등의 파라미터를 추가하여 페이지네이션을 구현하거나, sort 파라미터를 사용하여 데이터를 정렬하는 방식으로 유연성을 줄 수 있음
    • ElasticSearch처럼 클라이언트가 필요한 정보를 세밀하게 조정할 수 있는 파라미터 설계가 필요함

결론

ElasticSearch는 RESTful 아키텍처의 원칙을 충실히 따르면서도, 확장성과 유연성을 모두 고려한 API 설계를 보여줌. ElasticSearch의 API 설계를 분석해본 결과, 내가 설계하는 소프트웨어에서도 적용할 수 있는 주요 포인트는 다음과 같음:

  1. 리소스 기반 설계: 모든 엔티티를 명확한 리소스로 보고 URI로 표현
  2. HTTP 메서드의 일관된 사용: 표준을 준수하며 각 메서드에 맞는 작업 수행
  3. Stateless 설계: 각 요청이 독립적이며, 서버가 클라이언트 상태를 기억하지 않음
  4. 일관된 URL 설계: 모든 엔드포인트가 예측 가능하도록 일관성 유지
  5. 파라미터를 통한 유연성 제공: 다양한 요구 사항에 대응할 수 있는 확장성을 제공

이러한 원칙을 내 소프트웨어 설계에 반영함으로써, 유지보수성과 확장성이 뛰어난 RESTful API를 구축할 수 있을 것임. ElasticSearch의 설계 사례는 API 설계의 좋은 참고 자료가 될 수 있으며, 이를 통해 더 나은 소프트웨어 아키텍처를 구축할 수 있음을 확인할 수 있음.
