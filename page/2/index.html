L4 스위치의 로드 밸런싱 과 웹 서버의 로드 밸런싱 차이

개요

  • 리버스 프록시의 로드 밸런싱과 L4 스위치를 통한 로드 밸런싱은 서로 다른 계층에서 동작하며, 각각 장단점이 있음
  • L4 스위치는 네트워크 계층에서 트래픽을 분산하는 반면,
  • 리버스 프록시는 애플리케이션 계층에서 트래픽을 분산
  • 그러면 왜 웹 서버를 통한 로드 밸런싱을 사용하는지, 그 이유를 비교하며 살펴보자

1. L4 스위치 로드 밸런싱 (전송 계층, 4계층)

  • L4 스위치는 OSI 모델의 4계층(전송 계층)에서 IP 주소와 포트 정보를 기반으로 트래픽을 분산
  • 이 방식은 TCP/UDP 연결 정보를 바탕으로 트래픽을 전달하기 때문에, 애플리케이션의 동작 방식에 대한 이해 없이 순수하게 네트워크 정보만을 가지고 로드 밸런싱을 수행

장점

  • 성능
    • L4 스위치는 네트워크 계층에서 동작하기 때문에 트래픽 처리 속도가 매우 빠름
    • 애플리케이션 계층까지 파악하지 않기 때문에 부하가 적음
  • 프로토콜 독립성: TCP/UDP와 같은 전송 계층 프로토콜만 처리하므로, HTTP 외에도 다양한 프로토콜을 지원
  • 안정성: 네트워크 장비의 특성상 안정적으로 트래픽을 분산할 수 있음

단점

  • 세부적인 트래픽 제어가 불가능
    • L4 스위치는 HTTP 요청의 헤더나 쿠키 등의 세부 정보에 접근할 수 없기 때문에, 애플리케이션 레벨에서 필요한 세밀한 로드 밸런싱(예: 세션 스티키네스, 특정 경로 기반 로드 밸런싱 등)이 불가능
  • 상태 기반 로드 밸런싱 불가능
    • L4 스위치에서는 사용자 세션이나 쿠키 같은 애플리케이션 상태를 인식할 수 없으므로, 세션 기반의 분산 처리가 어려움

2. 웹 서버를 통한 리버스 프록시 로드 밸런싱 (애플리케이션 계층, 7계층)

  • Nginx나 Apache 같은 웹 서버는 리버스 프록시로서 **애플리케이션 계층(7계층)**에서 트래픽을 분산함
  • HTTP 및 HTTPS 요청을 직접 처리하고, HTTP 헤더나 쿠키와 같은 애플리케이션 관련 데이터를 기반으로 로드 밸런싱을 수행할 수 있음

장점

  • 세부적인 트래픽 제어 가능

    • 웹 서버는 HTTP 요청의 세부적인 정보(예: URL 경로, 헤더, 쿠키, 파라미터)를 분석하여, 요청을 세밀하게 분산할 수 있음
    • 이를 통해 경로 기반(path-based) 또는 도메인 기반 로드 밸런싱이 가능합니다.

  • 세션 관리

    • 웹 서버는 세션 스티키네스(Session Stickiness)를 설정하여, 사용자의 요청을 같은 서버로 보내는 방식을 지원
    • 이를 통해 로그인 상태나 장바구니 같은 사용자 상태를 유지할 수 있음

  • SSL/TLS 처리

    • 웹 서버는 SSL/TLS 암호화를 처리할 수 있음
    • 웹 서버가 SSL 종료를 처리하고, 내부로는 HTTP 요청을 전달할 수 있음

  • 다양한 부하 분산 방식 라운드 로빈, 최소 연결 방식, IP 해싱 등 다양한 로드 밸런싱 전략을 적용할 수 있음

단점

  • 성능

    • L4 스위치보다 **처리 속도가 느릴 수 있음
    • 특히 대규모 트래픽 환경에서는 L4 스위치만큼의 성능을 보장하기 어려울 수 있음

  • 복잡성

    • L4 스위치에 비해 웹 서버는 더 많은 구성 요소와 설정을 요구
    • SSL 처리, 상태 관리 등의 기능을 위해 설정이 복잡할 수 있음

3. 웹 서버를 통한 로드 밸런싱을 사용하는 이유

1) 애플리케이션 레벨의 세밀한 제어

  • L4 스위치에서는 네트워크 트래픽만을 기반으로 단순하게 분산하지만, 웹 서버를 통한 리버스 프록시에서는 HTTP 헤더, 쿠키, URL 경로와 같은 정보를 기반으로 세밀한 트래픽 관리가 가능
  • 예를 들어, 웹 서버는 사용자의 세션을 유지하거나, 특정 API 경로에 따라 다른 서버로 트래픽을 분산하는 등의 작업을 수행할 수 있음

2) SSL/TLS 처리

  • 웹 서버는 SSL 종료를 처리할 수 있음
  • L4 스위치는 단순히 트래픽을 분산하지만, Nginx/Apache와 같은 웹 서버는 SSL/TLS 암호화를 처리한 후, 내부 네트워크에서는 HTTP로 통신할 수 있음
  • 이 방식은 보안과 성능을 모두 고려한 설계입니다.

3) 복잡한 로드 밸런싱 전략 지원

  • 웹 서버는 세션 기반 로드 밸런싱, 라운드 로빈이나 최소 연결 방식 등의 다양한 로드 밸런싱 알고리즘을 지원
  • 이를 통해 웹 애플리케이션에서 요구하는 다양한 로드 분산 정책을 구현할 수 있음

4) 유연성

  • 웹 서버는 모니터링 및 로깅 기능도 함께 제공하여, 애플리케이션 상태를 추적하고 트래픽 분산을 유연하게 조정할 수 있음
  • 또한, 새로운 서버 추가 시 더 간단하게 확장성을 제공할 수 있음

4. 결론

  • L4 스위치는 성능과 네트워크 기반의 단순한 트래픽 분산에 강점이 있음
  • 반면에 **리버스 프록시 웹 서버(Nginx, Apache)**는 애플리케이션 계층에서 HTTP 요청을 세밀하게 처리하고, SSL 종료와 세션 관리를 지원할 수 있는 강력한 기능을 제공
  • 따라서, 단순한 네트워크 트래픽 분산이 필요한 경우에는 L4 스위치를, HTTP 요청 기반의 세밀한 제어와 SSL 처리가 필요한 경우에는 리버스 프록시 웹 서버를 사용하는 것이 일반적

마이크로서비스 아키텍처란

마이크로서비스 아키텍처(Microservices Architecture)란?

  • **마이크로서비스 아키텍처(Microservices Architecture)**는 최근 많은 기업과 개발 팀에서 선호하는 소프트웨어 설계 방식
  • 이 아키텍처는 대규모 애플리케이션을 독립적인 작은 서비스 단위로 나누어 개발하고 배포하는 방식으로, 특히 확장성과 유지 보수성이 중요한 프로젝트에서 강력한 장점을 제공
  • 이 포스팅에서는 마이크로서비스 아키텍처가 무엇인지, 어떻게 동작하는지, 그리고 이를 도입함으로써 얻을 수 있는 장점과 고려해야 할 점을 간단하게 정리해봄

1. 마이크로서비스 아키텍처란?

  • 마이크로서비스 아키텍처는 애플리케이션을 여러 개의 작은 독립적인 서비스로 나누어 개발하는 소프트웨어 설계 패턴
  • 각 서비스는 하나의 독립된 기능을 담당하며, 서로 간에 느슨하게 결합되어 있으며, 이러한 서비스들은 각기 다른 프로그래밍 언어, 데이터베이스, 그리고 독립적인 배포 주기를 가질 수 있음

주요 개념

  • 작고 독립된 서비스

    • 애플리케이션이 여러 개의 작은 서비스로 분리됨
    • 각 서비스는 하나의 기능이나 비즈니스 로직을 담당

  • 독립적인 배포

    • 각 서비스는 독립적으로 배포 및 업데이트가 가능
    • 이는 애플리케이션의 다른 부분에 영향을 주지 않고 서비스별로 빠르게 변화할 수 있다는 뜻

  • 자율적인 개발 팀

    • 각 서비스는 자율적으로 개발되며, 각 팀은 특정 서비스만을 책임을 짐
    • 팀 간 의존성을 줄이고 더 빠른 개발이 가능

예시

예를 들어, 온라인 쇼핑몰 애플리케이션이 있다고 가정하면, 이 애플리케이션은 여러 서비스로 나눌 수 있음

  • 사용자 서비스: 사용자 관리(로그인, 회원가입 등)를 담당
  • 상품 서비스: 상품 목록을 관리하고, 상품 정보를 제공
  • 주문 서비스: 주문 처리 및 결제 관리
  • 배송 서비스: 배송 추적 및 관리

이 각각의 서비스는 독립적으로 개발되고, 독립적으로 배포될 수 있음

2. 마이크로서비스 아키텍처의 장점

1) 확장성 (Scalability)

  • 마이크로서비스는 각 서비스가 독립적으로 확장될 수 있음
  • 특정 서비스에 트래픽이 집중되면, 해당 서비스만 추가 서버를 배포하거나 자원을 늘릴 수 있음
    • 이는 전체 애플리케이션을 확장할 필요 없이, 필요한 부분만 확장할 수 있어 자원 효율성을 높일 수 있음
  • 쇼핑몰에서 주문 서비스에 트래픽이 몰리는 경우, 주문 서비스만 확장하고 다른 서비스는 그대로 유지할 수 있음

2) 유연한 기술 선택 (Technology Flexibility)

  • 마이크로서비스 아키텍처에서는 각 서비스가 독립적이기 때문에 각 서비스에 적합한 기술 스택을 선택할 수 있음
  • 예를 들어, 일부 서비스는 Python으로 개발하고, 다른 서비스는 Java나 Node.js로 개발할 수 있음
  • 또한 각 서비스는 자신에게 가장 적합한 데이터베이스를 선택할 수 있음
  • 상품 관리 서비스는 MySQL, 사용자 인증 서비스는 NoSQL 데이터베이스를 사용할 수 있음

3) 독립적인 배포 및 업데이트

  • 마이크로서비스 아키텍처는 각 서비스가 독립적으로 배포되기 때문에, 한 서비스의 업데이트가 다른 서비스에 영향을 미치지 않음
  • 이를 통해 더 빠르고 자주 업데이트할 수 있으며, 새로운 기능을 추가하거나 버그를 수정할 때 더 안정적으로 배포할 수 있음
  • 주문 처리 기능에 버그가 발생해도, 사용자 인증 서비스는 문제없이 계속 동작할 수 있음

4) 자율적인 개발 및 팀 구성

  • 각 마이크로서비스는 독립적인 개발 팀이 관리할 수 있음
  • 각 팀은 자신이 맡은 서비스에 집중하여 개발하고 배포할 수 있으며, 다른 팀과의 의존성이 줄어들기 때문에 개발 속도가 빨라짐
  • 또한 조직 내 여러 팀이 병렬로 일할 수 있어 효율성이 증가
  • 하나의 팀이 상품 관리 서비스를, 다른 팀이 사용자 관리 서비스를 개발하면서 동시에 배포할 수 있음

5) 장애 격리 (Fault Isolation)

  • 마이크로서비스 아키텍처에서는 특정 서비스에 문제가 생기더라도, 다른 서비스에는 영향을 미치지 않음
  • 문제가 발생한 서비스만을 재시작하거나 복구하면 되기 때문에, 전체 애플리케이션의 가용성이 유지됨
  • 주문 서비스가 일시적으로 장애가 발생하더라도, 사용자 관리나 상품 조회 서비스는 정상적으로 동작할 수 있음

3. 마이크로서비스 아키텍처의 단점과 도전 과제

1) 복잡한 서비스 간 통신

  • 마이크로서비스는 여러 서비스로 분리되므로, 서비스 간 통신이 복잡해질 수 있음
  • 각 서비스가 서로 데이터를 주고받는 과정에서 네트워크 레이턴시, 장애 처리, 통신 실패 등 다양한 문제가 발생할 수 있음
  • 해결책: 서비스 간의 통신을 REST API, gRPC, 메시지 큐(예: RabbitMQ, Kafka) 등을 통해 효율적으로 처리할 수 있음

2) 분산 시스템 관리의 복잡성

  • 마이크로서비스는 서비스가 많아지면, 분산 시스템을 관리하는 데 복잡도가 증가함
  • 여러 서비스의 로그를 관리하고, 오류를 추적하는 것이 어려워질 수 있음
  • 각 서비스가 독립적으로 동작하기 때문에, 서비스 간 데이터 일관성을 유지하는 것이 중요한 과제
  • 해결책: 로깅 및 모니터링 시스템(예: ELK Stack, Prometheus, Grafana)을 사용하여 모든 서비스를 통합적으로 모니터링하고, 분산 트레이싱(예: Jaeger, Zipkin)을 통해 각 서비스 간의 요청 흐름을 추적할 수 있음

3) 데이터 관리의 복잡성

  • 각 서비스가 자신의 독립적인 데이터베이스를 가질 경우, 서비스 간의 데이터 일관성을 유지하는 것이 어려울 수 있음
  • 특히 여러 서비스가 동일한 데이터에 접근해야 하는 경우, 이를 효율적으로 관리하는 것이 도전 과제가 될 수 있음
  • 해결책: **이벤트 소싱(Event Sourcing)**이나 CQRS(Command Query Responsibility Segregation) 패턴을 통해 데이터 동기화를 관리하고, 서비스 간의 데이터 의존성을 최소화하는 설계가 필요

4) 배포 및 테스트의 어려움

  • 마이크로서비스는 여러 개의 서비스로 구성되어 있기 때문에, 전체 시스템의 통합 테스트가 복잡할 수 있음
  • 또한 각 서비스를 독립적으로 배포하는 과정에서 배포 파이프라인을 관리하는 것도 도전 과제
  • 해결책: CI/CD(지속적 통합/지속적 배포) 파이프라인을 구축하여 각 서비스를 자동으로 테스트하고 배포할 수 있는 환경을 조성하는 것이 중요

4. 마이크로서비스 아키텍처를 위한 주요 기술 스택

1) API 게이트웨이 (API Gateway)

  • API 게이트웨이는 여러 마이크로서비스에 대한 클라이언트 요청을 중앙에서 관리하는 역할을 함
  • API 게이트웨이를 통해 서비스 간 통신을 간편하게 할 수 있고, 인증, 로깅, 요청 필터링 등의 기능도 제공할 수 있음
  • 대표적인 API 게이트웨이: Kong, NGINX, AWS API Gateway

2) **컨테이너(Container)**와 오케스트레이션

  • 마이크로서비스는 컨테이너 기반으로 배포되는 경우가 많음

  • Docker와 같은 컨테이너 기술을 사용하면 서비스 간의 종속성을 분리하고, 독립적인 환경에서 배포 및 실행할 수 있음

  • 여러 컨테이너를 효율적으로 관리하기 위해 Kubernetes와 같은 오케스트레이션 도구가 필요

    • 컨테이너 도구: Docker

    • 오케스트레이션 도구: Kubernetes, Docker Swarm

3) 서비스 메쉬(Service Mesh)

  • 서비스 메쉬는 마이크로서비스 간의 통신을 관리하고, 보안, 모니터링, 트래픽 관리를 자동화하는 기술
  • 이를 통해 서비스 간의 통신이 더 효율적으로 이루어질 수 있으며, 네트워크 문제가 발생할 때에도 이를 쉽게 해결할 수 있음
  • 서비스 메쉬 도구: Istio, Linkerd

5. 마이크로서비스 아키텍처의 활용 사례

1) Netflix

  • Netflix는 마이크로서비스 아키텍처를 가장 성공적으로 도입한 기업 중 하나
  • Netflix는 다양한 스트리밍 기능을 담당하는 서비스를 개별 마이크로서비스로 분리하고, 트래픽 급증에 대비해 자동 확장 기능을 사용
  • 이를 통해 전 세계적으로 안정적인 스트리밍 서비스를 제공할 수 있음

2) Amazon

  • Amazon 역시 마이크로서비스 아키텍처의 대표적인 성공 사례
  • Amazon은 주문 처리, 결제, 배송 추적 등 여러 기능을 독립적인 서비스로 나누어 관리하고 있으며, 이를 통해 각 서비스가 독립적으로 확장 및 배포될 수 있음
  • 이로 인해 대규모 트래픽에도 유연하게 대응

결론

  • **마이크로서비스 아키텍처(Microservices Architecture)**는 대규모 애플리케이션을 여러 개의 작은 서비스로 나누어 개발하고 배포하는 효율적인 소프트웨어 설계 방식

  • 이 아키텍처는 확장성, 유연성, 독립적인 배포 등의 강력한 장점을 제공하여, 특히 빠르게 변화하는 대규모 시스템에 적합

  • 하지만 그만큼 복잡한 통신 관리, 데이터 일관성 유지 등의 도전 과제도 존재

  • 이를 해결하기 위해 API 게이트웨이, 컨테이너 오케스트레이션, 서비스 메쉬와 같은 기술을 함께 사용하여 마이크로서비스의 장점을 극대화할 수 있음

  • 마이크로서비스 아키텍처는 성공적인 대규모 애플리케이션을 구축하는 데 매우 유용한 도구이지만, 적절한 설계와 관리가 필수적

  • 이를 통해 더 유연하고 확장 가능한 시스템을 만들 수 있음

Stateless vs Stateful

개요

  • Stateless와 Stateful은 웹 개발에서 상태 관리의 차이를 설명하는 중요한 개념
  • 이 두 용어는 클라이언트와 서버 간의 데이터 교환 방식과 상태 관리 방식에서 큰 차이가 있음

1. Stateless(상태 비저장)란?

  • Stateless는 말 그대로 상태를 저장하지 않는 것을 의미
  • 서버와 클라이언트 간의 각 요청이 서로 독립적이며, 서버가 클라이언트의 상태 정보를 저장하지 않는 상황을 설명
  • Stateless 시스템에서는 각 요청이 이전 요청과 무관하게 처리

예시: HTTP 프로토콜**

  • HTTP는 Stateless 프로토콜로, 웹 서버는 각 요청을 독립적으로 처리
  • 예를 들어, 사용자가 첫 번째 페이지를 요청한 뒤, 다른 페이지를 요청할 때, 서버는 사용자가 이전에 어떤 페이지를 요청했는지 기억하지 않고 새로운 요청으로 처리

2. Stateful(상태 저장)란?

  • ** Stateful은 서버가 클라이언트의 상태를 기억하고 저장하는 시스템을 의미
  • 즉, 각 요청이 독립적이지 않고, 서버가 이전 요청에서 발생한 상태 정보를 유지하여 다음 요청을 처리할 때 그 정보를 참고하는 방식

예시: 온라인 게임 서버

  • 온라인 게임 서버는 Stateful 시스템
  • 사용자가 게임을 시작할 때 서버는 사용자의 세션 정보를 기억하며, 다음 요청 시 이 정보를 기반으로 사용자가 어떤 상태에 있는지 기억하고 그에 맞는 처리를 할 수 있음

2. Stateless와 Stateful의 차이점

  • Stateless와 반대되는 개념으로는 **Stateful(상태 저장)**이 있음
  • 두 개념을 비교하여 이해하면 Stateless의 의미를 좀 더 쉽게 이해할 수 있음

Stateless (상태 비저장)

  • 각 요청은 서버에 의해 독립적으로 처리되며, 이전 요청의 상태를 기억하지 않음
  • 서버는 클라이언트의 상태를 저장하지 않기 때문에 요청마다 필요한 정보를 클라이언트가 매번 전달해야 함
  • 대표적인 예시: HTTP 프로토콜, REST API

Stateful (상태 저장)

  • 각 요청은 서버가 이전 요청의 상태를 기억하고, 그 상태에 기반하여 처리됨
  • 서버는 클라이언트와의 세션을 유지하며, 클라이언트의 상태를 지속적으로 저장하고 관리
  • 대표적인 예시: 온라인 게임 서버, 채팅 애플리케이션
구분 Stateless (상태 비저장) Stateful (상태 저장)
상태 관리 각 요청은 독립적이며, 상태 정보를 저장하지 않음 이전 요청의 상태를 기억하고, 상태를 저장함
클라이언트 매번 요청할 때 필요한 정보를 전달해야 함 세션을 유지하며, 요청에 필요한 상태 정보를 서버가 기억함
예시 HTTP, REST API, 서버리스 아키텍처 온라인 게임, 채팅 애플리케이션

3. Stateless vs Stateful: 차이점

구분 Stateless(상태 비저장) Stateful(상태 저장)

요청 처리 방식 각 요청은 독립적이며, 이전 요청과 무관하게 처리됨 각 요청은 이전 요청의 상태에 기반하여 처리됨
클라이언트 요청마다 필요한 모든 정보를 서버에 전달해야 함 클라이언트는 상태 정보를 서버가 유지하기 때문에 일부 정보를 생략 가능
확장성 높음: 서버에 상태를 저장하지 않으므로, 쉽게 확장 가능 낮음: 상태를 저장해야 하므로 서버 간 데이터 동기화 필요
서버 장애 시 서버 장애가 발생해도 다른 서버에서 요청을 처리할 수 있음 서버 장애 시 상태 정보를 잃을 수 있으며, 세션이 중단될 수 있음
예시 HTTP, REST API 온라인 게임, 채팅 애플리케이션

4. Stateless의 장점과 단점

장점

  1. 확장성

    • Stateless 시스템에서는 서버가 클라이언트의 상태를 저장하지 않으므로, 서버를 추가하거나 트래픽이 많아질 때 확장성이 높음
    • 트래픽이 급증하는 상황에서도 각 요청이 독립적이기 때문에 여러 서버에서 요청을 분산 처리할 수 있음

  2. 단순성

    • 서버가 상태를 관리하지 않기 때문에 시스템 설계가 더 단순, 서버는 요청을 처리하고, 응답을 보내기만 하면 됨
    • 전자 상거래 사이트에서 상품을 검색할 때, 서버는 이전 검색 내역을 기억할 필요 없이 각 요청에 대해 독립적으로 처리

  3. 안정성

    • 한 서버가 장애가 발생하더라도, 다른 서버가 요청을 처리할 수 있는 가능성이 높아짐, 이는 장애 복구에 유리

    • 서버 한 대가 다운되더라도, 요청은 다른 서버로 처리될 수 있어 서비스가 지속됨

단점

  1. 상태 관리의 불편함
    • Stateless 시스템에서는 상태를 서버가 저장하지 않으므로, 클라이언트가 매번 상태 정보를 전달해야 함
    • 이는 클라이언트 측의 부담을 증가시키고, 자주 상태 정보를 갱신해야 하는 애플리케이션에서는 비효율적일 수 있음
    • 쇼핑몰에서 장바구니 상태를 유지하려면, 클라이언트는 매번 장바구니 정보를 요청에 포함해야 함
  2. 인증의 복잡성
    • 인증 상태를 저장하지 않기 때문에, 사용자가 로그인한 상태를 기억하려면 클라이언트가 매번 인증 정보를 포함하여 요청을 보내야 함
    • 사용자가 로그인을 했더라도, 서버는 그 상태를 기억하지 않기 때문에 JWT와 같은 토큰을 요청마다 포함시켜야 함

5. Stateful의 장점과 단점

장점

  1. 상태 유지의 편리함

    • Stateful 시스템에서는 서버가 클라이언트의 상태를 기억하므로, 클라이언트가 매번 모든 정보를 포함해서 보내지 않아도 됨

    • 온라인 게임에서 사용자의 진행 상태를 저장하고 관리하여, 사용자는 게임을 다시 시작할 때 이전 상태를 그대로 이어서 플레이할 수 있음

  2. 연속성

    • Stateful 시스템은 요청 간에 연속성을 유지할 수 있어, 장기적인 세션 유지가 필요한 애플리케이션에서 유리
    • 채팅 애플리케이션에서, 사용자가 채팅을 하면서 세션을 유지해 이전 대화 내용에 접근할 수 있음

단점

  1. 확장성의 제한

    • 서버가 상태 정보를 유지하므로, 서버를 추가하거나 여러 서버에서 상태를 동기화하는 데 복잡성이 증가

    • 특히, 상태가 분산된 서버에 저장될 경우 데이터 동기화가 어려움

    • 상태를 유지하는 온라인 게임에서 사용자가 다른 서버로 옮겨갈 때, 상태 정보를 동기화하는 데 문제가 생길 수 있음

  2. 서버 장애의 영향

    • Stateful 시스템에서는 클라이언트의 상태가 특정 서버에 저장되기 때문에, 그 서버가 장애를 일으키면 상태 정보가 유실될 수 있음
    • 사용자가 쇼핑몰에서 로그인한 상태에서 서버가 다운되면, 세션 정보가 유실되어 사용자는 다시 로그인해야 할 수 있음

6. Stateless vs Stateful: 언제 어떤 방식을 사용할까?

  • Stateless 시스템은 확장성과 안정성이 중요한 웹 애플리케이션, 특히 REST API나 HTTP 기반 통신에서 자주 사용됨, 이 방식은 트래픽이 많아도 서버가 상태를 유지하지 않기 때문에 성능이 높고, 관리가 간편
  • Stateful 시스템은 연속적인 세션 유지가 중요한 온라인 게임이나 채팅 애플리케이션과 같은 곳에서 많이 사용됨, 상태를 지속적으로 유지해야 하는 경우, Stateful 아키텍처가 더 적합

6. Stateless에서 상태 관리는 어떻게 할까?

  • Stateless 아키텍처에서는 서버가 상태를 저장하지 않기 때문에, 상태 정보는 다른 방식으로 관리해야 함
  • 대표적인 방법은 **쿠키(Cookie)**나 **토큰(Token)**을 사용하는 방식입니다.

1) 쿠키(Cookie)

  • 쿠키는 클라이언트의 브라우저에 저장되는 작은 데이터 조각
  • 서버는 상태를 저장하지 않지만, 클라이언트 측에서 상태 정보를 쿠키로 저장하고, 이후 요청마다 쿠키를 서버로 전달하여 상태를 유지할 수 있음
  • 예시
    • 로그인 상태를 유지하기 위해 클라이언트는 로그인 성공 후 서버로부터 세션 ID가 포함된 쿠키를 받음
    • 이후 모든 요청에 이 쿠키를 포함하여 서버에 전달하면, 서버는 이 쿠키를 기반으로 사용자의 로그인 상태를 확인할 수 있음

2) 토큰(Token)

  • 토큰 기반 인증(예: JWT, OAuth)은 클라이언트가 로그인 후 인증 토큰을 받아 요청에 포함하여 서버에 전달하는 방식
  • 서버는 이 토큰을 검증하여 사용자의 인증 상태를 확인할 수 있음
  • 예시
    • 사용자가 로그인 후 JWT(Json Web Token)를 발급받으면, 모든 요청에 이 토큰을 포함하여 서버에 보냄
    • 서버는 토큰을 확인해 사용자의 인증 상태를 유지

결론

  • Stateless와 Stateful 시스템의 차이는 상태 정보의 관리에 있음
  • Stateless 시스템은 확장성과 단순성, 안정성이 장점이지만, 상태 정보를 매번 요청해야 하는 불편함이 있음
  • 반면 Stateful 시스템은 상태 정보를 유지할 수 있어 편리하지만, 확장성이 떨어지고 서버 장애에 취약할 수 있음
  • 각 애플리케이션의 요구사항에 맞춰 Stateless와 Stateful 아키텍처를 적절히 선택하여 사용해야 함
  • 특히, 대규모 트래픽 처리가 필요한 웹 애플리케이션에서는 Stateless 아키텍처가, 세션 유지가 필수인 애플리케이션에서는 Stateful 아키텍처가 적합한 선택이 될 수 있음

서버리스 아키텍처

서버리스 아키텍처(Serverless Architecture)란?

  • **서버리스 아키텍처(Serverless Architecture)**는 현대의 클라우드 컴퓨팅 환경에서 빠르게 인기를 얻고 있는 개념
  • ‘서버리스’라는 용어는 서버가 없다는 의미로 오해할 수 있지만, 실제로는 서버를 개발자가 직접 관리하지 않고도 애플리케이션을 실행할 수 있다는 의미
  • 이 포스팅에서는 서버리스 아키텍처가 무엇인지, 이를 도입하면 어떤 장점이 있는지, 그리고 서버리스 아키텍처를 활용하는 방법을 쉽게 설명

1. 서버리스 아키텍처(Serverless Architecture)란?

  • **서버리스(Serverless)**란 애플리케이션을 실행하기 위해 물리적 서버나 가상 서버를 직접 관리할 필요가 없는 클라우드 컴퓨팅 방식

  • 서버리스 아키텍처를 도입하면 서버 인프라의 프로비저닝(provisioning), 운영 및 관리가 클라우드 서비스 제공자(예: AWS, Google Cloud, Azure)에게 맡겨지며, 개발자는 코드 작성과 애플리케이션 로직 구현에만 집중할 수 있음

  • 즉, 서버리스 아키텍처에서는 개발자가 서버를 직접 설정하거나 관리하지 않으며, 클라우드 제공자가 자동으로 서버 리소스를 할당하고 운영

서버리스 아키텍처의 주요 특징

  • 자동 스케일링: 서버리스 환경에서는 트래픽이 증가하면 서버 자원이 자동으로 확장되며, 트래픽이 적으면 자동으로 축소됩니다.
  • 사용량 기반 과금: 서버리스 아키텍처에서는 애플리케이션이 실제로 실행된 시간만큼 비용이 부과됩니다. 즉, 애플리케이션이 사용되지 않을 때는 비용이 발생하지 않습니다.
  • 서버 관리 불필요: 개발자는 서버를 설정, 유지 보수할 필요가 없으며, 코드 작성 및 배포에만 집중할 수 있음

2. 서버리스 아키텍처의 구성 요소

  • 서버리스 아키텍처는 주로 두 가지 주요 구성 요소로 이루어짐

1) FaaS (Function as a Service)

  • FaaS는 서버리스 환경에서 애플리케이션의 비즈니스 로직을 함수 단위로 실행하는 방식
  • 개발자는 각 기능을 함수로 작성하고, 클라우드 제공자가 해당 함수를 실행, 관리, 확장하는 방식
  • 대표적인 예로는 AWS Lambda, Google Cloud Functions, Azure Functions 등이 있음
  • AWS Lambda: AWS에서 제공하는 서버리스 함수 실행 서비스로, 특정 이벤트(예: HTTP 요청, 데이터베이스 트리거) 발생 시 함수가 실행됨

FaaS의 특징

  • 이벤트 기반: HTTP 요청, 파일 업로드, 데이터베이스 업데이트 등 특정 이벤트가 발생할 때 함수가 실행됨
  • 짧은 실행 시간: 각 함수는 짧은 시간 동안 실행되고, 실행이 완료되면 자동으로 종료됨
  • 자동 스케일링: 트래픽이 많아지면 함수가 자동으로 더 많은 인스턴스에서 실행되며, 성능이 유지됨

FaaS의 예시

2) BaaS (Backend as a Service)

  • BaaS는 클라우드 제공자가 제공하는 백엔드 서비스를 활용하여 서버나 데이터베이스를 직접 관리하지 않고도 필요한 기능을 사용할 수 있게 해줌
  • BaaS 서비스에는 데이터베이스 관리, 사용자 인증, 파일 저장소 등이 포함될 수 있음
  • 대표적인 BaaS 서비스로는 Firebase, AWS Amplify 등이 있습니다.

BaaS에서 제공하는 주요 기능

  • 데이터베이스 관리: 클라우드에서 제공하는 데이터베이스 서비스를 사용하여, 직접 서버를 설정하지 않고도 데이터를 관리할 수 있음 (예: Firebase Firestore, AWS DynamoDB)
  • 사용자 인증(Authentication): OAuth, JWT 등을 사용하여 손쉽게 사용자 인증 기능을 구현할 수 있음 (예: Firebase Authentication)
  • 파일 저장 및 배포: 클라우드 스토리지 서비스(예: AWS S3, Google Cloud Storage)를 통해 파일을 저장하고 배포할 수 있음

3. 서버리스 아키텍처의 장점

1) 비용 효율성

  • 서버리스 아키텍처의 가장 큰 장점 중 하나는 사용한 만큼만 지불하는 과금 방식
  • 전통적인 서버 방식에서는 서버를 항상 켜놓고 유지해야 하기 때문에 유휴 시간에도 비용이 발생하지만, 서버리스 아키텍처에서는 코드가 실행된 시간만큼만 비용을 지불하면 됨
  • 이는 특히 트래픽이 불규칙한 애플리케이션에서 큰 비용 절감 효과를 제공

2) 자동 확장성

  • 서버리스 아키텍처에서는 트래픽의 변화에 따라 서버 리소스가 자동으로 확장 됨
  • 개발자가 직접 서버 자원을 모니터링하거나 확장하지 않아도, 클라우드 제공자가 자동으로 애플리케이션의 확장성을 보장
  • 이를 통해 애플리케이션이 갑자기 트래픽이 증가할 때도 성능 저하 없이 대응할 수 있음

3) 서버 관리 불필요

  • 서버리스 아키텍처에서는 서버의 설정, 유지 보수, 보안 패치 등을 직접 관리할 필요가 없음
  • 모든 서버 관리 작업은 클라우드 제공자가 처리하며, 개발자는 애플리케이션 개발에만 집중할 수 있음
  • 이는 DevOps의 부담을 줄이고, 더 빠르게 서비스를 개발하고 배포할 수 있도록 함

4) 신속한 개발 및 배포

  • 서버리스 환경에서는 각 기능을 독립적으로 개발하고 배포할 수 있기 때문에, 빠른 배포 주기를 유지할 수 있음
  • 특히 스타트업이나 신속한 개발이 필요한 프로젝트에서 유리

4. 서버리스 아키텍처의 단점 및 고려 사항

1) 실행 시간 제한

  • 서버리스 함수(FaaS)는 짧은 실행 시간을 갖음. 예를 들어, AWS Lambda는 기본적으로 15분의 실행 시간 제한이 있음
  • 따라서 장시간 실행되는 작업이나 복잡한 배치 작업에는 적합하지 않을 수 있음

2) 콜드 스타트 문제

  • 서버리스 함수는 요청이 발생할 때 새로운 인스턴스를 생성하여 함수가 실행되는데, 이 과정에서 **콜드 스타트(cold start)**라고 불리는 지연 시간이 발생할 수 있음
  • 이는 특히 짧은 응답 시간이 중요한 애플리케이션에서 성능 문제로 이어질 수 있음

3) 복잡한 모니터링 및 디버깅

  • 서버리스 아키텍처에서는 서버 관리가 클라우드 제공자에게 맡겨지기 때문에, 문제가 발생했을 때 모니터링 및 디버깅이 더 복잡할 수 있음
  • 클라우드 제공자가 제공하는 로깅 및 모니터링 도구를 사용해야 하며, 네트워크 트래픽을 직접 관리하기 어려움

4) 벤더 종속성

  • 서버리스 환경을 제공하는 클라우드 제공자(AWS, Google Cloud, Azure)마다 서버리스 서비스의 API와 동작 방식이 다름
  • 이로 인해 특정 제공자에 종속될 수 있으며, 나중에 다른 클라우드 제공자로 이전하기 어렵다는 벤더 락인(Vendor Lock-in) 문제가 발생할 수 있음

5. 서버리스 아키텍처 활용 사례

1) 이벤트 기반 애플리케이션

  • 서버리스 함수는 이벤트 기반으로 동작하기 때문에, 사용자의 특정 요청이나 이벤트가 발생했을 때만 실행되는 애플리케이션에 적합
  • 예를 들어, 파일 업로드, 데이터베이스 변경 등을 트리거로 함수가 실행될 수 있음

2) 마이크로서비스 아키텍처

  • 서버리스 아키텍처는 각 기능을 작은 단위의 서비스로 나누는 마이크로서비스 아키텍처와 잘 맞음
  • 각 마이크로서비스는 독립적으로 배포 및 확장될 수 있으며, 서버리스로 운영하면 자동 확장성도 확보할 수 있음

3) 비정기적 트래픽 처리

  • 서버리스는 트래픽이 일정하지 않은 애플리케이션에 적합
  • 예를 들어, 이벤트성 사이트나 월별 정산 작업처럼 일정하지 않은 트래픽이 발생하는 경우 서버리스 아키텍처를 사용하면 비용을 절감하고 성능을 보장할 수 있음

결론

  • **서버리스 아키텍처(Serverless Architecture)**는 서버 관리를 최소화하고 개발에만 집중할 수 있게 해주는 효율적인 클라우드 컴퓨팅 방식

  • 특히 사용량 기반 과금 모델과 자동 확장 기능을 제공하여 비용 절감과 확장성을 동시에 확보할 수 있음

  • 서버리스 아키텍처를 도입하면 애플리케이션 개발 및 배포 속도를 크게 향상시킬 수 있지만, 콜드 스타트 문제, 실행 시간 제한, 벤더 종속성 등 몇 가지 단점을 고려해야 함

  • 이를 적절히 이해하고 활용한다면, 서버리스 아키텍처는 클라우드 네이티브 애플리케이션을 위한 훌륭한 선택지가 될 수 있음

리버스 프록시

리버스 프록시(Reverse Proxy)란? 쉽게 이해하는 개념과 활용

  • 인터넷에서 웹 서버나 애플리케이션 서버의 트래픽을 효율적으로 관리하고 보안을 강화하는 중요한 기술 중 하나가 리버스 프록시(Reverse Proxy)
  • 복잡한 웹 시스템에서 자주 사용되는 핵심 기술이지만, 처음 접하는 사람에게는 다소 생소할 수 있기에, 이 포스팅에서는 리버스 프록시가 무엇인지, 어떻게 동작하는지, 그리고 이를 활용하는 주요 이점들을 쉽게 설명

1. 리버스 프록시(Reverse Proxy)란?

  • 클라이언트(사용자)가 요청한 트래픽을 여러 서버로 분배하거나 필터링하는 중간 역할을 하는 서버

  • 쉽게 말해, 클라이언트가 웹사이트에 요청을 보내면, 리버스 프록시가 이 요청을 대신 받아 내부 서버(웹 서버, 애플리케이션 서버)에 전달하고, 응답을 다시 클라이언트에게 돌려줌

  • 리버스 프록시는 서버 앞에 위치하여 클라이언트와 서버 간의 모든 트래픽을 중재하는 역할을 하며, 서버의 IP 주소나 내부 구조를 클라이언트에게 노출하지 않고 보호할 수 있음

리버스 프록시의 동작 구조

  1. 클라이언트가 특정 웹사이트에 접속하려고 요청을 보냄
  2. 요청은 리버스 프록시 서버로 전달
  3. 리버스 프록시 서버는 내부 서버 중 하나(웹 서버나 애플리케이션 서버)로 요청을 전달
  4. 내부 서버는 요청을 처리하고, 그 결과를 리버스 프록시 서버에 돌려줌
  5. 리버스 프록시 서버는 클라이언트에게 응답을 보냄

예시

  • 클라이언트가 www.example.com에 접속하려고 할 때, 클라이언트는 실제 서버의 IP 주소나 구조를 모르고, 리버스 프록시 서버가 요청을 대신 받아 처리한 후 클라이언트에게 결과를 전달

2. 리버스 프록시의 주요 역할

리버스 프록시는 단순히 요청을 전달하는 역할뿐만 아니라, 다양한 기능을 통해 웹 서비스의 성능을 향상시키고, 보안 및 확장성을 개선할 수 있음. 여기서 리버스 프록시의 주요 기능과 장점을 살펴보자

1) 로드 밸런싱 (Load Balancing)

  • 리버스 프록시는 로드 밸런서의 역할을 할 수 있음
  • 여러 대의 서버에 클라이언트 요청을 분배함으로써 트래픽을 균등하게 나누고, 한 서버에 과부하가 걸리지 않도록 도와줌
  • 예시
    • 리버스 프록시 서버가 3개의 웹 서버(A, B, C) 중 한 서버에만 트래픽이 몰리지 않게 분산하여 트래픽을 처리
    • 이로 인해 서버의 부하를 줄이고, 안정적인 서비스 제공이 가능

2) 보안 강화

  • 리버스 프록시는 서버의 IP 주소와 내부 구조를 감추는 역할을 하기 때문에, 외부 공격자가 내부 서버에 직접 접근하기 어려움
  • 또한 SSL 인증서를 리버스 프록시 서버에 설치하여, HTTPS 요청을 처리하고 내부 서버와의 통신은 HTTP로 설정하여 보안을 강화할 수 있음
  • 방화벽 역할: 리버스 프록시는 방화벽처럼 동작하여, 원치 않는 요청이나 악의적인 공격을 차단할 수 있습니다.

3) 캐싱(Cache)

  • 리버스 프록시는 정적 콘텐츠(이미지, CSS, JavaScript 파일 등)를 캐시하여 클라이언트가 동일한 요청을 보낼 때 더 빠르게 응답할 수 있음
  • 이를 통해 서버 부하를 줄이고, 응답 시간을 개선할 수 있음
  • 예시
    • 클라이언트가 자주 요청하는 이미지 파일이나 CSS 파일을 리버스 프록시 서버가 캐시하여, 서버에 재요청하지 않고 캐시된 데이터를 클라이언트에 반환
    • 이렇게 하면 속도가 빨라지고 서버 리소스가 절약됨

4) SSL 종료(SSL Termination)

  • 리버스 프록시는 SSL 종료 기능을 통해 HTTPS 요청을 처리할 수 있음
  • 리버스 프록시가 클라이언트로부터 HTTPS 요청을 받아 SSL/TLS 암호화를 해제한 후, 내부 서버에는 HTTP 요청으로 전달할 수 있음, 이 방법을 통해 내부 서버는 복잡한 SSL 처리를 하지 않고도 안전하게 운영될 수 있음
  • 예시
    • Nginx 리버스 프록시 서버가 HTTPS 요청을 처리하고, 내부 서버는 HTTP로만 통신하게 하여 부하를 줄이면서 보안성을 유지

5) 다중 서버 통합 (Server Aggregation)

  • 리버스 프록시는 여러 서버의 응답을 통합하여 클라이언트에게 전달할 수 있음
  • 예를 들어, 사용자가 서로 다른 서비스에 대한 요청을 보낼 때, 리버스 프록시는 각 요청을 적절한 서버로 분배하고, 그 결과를 하나의 응답으로 통합해 반환할 수 있음
  • 예시
    • 리버스 프록시 서버가 API 서버, 이미지 서버, 데이터베이스 서버 등 여러 서버의 응답을 통합해 클라이언트에게 하나의 웹 페이지로 반환

3. 리버스 프록시의 활용 예시

리버스 프록시는 여러 환경에서 유용하게 사용될 수 있으며, 그 중 몇 가지 대표적인 예시는 다음과 같음

1) Nginx를 사용한 리버스 프록시 설정

  • Nginx는 가장 널리 사용되는 리버스 프록시 서버 중 하나로, 정적 파일 제공, 로드 밸런싱, SSL 처리 등 다양한 기능을 제공

위 설정은 Nginx가 클라이언트 요청을 받아 localhost:5000에서 실행 중인 내부 서버로 요청을 전달하는 리버스 프록시 역할을 하게됨

2) 클라우드 환경에서의 리버스 프록시 사용

  • 리버스 프록시는 클라우드 환경에서 다중 서버 관리와 확장성을 위해 자주 사용됨
  • 특히 AWS, GCP와 같은 클라우드 플랫폼에서는 로드 밸런싱, 보안 및 자동 확장을 위해 리버스 프록시를 사용하여 트래픽을 효율적으로 관리

4. 리버스 프록시와 포워드 프록시(Forward Proxy)의 차이점

리버스 프록시와 포워드 프록시(Forward Proxy)는 개념이 비슷하지만, 역할과 동작 방식이 다름

  • 리버스 프록시는 클라이언트의 요청을 받아 내부 서버로 전달하고, 서버의 응답을 다시 클라이언트에게 돌려주는 역할을 함, 주로 서버 보호 및 트래픽 관리에 사용됩니다.
  • 포워드 프록시는 클라이언트가 직접 요청할 수 없는 외부 서버로의 요청을 대신 보내주는 역할을 함, 주로 사용자의 익명성을 보호하거나 차단된 콘텐츠에 접근할 때 사용됨
구분 리버스 프록시(Reverse Proxy) 포워드 프록시(Forward Proxy)
동작 대상 서버 보호 및 내부 서버로의 요청 중계 클라이언트 보호 및 외부 서버로의 요청 중계
목적 서버의 로드 밸런싱, 보안 강화, 캐싱 처리 클라이언트의 익명성 보호, 네트워크 차단 우회
사용 예시 Nginx, Apache, AWS ELB VPN, 회사나 학교에서 특정 웹사이트 차단 우회

5. 리버스 프록시의 장점 요약

  • 보안 강화: 서버 IP 및 내부 구조 보호, SSL 종료 제공
  • 로드 밸런싱: 트래픽 분산으로 성능 및 안정성 향상
  • 캐싱 제공: 정적 파일 캐시를 통해 빠른 응답 속도 제공
  • SSL 처리: HTTPS 요청을 처리하여 보안성 강화
  • 서버 확장성: 여러 서버를 통합하고 분산하여 더 많은 트래픽 처리 가능

결론

  • 리버스 프록시(Reverse Proxy)는 웹 서비스의 성능과 보안을 크게 향상시킬 수 있는 강력한 기술
  • 특히, 트래픽을 관리하고 보안을 강화하는 데 탁월한 역할을 함
  • Nginx와 같은 리버스 프록시 서버를 올바르게 설정하면, 서비스의 확장성과 안정성을 확보하면서도, 더 나은 사용자 경험을 제공할 수 있음
  • 리버스 프록시는 로드 밸런싱, 캐싱, 보안, SSL 처리 등 다양한 기능을 제공하므로, 대규모 웹 서비스나 클라우드 환경에서 필수적인 도구로 활용되고 있음

웹서버와 WAS 차이

1. 웹 서버 (Web Server)

  • 기본 역할

    • 웹 서버는 정적 콘텐츠(HTML, CSS, JavaScript, 이미지 파일 등)를 클라이언트(주로 브라우저)에 제공하는 역할을 함
    • 정적 콘텐츠: 변경되지 않는 파일들을 그대로 클라이언트에 전송
    • 대표적인 웹 서버로는 Nginx, Apache HTTP Server가 있음

  • 동작 방식

    • 클라이언트(브라우저)가 요청을 보내면, 웹 서버는 요청된 리소스(파일)를 찾아서 클라이언트에게 전달
    • 예를 들어, 사용자가 웹 페이지를 요청하면, 웹 서버는 해당 페이지의 HTML 파일을 클라이언트에 전송

  • 동적 콘텐츠 처리

    • 웹 서버는 동적 콘텐츠(사용자 요청에 따라 실시간으로 생성되는 데이터)는 직접 처리하지 못함
    • 동적 요청은 WAS나 애플리케이션 서버로 전달되어야 함
    • 동적 콘텐츠는 데이터베이스와 상호작용하거나 사용자 입력에 따라 변하는 데이터를 생성하는 작업을 의미

  • 주요 특징

    • 정적 파일(HTML, CSS, 이미지) 제공
    • HTTP 요청 및 응답을 처리
    • 동적 요청은 직접 처리하지 않고, WAS로 전달

2. WAS (Web Application Server)

  • 기본 역할

    • WAS는 동적 콘텐츠를 처리하는 서버
    • 웹 서버가 처리하지 못하는 동적 요청(데이터베이스 연결, 비즈니스 로직 처리 등)을 처리하고, 동적 웹 애플리케이션을 실행하는 역할을 함
    • 동적 콘텐츠: 사용자의 요청에 따라 실시간으로 생성되는 콘텐츠(예: 데이터베이스 조회, 사용자 세션 처리, 트랜잭션 등)를 생성하여 클라이언트에게 전달

  • 동작 방식

    • 클라이언트 요청 중에서 동적인 웹 애플리케이션 처리 요청이 있을 경우, WAS는 이를 처리하고 그 결과를 웹 서버를 통해 클라이언트에게 전달
    • 예를 들어, 사용자가 로그인 요청을 하면, WAS는 데이터베이스와 통신하여 해당 사용자의 정보를 확인하고 그 결과를 클라이언트에게 전달

  • 예시

    • Tomcat, JBoss, WebLogic 등이 대표적인 WAS
    • 이러한 WAS는 서블릿 컨테이너를 사용하여 자바 기반의 웹 애플리케이션을 구동하거나, PHP, Python, Ruby와 같은 언어로 작성된 웹 애플리케이션을 처리

  • 주요 특징

    • 비즈니스 로직을 처리하는 역할
    • 동적 웹 애플리케이션 실행
    • 세션 관리, 트랜잭션 처리, 데이터베이스와 상호작용
    • WAS는 자체적으로 웹 서버 기능을 포함할 수 있으므로, 정적 콘텐츠도 처리 가능하지만, 보통은 웹 서버(Nginx, Apache)와 함께 사용

3. 웹 서버와 WAS의 차이

구분 웹 서버 (Web Server) WAS (Web Application Server)
기본 역할 정적 콘텐츠(HTML, CSS, 이미지 등)를 제공 동적 콘텐츠(데이터베이스, 비즈니스 로직)를 처리
정적/동적 처리 정적 파일을 처리 (동적 파일은 WAS로 전달) 동적 웹 애플리케이션 실행 및 비즈니스 로직 처리
예시 Nginx, Apache HTTP Server Tomcat, JBoss, WebLogic
기술 스택 HTTP 프로토콜 처리, 정적 파일 서빙 서블릿, JSP, Spring 등 웹 애플리케이션 프레임워크 구동
주요 사용 사례 정적 웹 사이트, 파일 서버 동적 웹 애플리케이션 서버 (로그인 처리, 데이터베이스 접근 등)
부가 기능 로드 밸런싱, 캐싱 등 트랜잭션 관리, 세션 관리, 데이터베이스 연결

4. 웹 서버와 WAS의 협업

  • 일반적으로 웹 서버(Nginx, Apache)와 WAS(Tomcat 등)는 함께 사용됨

  • 웹 서버는 클라이언트로부터 모든 요청을 받아서, 정적 콘텐츠는 직접 처리하고, 동적 콘텐츠 요청은 WAS에 전달

  • 이와 같은 분리된 구조는 성능과 보안 면에서 더 효율적

    • 웹 서버는 정적 리소스를 빠르게 제공하고,

  • WAS는 복잡한 비즈니스 로직을 처리하며 데이터베이스와 상호작용

부록 1. 궁금했던 점

Spring Boot와 같은 현대적인 웹 프레임워크에서는 정적 페이지와 동적 페이지가 모두 함께 작성되며, 웹 서버를 별도로 사용하지 않고도 서버(Tomcat WAS 등)에서 처리할 수 있는데 웹 서버가 왜 필요할까?

Spring Boot와 WAS (Tomcat 등)의 기본 동작

  • Spring Boot는 기본적으로 내장형 WAS(예: Tomcat, Jetty)를 포함하고 있음

  • 따라서 별도의 외부 WAS를 설정하지 않고도 정적 페이지(HTML, CSS, 이미지 등)와 동적 페이지(Java 기반의 서버 로직)를 모두 처리할 수 있음

    • 정적 콘텐츠: Spring Boot 애플리케이션 내에서 src/main/resources/static 디렉터리에 정적 파일을 두면, Spring Boot 내장 WAS가 이를 처리

    • 동적 콘텐츠: 동적 요청은 Java 코드로 작성된 비즈니스 로직에 따라 처리되며, Spring MVC 패턴으로 요청-응답 흐름을 처리

따라서, Spring Boot 애플리케이션 하나만으로도 정적/동적 리소스 처리 및 배포가 모두 가능

그렇다면, 왜 웹 서버(Nginx, Apache)가 필요할까?

일반적으로 Spring Boot와 같은 애플리케이션 서버만으로도 배포가 가능하지만, 웹 서버를 추가로 사용하면 몇 가지 장점이 있기 때문에 대규모 환경이나 프로덕션에서는 여전히 웹 서버를 사용하는 경우가 많음

1) 정적 파일 제공의 효율성

  • Nginx나 Apache 같은 웹 서버는 정적 파일(HTML, CSS, 이미지 등)을 제공하는 데 매우 최적화되어 있음
  • 웹 서버는 정적 파일을 매우 빠르게 처리할 수 있으며, 정적 파일에 대해 캐싱, 압축 등의 기능을 제공하여 성능을 향상시킬 수 있는 반면
  • Spring Boot 내장 WAS는 이러한 정적 파일 서빙에 최적화되어 있지 않기 때문에, 대규모 트래픽 상황에서 웹 서버가 정적 콘텐츠를 처리하면 부하를 줄일 수 있음

2) 로드 밸런싱 및 리버스 프록시

  • Nginx나 Apache는 리버스 프록시로 설정하여, 여러 Spring Boot 애플리케이션 인스턴스 간에 로드 밸런싱을 할 수 있음

    • 이렇게 하면 트래픽을 균등하게 분배할 수 있어 서버 과부하를 방지할 수 있음

  • 예를 들어, Nginx가 클라이언트 요청을 받아 여러 WAS 인스턴스(Spring Boot)로 요청을 분산시키는 구조로 만들면, 하나의 서버가 과부하에 걸리는 것을 방지할 수 있음

3) 보안 (SSL/TLS 처리)

  • SSL 인증서를 설정하여 HTTPS를 적용할 때는 Nginx나 Apache와 같은 웹 서버를 사용하는 것이 일반적
  • 웹 서버는 리버스 프록시 역할을 하며, SSL/TLS를 통한 암호화된 통신(HTTPS)을 처리하는 데 있어 효율적이고 편리한 설정을 제공
  • 리버스 프록시 역할을 하는 웹 서버는 클라이언트로부터 받은 HTTPS 요청을 처리하고, 내부 네트워크에서는 HTTP로 WAS에 요청을 전달
  • 이렇게 구성하면 외부 통신에서는 SSL/TLS 암호화를 통해 보안성을 유지하면서도, 내부 통신은 복잡한 암호화 작업 없이 효율적으로 처리할 수 있음

4) 확장성 및 장애 대응

  • 웹 서버는 요청을 다수의 애플리케이션 서버(WAS)로 분산시킬 수 있음

    • 여러 Spring Boot 인스턴스를 실행하여 마치 하나의 서버처럼 운영할 수 있기 때문에, 확장성을 확보할 수 있음

  • 예를 들어, Nginx가 3개의 Spring Boot 서버에 트래픽을 균등하게 나눠주면, 한 서버에 문제가 생겨도 다른 서버에서 요청을 처리할 수 있어 고가용성을 유지할 수 있음

5) 캐싱 및 성능 최적화

  • Nginx나 Apache는 정적 콘텐츠에 대해 캐싱을 제공하여 성능을 향상시킬 수 있음

    • Spring Boot는 기본적으로 캐싱 기능이 없으므로, 이러한 웹 서버를 추가하면 페이지 로딩 속도를 개선할 수 있음

  • 웹 서버가 캐시된 정적 리소스를 제공하면, 애플리케이션 서버(WAS)의 부하가 줄어들어 더 많은 동적 요청을 처리할 수 있음

웹 서버와 WAS가 함께 사용되는 아키텍처

프로덕션 환경에서는 일반적으로 **Nginx/Apache(웹 서버)**가 리버스 프록시로 동작하고, **Spring Boot(내장 WAS)**가 실제 애플리케이션 로직을 처리하는 구조로 구성

  • 웹 서버(Nginx): 정적 파일 서빙, 로드 밸런싱, SSL 처리, 리버스 프록시 역할
  • WAS(Spring Boot 내장 Tomcat): 동적 요청 처리, 데이터베이스 연동, 비즈니스 로직 처리

이와 같은 분리된 아키텍처를 통해 성능과 확장성을 극대화할 수 있음

CI/CD 파이프라인

개요

  • CI/CD 파이프라인은 소프트웨어 개발 및 배포 과정을 자동화하여, 코드를 신속하고 안정적으로 개발하고 배포할 수 있도록 도와주는 방법론
  • 파이프라인을 통해 소프트웨어의 품질을 유지하면서도 개발 주기를 단축할 수 있으며, 개발자들은 새로운 기능을 보다 빠르게 사용자에게 제공할 수 있음
  • CI/CD 파이프라인은 지속적 통합(Continuous Integration)과 지속적 전달/배포(Continuous Delivery/Deployment) 과정을 포함하여, 개발의 모든 단계를 자동화하고 효율적으로 관리할 수 있도록 설계됨

지속적 통합(Continuous Integration, CI)

  • 개발자들이 작성한 코드를 정기적으로, 보통 하루에 여러 번, 중앙 리포지토리에 병합하고 빌드하고 테스트를 자동화 하는 것을 의미
  • CI 프로세스는 다음과 같은 단계로 이루어짐
    • 코드 병합 : 발자들은 각각의 기능을 개발한 후, 이를 중앙 리포지토리에 병합합니다. 이 과정에서 충돌이 발생할 수 있으며, CI 툴은 이를 자동으로 해결하거나 알림
    • 빌드 자동화 : 병합된 코드가 빌드되며, 빌드 실패 시 이를 즉시 알려줌
    • 자동화된 테스트 : 빌드된 코드에 대해 단위 테스트, 통합 테스트 등이 자동으로 수행됨, 오류가 발생하면 개발자에게 알림이 전달됨

지속적 전달/배포(Continuous Delivery/Deployment, CD)

  • CI가 성공적으로 이루어진 후에는 지속적 전달/배포(CD) 단계가 이어짐
  • 이 단계에서는 코드가 자동으로 배포될 수 있도록 준비되거나 실제로 배포됨
  • CD는 크게 두 가지로 나누어 짐
    1. 지속적 전달(Continuous Delivery)
      • 코드를 스테이징 환경까지 자동으로 배포하고, 운영 환경에 배포하는 과정은 수동으로 수행
      • 이 방법은 최종 사용자에게 배포되기 전에 사람이 마지막으로 검토할 시간을 제공
    2. 지속적 배포(Continuous Deployment)
      • 모든 테스트를 통과한 코드를 자동으로 운영 환경까지 배포
        • 완전한 자동화를 지향하며, 코드는 승인 절차 없이 자동으로 사용자에게 배포됨

CI/CD 의 중요성

소프트웨어 개발과 배포 프로세스를 빠르고 일관되게 수행할 수 있도록 도와줌

  • 신속한 피드백 제공: 개발자들은 코드 변경 후 즉시 피드백을 받을 수 있어, 문제를 조기에 발견하고 해결할 수 있음
  • 품질 향상: 자동화된 테스트와 검증 절차를 통해 코드 품질을 보장하고, 버그를 줄임
  • 배포 주기 단축:**** 코드가 신속하게 배포되므로 사용자에게 새로운 기능을 더 자주 제공할 수 있음
  • 안정성 향상: 자동화된 배포 프로세스를 통해 배포 오류를 줄이고, 일관된 배포를 보장
  • 개발자와 운영자 간의 갈등 해소: 변경 사항을 자주 푸시하고자 하는 개발자와 안정적인 애플리케이션을 원하는 운영자 간의 갈등을 효과적으로 해결할 수 있음

CI/CD 파이프라인

효과적인 CI/CD 파이프라인을 구축하기 위해서는 다양한 구성 요소들이 필요

  1. 버전 관리 시스템 (Version Control System, VCS)
    • 주요 역할 : 소스 코드를 관리하고 변경 이력을 추적
    • 예시 도구 : Git, GitHub, GitLab, Bitbucket
    • 설명
      • 개발자들이 작성한 코드는 버전 관리 시스템에서 관리됨
      • CI/CD 파이프라인은 VCS에서 코드 변경 사항을 감지하고, 이를 기반으로 빌드 및 테스트를 자동으로 실행
  2. Continuous Integration (CI)
    • 주요 역할 : 코드가 병합될 때마다 자동으로 빌드하고 테스트
    • 예시 도구 : Jenkins, GitHub Actions, GitLab CI, CircleCI, Travis CI
    • 설명 : CI 단계에서는 코드가 병합될 때마다 자동으로 빌드가 실행되고, 단위 테스트, 통합 테스트 등의 테스트가 수행됨, 이를 통해 코드의 품질을 보장하고, 문제를 조기에 발견할 수 있음
  3. 테스트 (Testing)
    • 주요 역할 : 소프트웨어의 기능, 성능, 보안 등을 검증
    • 예시 도구 : JUnit, Selenium, pytest, Postman, Cypress, SonarQube
    • 설명
      • CI/CD 파이프라인에서는 여러 단계의 테스트가 포함
      • 단위 테스트(Unit Test), 통합 테스트(Integration Test), 기능 테스트(Functional Test), 보안 테스트(Security Test) 등이 자동으로 수행
  4. Continuous Deployment/Delivery (CD)
    • 주요 역할 : 테스트를 통과한 코드를 자동으로 배포
    • 예시 도구 : Jenkins, GitLab CI/CD, ArgoCD, Spinnaker, AWS CodeDeploy
    • 설명
      • Continuous Delivery: 코드를 스테이징 환경까지 자동으로 배포하며, 운영 환경으로의 배포는 수동으로 수행
      • Continuous Deployment: 모든 테스트를 통과한 코드를 운영 환경까지 자동으로 배포
  5. 아티팩트 관리 (Artifact Management)
    • 주요 역할 : 빌드된 바이너리 파일, 라이브러리, 패키지 등을 관리
    • 예시 도구 : JFrog Artifactory, Nexus, AWS S3
    • 설명 : 빌드 과정에서 생성된 아티팩트는 아티팩트 관리 도구에 저장되며, 이후의 배포 과정에서 사용됨
  6. 컨테이너화 및 오케스트레이션 (Containerization & Orchestration)
    • 주요 역할 : 애플리케이션을 컨테이너로 패키징하고, 이를 관리 및 배포
    • 예시 도구 : Docker, Kubernetes, OpenShift
    • 설명
      • 컨테이너화는 애플리케이션을 운영 환경에 종속되지 않고 일관되게 배포할 수 있도록 함
      • 오케스트레이션 도구는 여러 컨테이너의 배포, 확장, 관리를 자동화합니다.
  7. 모니터링 및 로깅 (Monitoring & Logging)
    • 주요 역할: 배포된 애플리케이션의 성능, 오류 등을 모니터링하고, 로그를 수집
    • 예시 도구: Prometheus, Grafana, ELK Stack (Elasticsearch, Logstash, Kibana), Splunk
    • 설명: 배포된 애플리케이션의 상태를 실시간으로 모니터링하여 문제를 조기에 감지하고 대응하며, 로그를 수집하여 문제 발생 시 원인을 분석할 수 있음
  8. 알림 및 피드백 (Notification & Feedback)
    • 주요 역할: 빌드, 테스트, 배포 등의 결과를 개발자에게 알림
    • 예시 도구: Slack, Microsoft Teams, 이메일, PagerDuty
    • 설명: 파이프라인에서 발생하는 주요 이벤트(예: 빌드 실패, 배포 완료 등)를 개발자에게 실시간으로 알리며, 이를 통해 빠르게 대응할 수 있음
  9. 시크릿 관리 (Secret Management)
    • 주요 역할: API 키, 데이터베이스 비밀번호 등 민감한 정보를 안전하게 관리
    • 예시 도구: HashiCorp Vault, AWS Secrets Manager, Azure Key Vault
    • 설명: 시크릿 관리는 민감한 정보를 안전하게 저장하고, 필요한 환경에서만 접근할 수 있도록 설정하여 보안을 강화

AES 암호화와 RSA 암호화

개요

  • AES와 RSA는 모두 현대 암호화 기술에서 널리 사용되는 암호화 알고리즘
  • 그러나 이 두 알고리즘은 서로 다른 방식으로 동작하며, 각기 다른 용도와 장단점을 가지고 있음

1. 암호화 방식

AES (Advanced Encryption Standard) 암호화 방식

  • 대칭 키 암호화 방식
    • 암호화와 복호화에 동일한 키를 사용
    • 따라서 보안을 위해 이 키는 안전하게 관리되어야 함, 키가 유출되면 전반적으로 위험에 처할 수 있음

RSA (Rivest-Shamir-Adleman) 암호화 방식

  • 비대칭키 암호화 방식
    • RSA는 공개키와 비밀키라는 두 개의 키를 사용
    • 공개키로 암호화된 데이터는 비밀키로만 복호화할 수 있음
    • 반대로 비밀키로 서명된 데이터는 공개 키로만 검증할 수 있음
  • 공개키와 개인키
    • 공개키
      • 일반적으로 서버가 공개키를 소유하고, 이를 클라이언트에게 배포
      • 공개키는 누구에게나 공개할 수 있으며, 이를 통해 클라이언트가 서버로 데이터를 안전하게 보낼 수 있음
      • 역할
        • 암호화 ****
          • 클라이언트가 서버에 데이터를 보낼 때, 서버의 공개키를 사용해 데이터를 암호화
          • 암호화된 데이터는 서버의 개인키가 없이는 복호화할 수 없음
        • 인증
          • 클라이언트는 서버가 제공하는 공개키를 사용해 서버의 신원을 확인할 수 있음
          • 공개키가 인증서와 함께 제공되며, 이 인증서를 통해 서버의 신뢰성을 검증
    • 개인키
      • 개인키는 서버가 소유하며, 외부에 절대 공개되지 않아야 함. 개인키는 암호화된 데이터를 복호화하고, 서버의 신원을 증명하는 데 사용
      • 역할
        • 복호화
          • 클라이언트가 서버의 공개키로 암호화한 데이터를 서버는 자신의 개인키로 복호화할 수 있음
        • 디지털 서명
          • 서버는 개인키로 데이터를 서명하여, 클라이언트에게 자신이 신뢰할 수 있는 서버임을 증명할 수 있음
          • 클라이언트는 서버의 공개키로 이 서명을 검증할 수 있음

2. 암호화 알고리즘의 보안 속성

기밀성 (Confidentiality)

  • AES
    • AES는 주로 기밀성을 보장하는 데 사용됨
    • 대칭 키 암호화 알고리즘으로, 데이터를 암호화하여 무단 접근으로부터 보호
    • AES의 기밀성은 매우 강력
  • RSA
    • RSA도 기밀성을 제공할 수 있지만, 주로 작은 데이터나 키를 암호화하는 데 사용됨
    • 비대칭 키 암호화 알고리즘으로, 공개 키로 암호화된 데이터는 비밀 키로만 복호화할 수 있음
    • 하지만 RSA는 AES보다 속도가 느리기 때문에, 대량의 데이터를 암호화하는 데는 적합하지 않음
    • 세션 키와 같은 중요한 정보를 안전하게 교환하는 데 주로 사용
  • cf) 기밀성이란
    • 민감한 정보가 무단으로 노출되지 않도록 보호하는 것
    • 데이터를 볼 권한이 없는 사람이 데이터에 접근하지 못하게 하고, 데이터가 안전하게 유지되도록함
    • 예시
      • 암호화 : 데이터를 암호화하여 오직 복호화 키를 가진 사람만이 데이터를 읽을 수 있도록함
      • 접근제어 : 스템에서 사용자마다 접근 권한을 부여하여, 허가된 사용자만 특정 데이터에 접근할 수 있도록 하는 것
      • 익명성 : 용자의 신원을 숨겨서 불필요한 정보 노출을 방지하는 것

무결성 (Integrity)

  • AES
    • 기본적으로 무결성을 제공하지 않음
    • 기밀성을 제공하지만, 메시지가 도중에 변조되었는지 확인하는 기능이 없음
      • 네트워크를 통해 AES로 암호화된 파일을 전송할 때, 누군가가 암호화된 데이터를 조작하여 수신자가 이를 복호화하면, 잘못된 정보가 나오더라도 수신자는 이 사실을 알 방법이 없음
    • 따라서 무결성을 보장하기 위해서는 별도의 MAC(Message Authentication Code)나 HMAC(Hash-based Message Authentication Code)와 같은 기술과 결합하여 사용해야함
  • RSA
    • RSA는 디지털 서명에 사용될 때 무결성을 보장할 수 있음
    • RSA 비밀 키로 서명된 데이터는 공개 키로 검증할 수 있음
    • 데이터가 서명 이후 변경되지 않았음을 확인할 수 있음
  • cf) 무결성이란
    • 무결성은 데이터를 전송하거나 저장할 때, 그 데이터가 전송 과정에서 변조되거나 손상되지 않았음을 보장하는 속성
    • 예를 들어, 누군가가 암호화된 메시지의 일부를 수정하거나 바꿨을 때, 수신자는 이 변경 사실을 알아야만 함
    • 이를 위해 암호화된 데이터의 변조 여부를 확인하는 무결성 체크가 필요
    • 무결성이 중요한 이유는, 데이터가 암호화되었다고 해서 항상 안전한 것은 아니기 때문
      • 데이터가 암호화되었더라도, 중간에 누군가가 암호문을 조작하면 복호화 후에는 의미 없는 데이터가 될 수 있고, 더 나아가 공격자는 이 조작을 통해 의도된 결과를 만들 수도 있음
      • 은행 송금 시스템에서 암호화된 데이터를 조작하여 금액이나 계좌 정보를 변경하는 경우가 있을 수 있음

신뢰성 (Authenticity)

  • AES
    • AES 자체로는 신뢰성을 보장하지 않음
    • 데이터를 보낸 사람의 신원을 확인하는 데 사용할 수 없음
    • AES는 인증된 키 교환 프로토콜과 함께 사용될 때, 신뢰성을 보장할 수 있는 기반을 제공
  • RSA
    • 신뢰성을 보장하는 데 매우 중요한 역할을 함
    • RSA 공개 키 암호화는 디지털 서명과 결합되어 사용자의 신원을 확인할 수 있게해 줌
    • 사용자가 RSA 비밀 키로 데이터를 서명하면, 그 사용자의 신원을 확인할 수 있으며, 이는 신뢰성을 보장
    • 따라서 RSA는 인증서 기반의 시스템에서 중요한 역할
  • cf) 신뢰성 이란
    • 데이터를 전송하거나 제공한 주체가 실제로 신뢰할 수 있는 자원임을 보장하는 속성
    • 데이터나 메시지가 실제 발신자로부터 온 것임을 증명하는 것을 의미하며, 데이터를 수신하는 쪽에서 데이터의 출처를 검증할 수 있게 함
    • 인증(authentication)과 매우 밀접한 개념
    • 예시
      • 디지털 서명 : 발신자가 자신의 비밀 키로 메시지에 서명하여, 수신자는 서명된 메시지가 발신자에게서 온 것임을 확인할 수 있음
      • 인증서 : SSL/TLS에서 사용되는 디지털 인증서는 웹사이트의 신원을 검증하여 사용자가 신뢰할 수 있는 서버와 통신하고 있음을 보장
      • 2단계 인증 : 사용자가 시스템에 로그인할 때, 비밀번호 외에 추가적인 인증 요소를 요구하여 사용자의 신원을 더 확실하게 확인

확장성

  • AES
    • 키 길이에 따라 확장할 수 있음
    • AES-128, AES-192, AES-256과 같이 다양한 키 길이에서 사용될 수 있음
  • RSA
    • 키 길이에 크게 의존
    • 2048비트, 4096비트 등으로 확장할 수 있음
    • 보안 요구사항에 따라 키 크기를 선택해야 함

3. 사용 사례

AES

  • 데이터 전송, 파일 암호화, VPN, 디스크 암호화 등 대량 데이터를 암호화해야 할 때 많이 사용
  • HTTPS 를 통해 전송되는 데이터를 암호화 하는데 AES가 자주 사용됨
  • 파일 및 디스크 암호화 및 암호화된 데이터 베이스, 클라우드 스토리지 보안, 스마트폰(Android, ios) 데이터 보안 등에 사용됨
  • 메세징 앱(WhatsApp, Telegram)과 같은 안전한 메시징 앱은 종단간 암호에 AES를 사용
  • VPN, 무선 통신(Wi-Fi) WPA2 보안 프로토콜 등에 사용

RSA

  • 작은 데이터 조각의 안전한 전송, 키 교환, 인증서 서명 등에 주로 사용됨

  • SSL/TLS 인증서에서 클라이언트와 서버 간의 키 교환 시 RSA가 사용됨

    ※ 한편 SSL/TLS 에서도 데이터를 암호화할때는 AES를 사용

5. 성능

AES

  • 하드웨어 가속이 가능하며, 특히 대량의 데이터를 빠르게 암호화하는 데 최적화되어 있음
  • 대칭 암호화 방식이기 때문에 상대적으로 연산이 간단하고 빠름
  • Time Complexity : O(n) (n은 데이터의 크기(블록 수))

RSA

  • 연산이 매우 복잡하며, 대량 데이터를 처리하는 데 적합하지 않음
  • 키의 길이가 길어질수록 연산 복잡도가 기하급수적으로 증가하여 더 느려짐
    • AES는 단순한 대칭 키 연산을 사용하여 데이터 블록을 암호화하는 반면, RSA는 큰 소수(prime number)와 모듈러 산술(modular arithmetic)을 사용하여 복잡한 수학적 연산을 수행하기 때문
  • 주로 소량의 데이터나 세션 키를 암호화하는 데 사용됨
  • Time Complexity : O(k^3) (k는 RSA의 크기(비트 수))

부록1. AES 암호화의 초기화 벡터 (Initialization Vector, IV)

목적

  • 임의성 추가
    • AES 암호화에서 특히 CBC 모드와 같은 블록 암호화 모드에서 사용됨
    • 초기화 벡터는 동일한 평문과 키를 사용하더라도 매번 다른 암호문을 생성할 수 있도록 임의성을 추가함
  • 보안 강화
    • 초기화 벡터는 암호화된 데이터의 보안성을 높이는 데 중요한 역할을 하며, 보통 각 암호화 작업마다 새로운 IV를 생성하여 사용
  • IV는 암호화된 데이터와 함께 저장되거나 전송되며, 복호화 시에도 동일한 IV를 사용하여 원본 데이터를 복구함

사용방법

  1. IV를 암호문에 포함하는 방법

  • 설명

    • 이 방법에서는 암호화할 때 생성된 IV를 암호문에 포함하여 저장하거나 전송
    • 일반적으로, IV는 암호문 앞에 추가되며, 암호화된 데이터와 함께 하나의 문자열로 결합

  • 장점

    • 단순화: IV를 따로 관리할 필요가 없으므로, 복잡한 데이터베이스 설계나 추가적인 IV 관리 로직이 필요하지 않음
    • 일관성: 모든 필요한 정보(암호문과 IV)가 하나의 데이터 블록으로 결합되므로, 복호화할 때 필요한 모든 정보가 함께 제공됨
    • 재사용 가능성: 데이터가 이동하거나 다른 시스템으로 전송될 때, 암호문과 IV가 함께 이동하므로 복호화 과정에서 누락되는 것을 방지할 수 있음

  • 단점

    • 약간의 오버헤드: 암호문에 IV를 추가하면 데이터의 길이가 늘어날 수 있음. 그러나 일반적으로 IV의 크기(예: 16바이트)는 상대적으로 작으므로 큰 문제가 되지 않음

  • 사용 사례

    • 웹 애플리케이션: HTTPS와 같은 프로토콜에서 암호화된 메시지와 함께 IV를 전송하는 경우
    • 파일 암호화: 암호화된 파일 자체에 IV를 포함시켜 저장하는 경우
    • 일반적인 API 암호화: API 응답이나 요청에서 암호화된 데이터와 IV를 함께 반환하는 경우

  • 코드

    • 암호화할 때 : IV를 생성하고, 암호화된 데이터 앞에 IV를 추가하여 저장함

    • 복호화할 때 : 암호문에서 IV를 추출하고, 이를 사용하여 복호화

  1. IV를 별도로 저장하는 방법

  • 설명

    • 이 방법에서는 IV를 암호문과 별도로 저장하거나 전송
    • 데이터베이스에 별도의 컬럼으로 IV를 저장하거나, 암호화된 데이터와는 독립적으로 IV를 관리

  • 장점

    • 유연성**:** IV를 별도로 관리하면, 암호문과 IV를 분리하여 저장하거나 전송할 수 있는 유연성을 제공
    • 효율성**:** 암호문 자체에 추가적인 데이터(IV)를 포함시키지 않으므로, 암호문 자체의 크기를 줄일 수 있음

  • 단점

    • 관리의 *복잡성*: IV를 별도로 관리해야 하므로, 추가적인 데이터베이스 설계와 관리 로직이 필요
    • 동기화 문제**:** 암호문과 IV가 분리되어 저장될 경우, 복호화 시 정확히 일치하는 IV를 찾는 데 오류가 발생할 가능성이 있음

  • 사용 사례

    • 고급 보안 애플리케이션**:** 매우 정밀한 데이터 관리가 필요한 경우, 암호문과 IV를 분리하여 관리
    • 분산 시스템**:** 여러 서버 또는 시스템에서 데이터를 처리할 때, IV를 별도로 관리하여 보안을 강화할 수 있음

  • 코드

    • 암호화할 때 : 새로운 IV를 생성하고, 암호화된 데이터와 함께 DB에 저장

    • 복호화할 때 : 해당 데이터를 복호화할 때, DB에서 저장된 IV와 암호화된 데이터를 함께 불러와 복호화

    • DB 테이블 구조

실무에서는 IV를 암호문에 포함하는 방법이 더 많이 아용되는 것으로 보임 → 암호화된 데이터와 IV를 함께 관리할 수 있어 편리하고, 복호화 과정에서 필요한 정보를 한 곳에서 쉽게 접근할 수 있기 때문으로 생각됨

IV를 별도로 저장하는 방법은 특정한 요구 사항이 있을 때나 매우 정밀한 보안 관리가 필요한 경우에 사용되지만, 관리의 복잡성 때문에 일반적으로는 덜 사용되는 것으로 생각됨

부록2. 해싱(Hashing)과 암호화(Encryption)의 차이점

Hashing 의 특징

  • 일방향성 : 해시 함수는 입력 값을 고정된 크기의 해시 코드로 변환하며, 이를 역으로 원래 데이터를 복구할 수 없음
  • 고정 길이 출력 : 입력 데이터의 길이와 상관없이 항상 고정된 길이의 해시 값을 출력
  • 충돌 회피 : 서로 다른 입력이 동일한 해시 값을 가지지 않도록 설계되어 있음
  • 빠른 연산 : 해시 함수는 빠른 속도를 자랑하며 대량의 데이터 처리에 적합, Time Complexity O(1)

암호화와 해싱의 차이점

특성 해싱 암호화
목적 데이터 무결성 검증, 빠른 검색, 비밀번호 저장 데이터 보호 및 기밀성 유지
복호화 가능 여부 불가능 가능
일방향성 있음 없음
출력 길이 고정 길이 가변 길이

해싱의 사용 사례

  • 데이터 무결성 검증
    • 파일 무결성 체크
      • 파일의 해시 값을 미리 계산하여 저장해두고, 이후에 파일이 변경되었는지 확인하기 위해 다시 해시 값을 계산하여 비교할 수 있음
    • 메시지 무결성 검증
      • 네트워크 통신에서 메시지 다이제스트(MD5, SHA-256 등)를 생성하여 메시지와 함께 전송하고, 수신 측에서 동일한 해시 알고리즘으로 메시지의 해시 값을 계산해 비교
  • 비밀번호 저장
    • 비밀번호 해싱
      • 사용자 비밀번호를 데이터베이스에 저장할 때, 해시 함수를 사용하여 비밀번호를 해시 처리한 후 저장
      • DB가 해킹당하더라도 실제 비밀번호를 알 수 없게됨
      • 사용자가 로그인할 때 입력한 비밀번호를 같은 해시 함수로 변환해 저장된 해시 값과 비교
      • 일반적으로 해시 값에 추가적인 안전성을 위해 솔트(Salt)를 추가하여 해시를 생성
  • 해시 테이블
    • 빠른 데이터 검색 : 키-값 쌍의 데이터를 저장하고 검색하는 데 사용
    • 캐싱 시스템 : 해시 테이블은 메모리 캐싱 시스템에서도 사용됨
  • 디지털 서명 및 인증서
    • 디지털 서명
      • 서명할 데이터의 해시 값을 계산하고, 그 해시 값을 개인 키로 암호화하여 서명
      • 수신 측은 공개 키로 서명을 복호화하여 해시 값을 얻고, 동일한 해시 함수를 사용해 데이터의 해시 값을 계산하여 비교
    • 디지털 인증서 : SSL/TLS 인증서와 같은 디지털 인증서에서도 해시 함수가 사용됨, 인증서의 내용을 해시한 값을 사용하여 인증서의 무결성을 검증

부록 3. AES 를 사용할 때 Base64 디코딩을 사용하는 이유

디코딩(Decoding) 이란

  • 복호화 가능: 디코딩은 일반적으로 인코딩된 데이터를 원래의 형식으로 복원하는 과정입니다. 인코딩이 가역적(되돌릴 수 있는)이라는 점에서 해싱과 다릅니다.
  • 데이터 복구: 디코딩의 목적은 인코딩된 데이터를 원래의 형식으로 되돌리는 것입니다. 예를 들어, Base64로 인코딩된 문자열을 디코딩하면 원래의 바이너리 데이터나 텍스트로 복구할 수 있습니다.
  • 인코딩/디코딩: 인코딩은 데이터를 특정 형식으로 변환하는 것이며, 디코딩은 그 데이터를 다시 원래의 형식으로 되돌리는 것입니다. 이는 데이터를 전송하거나 저장할 때 특정 형식이 요구될 때 사용됩니다.
  • 예시: Base64, URL 인코딩, UTF-8 등 다양한 인코딩 방식이 있으며, 이들의 디코딩 과정도 가능합니다.

사용 이유

  • 이진 데이터를 텍스트로 변환
    • AES 암호화의 결과는 텍스트가 아니라 이진 데이터로 사람이 읽을 수 없으며, 이메일 전송, URL, JSON, XML, 로그 파일 등 텍스트 기반의 시스템에서 제대로 처리되지 않을 수 있음
    • Base64 인코딩은 이 이진 데이터를 안전한 ASCII 텍스트로 변환하여 이러한 시스템에서 안전하게 다룸
  • 전송 과정에서 데이터 손상 방지
    • 이진 데이터는 전송 과정에서 텍스트 기반의 프로토콜(예: 이메일, HTTP, 웹 소켓 등)을 통해 전송될 때 손상될 수 있는데, Base64로 인코딩된 데이터는 이러한 문제를 피할 수 있음
    • 예를 들어, 이메일 전송 시 SMTP 프로토콜은 7비트 ASCII 텍스트만 안전하게 전송할 수 있기 때문에, 이진 데이터를 Base64로 인코딩하여 텍스트로 전환해 전송
  • 플랫폼 독립성
    • Base64 인코딩된 텍스트는 어떤 시스템에서든 동일하게 인식되고 처리될 수 있음
  • 데이터 저장 및 출력의 용이성
    • AES로 암호화된 이진 데이터를 로그 파일이나 데이터베이스에 저장할 때, 이진 데이터를 직접 저장하면 데이터 손상 또는 오류가 발생할 수 있는데, Base64로 인코딩하면 이러한 데이터를 안전하게 저장하고 나중에 디코딩하여 원래의 이진 데이터로 복원할 수 있음

결론

AES와 RSA는 각각의 장점과 단점을 가지고 있으며, 서로 상호 보완적인 암호화 기술이다. AES는 대칭 키 암호화 방식으로, 대량의 데이터를 빠르게 암호화하는 데 적합하며, 기밀성과 성능 측면에서 우수한 선택이다. 반면에, RSA는 비대칭 키 암호화 방식으로, 작은 데이터나 세션 키의 안전한 전송, 그리고 디지털 서명과 같은 신뢰성 보장이 필요한 상황에서 중요한 역할을 한다.

두 기술은 각각의 강점을 발휘하여, 현대 암호화 시스템에서 함께 사용된다. 예를 들어, SSL/TLS 프로토콜에서는 RSA를 사용하여 초기 세션 키를 안전하게 교환하고, 이후의 데이터 전송은 AES를 사용하여 빠르게 암호화하는 방식이 채택된다. 이와 같은 방식으로, AES와 RSA는 보안과 성능을 모두 만족시키는 강력한 암호화 체계를 구축하는 데 필수적인 요소이다.

따라서, AES와 RSA는 단독으로 사용되기보다는 서로 보완하며 사용될 때, 더욱 안전하고 효율적인 암호화 솔루션을 제공한다. 이 둘의 조화는 현대의 보안 시스템이 직면한 다양한 요구 사항을 충족시키는 데 중요한 역할을 한다.

CORS

개요

CORS (Cross-Origin Resource Sharing)는 웹 애플리케이션에서 보안 문제를 해결하기 위해 만들어진 정책으로, 서로 다른 출처 간에 자원 요청을 허용하는 방법을 정의합니다. 이 글에서는 CORS의 개념, 동작 방식, 그리고 Spring Boot에서 CORS 설정 방법을 자세히 알아보자

CORS(Cross Origin Resource Sharing) 란

  • CORS는 웹 애플리케이션이 서로 다른 출처의 자원에 접근할 수 있도록 브라우저에 허용하는 체제
  • 예를 들어, http://example.com에서 호스팅되는 웹 애플리케이션이 http://api.example.com의 데이터를 요청하고자 할 때 CORS 정책이 필요
  • 브라우저는 보안상의 이유로 동일 출처 정책(Same-Origin Policy)을 기본적으로 따르며, 이를 통해 다른 출처에서의 리소스 접근을 제한

출처(Origin)란?

  • 출처는 URL의 프로토콜, 호스트, 포트 번호의 조합을 말합니다. 이 세 가지가 모두 동일한 경우 동일 출처라고 하며, 하나라도 다르면 다른 출처로 간주

  • Protocol + Host + Port 3가지가 같으면 동일 출처(Origin)라고 함

동일 출처 예시

예시 설명
http://example.com:80
http://example.com		 같은 포트 번호 사용
http://example.com/app1/index.html
http://example.com/app2/index.html		 Protocol, Host, Port이 같으며, Path부터 다름

다른 출처 예시

예시 설명
http://example.com/app1
https://example.com/app2		 프로토콜이 다름
http://example.com
http://www.example.com
http://myapp.example.com		 호스트가 다름
http://example.com
http://example.com:8080		 포트가 다름

CORS의 필요성

  • 웹 페이지에서 <img>, <script>, <iframe> 등을 사용하여 외부 자원을 불러오는 것은 매우 일반적

  • 그러나, 이러한 요청이 악의적인 목적을 가진 경우 보안 위험이 발생할 수 있음

  • 예를 들어, 사용자가 악성 스크립트를 포함한 페이지를 방문하면 그 페이지가 사용자의 인증 정보를 사용해 다른 출처에 유해한 요청을 보낼 수 있음

  • CORS는 이러한 보안 문제를 해결하기 위해 브라우저가 다른 출처로부터의 요청을 적절하게 제한하고, 허용된 경우에만 요청을 수행할 수 있도록 도와줌

동일 출처 정책(Same-origin policy)

  • 동일 출처 정책(Same-origin policy)는 다른 출처로부터 조회된 자원들의 읽기 접근을 막아 다른 출처 공격을 예방합

  • 그러나, 다른 출처에서 얻은 이미지를 담는 , 외부 주소를 담는 같은 여러 태그들을 허용

  • 동일 출처 정책의 정확한 구현 명세는 없지만 최신의 브라우저들은 일정 규칙을 따름 ( RFC6454 )

동일 출처 요청 vs 다른 출처 요청 그림

  • 요청하는 클라이언트와 요청받는 서버가 같은 출처에 있으면 동일 출처
  • 서로 다른 서버에 있으면 다른 출처 요청
    • domain-a.com 유저가 domain-a.com 서버에 요청하면 동일 정책이기 때문에 아무런 문제가 없음
    • domain-a.com 유저가 domain-b.com 서버에 요청하면 호스트가 다르기 때문에 다른 출처 요청을 함
    • 도메인 이외에, 같은 프로젝트 내에 정의된 css 파일 요청은 동일 출처 요청이고, font 같이 다른 외부 사이트에서 실시간으로 import를 통해 가져온다면 다른 출처 요청 이처럼, 같은 출처가 아닌 외부에 자원을 요청하는 경우가 있는지 잘 확인해보아야 함

기본적으로 동일 출처 요청만 자유롭게 요청이 가능하며 동일 출처 정책(Same-Origin Policy) 이라고 함

하지만 기준을 완화하여 다른 출처 요청도 할 수 있도록 기준을 만든 체제가 다른 출처 정책(Cross-Origin Policy) 임

CORS의 요청 유형

1. 단순 요청 (Simple Request)

  • 사용 시기

    • GET, POST, HEAD와 같은 안전한 HTTP 메서드를 사용하는 경우.

    • 요청 헤더가 CORS 안전 리스트에 있는 헤더들(예: Accept,Accept-Language, Content-Type 등이 일반적인 값으로 설정된 경우)

      • Contet-Type 헤더는 application/x-www-form-urlencoded, multipart/form-data and text/plain 만 가능

    • 보안이 크게 중요하지 않거나, 간단한 데이터 요청을 수행할 때 적합합니다.

  • 장점 : 브라우저가 추가적인 프리플라이트 요청을 보내지 않으므로 요청이 간소화되고 성능이 조금 더 좋을 수 있음

  • 단점 : 특정 메서드와 헤더에만 제한되므로 유연성이 떨어집니다.

  • 특징 : 브라우저는 요청의 출처를 서버에 전달하며, 서버는 Access-Control-Allow-Origin 헤더를 통해 요청을 허용할지를 결정

  • 사용 예시

    • 간단한 GET 요청으로 API 데이터를 받아오는 경우

      • 성공 케이스

      • 실패 케이스

    • 논리 도해

2. 프리 플라이트(Preflight Request)

  • 사용 시기

    • PUT, DELETE와 같은 안전하지 않은 HTTP 메서드를 사용하거나, 사용자 정의 헤더가 있는 요청을 보낼 때
    • 요청이 중요한 데이터를 수정하거나 민감한 작업을 수행하는 경우
    • 보안이 중요한 경우나, 서버가 클라이언트의 요청을 철저히 검토하고 허용할 필요가 있을 때 적합

  • 장점

    • 서버가 OPTIONS 메소드를 통해 요청을 사전에 검사하고 허용할지 여부를 결정할 수 있으므로 보안성이 높음
    • 서버에서 허용된 메서드와 헤더를 명확하게 정의하여 허용되지 않은 요청을 사전에 차단할 수 있음

  • 단점

    • 브라우저가 두 번의 요청(프리플라이트 요청과 실제 요청)을 보내기 때문에 성능에 약간의 영향을 미칠 수 있음

  • 사용 예시

    • 클라이언트에서 데이터를 삭제하거나 업데이트하는 요청(예: REST API에서 DELETE 요청)에서 사용

    • 프리플라이트 요청은 브라우저가 실제 요청을 보내기 전에 서버에게 미리 “이런 요청을 보내도 되겠습니까?”라고 물어보는 사전 요청

    • 요청은 주로 안전하지 않은 HTTP 메서드(예: PUT, DELETE)나, 사용자 정의 헤더가 포함된 요청을 보낼 때 사용됨

    💡 **참고사항**
    • 요청 헤더에는 다음 값이 포함
      • origin : 어디서 요청을 했는지 서버에 알려주는 주소
      • access-control-request-method : 실제 요청이 보낼 HTTP 메서드
      • access-control-request-headers : 실제 요청에 포함된 header
    • 응답 헤더에는 다음 값이 포함
      • access-control-allow-origin : 서버가 허용하는 출처
      • access-control-allow-methods : 서버가 허용하는 HTTP 메서드 리스트
      • access-control-allow-headers : 서버가 허용하는 header 리스트
      • access-control-max-age : 프리 플라이트 요청의 응답을 캐시에 저장하는 시간
      • 플리플라이트 요청은 OPTIONS 를 사용해 자신의 주소 https://www.api.com?q=test 를 보내며, 또한 origin, access-control-request-method, access-control-request-headers를 같이 보냄
      • 정상적인 응답으로 access-control-allow-origin, access-control-allow-method, access-control-allow-headers, access-control-max-age를 응답받음
      • 정상 요청과 응답이 가능하다는 프리 플라이트 덕분에 실제 요청을 함 그리고 정상적인 응답을 받음

3. 신용 요청(Credentialed Request)

  • 사용 시기

    • 인증 정보(쿠키, 세션 정보, 인증 헤더, TLS 클라이언트 인증서 등)를 포함하여 요청을 보낼 때
    • 사용자의 인증 상태를 유지하면서 서버와 통신해야 하는 경우
    • 인증이 중요한 경우, 특히 사용자의 민감한 데이터를 다루는 서비스에서 사용됨

  • 특징

    • CORS 정책상 기본적으로 신용 요청은 차단되지만, 서버가 Access-Control-Allow-Credentials 헤더를 true로 설정한 경우 허용됨

  • 장점

    • 인증 정보를 포함하여 보안된 요청을 보낼 수 있음

  • 단점

    • 서버가 Access-Control-Allow-Credentials: true를 설정해야 하고, Access-Control-Allow-Origin에서 *을 사용할 수 없으므로 설정이 까다로울 수 있음

  • 사용 예시

    • 인증 정보를 포함하여 보내는 요청
    • 사용자가 로그인 상태에서 서버의 개인 데이터를 조회하거나 수정하는 요청

  • 만약 서버 응답에 access-control-allow-credentials 가 true로 설정되지 않았거나 access-control-allow-origin 헤더에 있는 값이 허용된 출처가 아니라면 아래와 같이 오류가 발생

성능 테스트란?

  • 앱이나 시스템이 주어진 조건에서 얼마나 효율적이고 안정적으로 동작하는지 평가하기 위한 테스트 과정

  • 시스템의 성능을 다양한 관점에서 측정하여, 실제 운영 환경에서 발생할 수 있는 문제를 미리 파악하고 해결하는 데 목적이 있음

성능 테스트의 필요성

2017년 구글이 발한 자료 내용을 살펴보자 [참고]

  • 모바일 환경에서 페이지 로드 타임이 1초에서 3초 가까이 되면 서비스 사용자 이탈률이 32%까지 증가
  • 5초 이상 이탈률이 90% 증가
  • 6초이상 이탈률이 106% 증가
  • 10초 이상 이탈률이 120% 증가

성능 테스트의 목적

  • 설정한 목표치 달성

    • 목표 : 초당 1000건의 요청 처리, 모든 조회 요청을 1초 이내로 응답

      → 배포 전 성능 테스트를 통해 목표치를 만족하는지 확인

    • 어떤 지표를 확인해야할까요?

      • 초당 1000건의 요청 처리 → Throughput (처리량)
      • 모든 조회 요청을 1초 이내로 응답 → Response Time(응답 시간)

  • 성능 병목 현상 파악

    • 응답 시간, 처리량 등 성능 지표를 통해 시스템의 병목 현상을 발견하고 해결

  • 가용성

    • 시스템이 서비스를 정상적으로 제공할 수 있는 상태

  • 기타

    • 확장성 확인
      • 시스템이 사용자 수나 데이터의 증가에 따라 잘 확장될 수 있는지를 확인
    • 안정성 검증
      • 시스템이 장시간 동안 안정적으로 운영될 수 있는지를 확인
    • 복구 능력 평가
      • 시스템이 장애 상황에서 얼마나 빠르고 신속하게 복구될 수 있는지를 평가
    • 최적화
      • 시스템의 자원 사용을 최적화하여 비용을 절감하고 성능을 향상

성능 지표 (Metrics)

성능 테스트의 결과를 정량적으로 평가하는 기준

  • 응답 시간(Response Time) : 사용자가 요청을 보낸 시점부터 응답을 받기까지의 시간

    • 평균 응답 시간 : 모든 요청에 대해 평균적으로 걸리는 시간을 계산
    • 최대/최소 응답 시간 : 특정 시점이나 특정 조건에서 발생하는 최고 및 최저 응답 시간을 파악
    • 백분위 응답 시간 : 예를 들어 95퍼센트 응답 시간은 전체 요청 중 95%가 그 시간 이내에 응답되었음을 의미

    • 응답 시간 = 150 + 200 + 100 = 450 ms

  • 처리량(Throughput) : 시스템이 단위 시간당 처리할 수 있는 요청이나 트랜잭션의 수

    • 초당 요청 수 (Requests per Second, RPS) : 시스템이 초당 처리할 수 있는 HTTP 요청의 수를 측정하여, 웹 앱의 처리 능력을 평가

      • 현재 시스템에서 처리량은 얼마일까요?
        • 100 rps → 서브 시스템 중 가장 처리량이 낮은 부분으로 계산할 수 있음
      • 병목 구간 : 하위 시스템 중 가장 낮은 처리량을 가지는 부분
        • 병목 구간이 아닌 곳을 성능 개선한다면 → 전체적인 성능 개선 실패 500rps를 700rps라고 해도 mysql에서 100 rps라면 여전히 100 rps

    • 트랜잭션 수 (Transactions per Second, TPS) : DB나 앱에서 초당 처리되는 트랜잭션 수를 평가하여, 시스템의 처리 능력을 파악

    • 데이터 전송량 : 네트워크를 통해 초당 전송되는 데이터 양(예: MB/s)을 측정하여, 대용량 데이터 전송 시의 성능을 평가

  • 자원 사용량(Resource Utilization) : CPU, 메모리, 네트워크 대역폭 등의 자원 사용률

  • 오류율(Error Rate) : 처리된 요청 중 실패한 요청의 비율

성능 테스트 유형 (Testing Types)

성능 테스트의 구체적인 방법을 나타내며, 각각의 유형은 특정한 목적을 달성하기 위해 사용됨

  • 부하 관련 테스트 (Load Testing)

    • 부하 테스트**(Load Testing)** : 예상되는 최대 사용자 수나 트래픽 조건에서 시스템 성능을 평가, 목표값에 해당되는 부하를 견딜 수 있는지 확인

    • 고정 부하 테스트 : 예상되는 최대 부하를 일정 시간 동안 유지하여 안정성 평가

    • 스파이크 테스트(Spike Testing) : 짧은 시간 동안 급격하게 부하를 증가시켜 시스템이 어떻게 대응하는지 평가, 사용량이 급증하는 상황에서 시스템이 견디고 성능에 문제가 없는지 확인

    • 피크 테스트(Peak Testing) : 예상되는 최고 부하 상태에서 시스템 성능을 평가

  • 한계점 및 회복력 테스트

    • 스트레스 테스트(Stress Testing) : 시스템이 최대치에 해당되는 부하를 받았을 때 성능과 회복 능력을 평가

      • 부하 테스트 대비 50% 혹은 그 이상의 부하를 주어 테스트
      • 한계점 파악: 시스템 성능 저하, 오류, 다운타임 발생 시점을 확인
      • 자원 소진 테스트: 자원이 소진될 때의 시스템 반응 평가
      • 복구 능력 테스트: 한계 초과 후 시스템이 얼마나 빠르게 정상 상태로 복구되는지 평가

    • 중단점 테스트(Breakpoint Testing) : 시스템이 감당할 수 있는 최대 부하 수준을 찾기 위해 점진적으로 부하를 증가시키며 테스트

    • 회복 테스트 : 시스템 장애 이후 정상 상태로 회복하는 능력을 평가

  • 지속성 및 안정성 평가

    • 내구 테스트 (Endurance/Soak Testing) : 장시간 동안 평균 사용률로 일정한 부하를 가하여 시스템 성능과 안정성을 평가

      • 지속적인 부하 테스트 : 장시간 동안 부하를 가해 성능 변화를 모니터링
      • 리소스 누수 확인 : 메모리 누수, 핸들 누수 등 장시간 테스트에서 발생하는 문제를 확인

    • 안정성 테스트 : 시스템이 오랜 시간 동안 지속적인 부하에서 얼마나 안정적으로 동작하는지 평가

  • 데이터 처리 관련 테스트

    • 볼륨 테스트

      : 대량의 데이터를 처리할 때 시스템 성능이 어떻게 변화하는지 평가

      • 대량 데이터 입력 : 시스템에 대량 데이터를 입력하여 성능 변화를 평가
      • 데이터베이스 성능 평가 : 대용량 데이터 처리 시 데이터베이스의 성능을 집중적으로 평가

  • 기본 기능 확인 테스트

    • 스모크 테스트(Smoke Testing) : 시스템의 주요 기능이 정상적으로 동작하는지 간단히 확인하는 초기 테스트

  • 내구 테스트 (Endurance/Soak Testing) : 장시간 동안 시스템의 안정성을 평가

  • 볼륨 테스트 (Volume Testing) : 대량의 데이터를 처리할 때 시스템 성능을 평가

성능 테스트 계획 수립

1. 응답의 최소 Response Time 정하기

  • 목표 응답 시간을 설정하여 사용자 경험을 최적화
  • 예를 들어, 금융 거래 시스템의 경우 0.3초 이하로 설정

2. 서비스가 Peak일 때의 사용자 수 구하기

  • 과거 데이터와 성장률을 반영하여 피크 타임 사용자 수를 추정하고, 예상보다 20~50% 더 여유 있게 설정

3. 시스템 자원 한계 파악

  • CPU, 메모리, 네트워크 대역폭 등의 자원 사용량을 모니터링하여 시스템이 처리할 수 있는 최대 부하를 파악
  • 자원 한계점을 넘지 않도록 계획을 세움

4. 테스트 시나리오 다양성 확보

  • 다양한 시나리오를 통해 실제 환경에서 발생할 수 있는 모든 상황을 테스트

부록 1. Response Time, Latency, Processing Time

Response Time (응답 시간)

  • 사용자가 어떤 요청을 보내고, 시스템이 해당 요청에 대한 응답을 전송하는 데 걸리는 시간
  • 일반적으로 밀리초 단위로 측정
  • 시스템 또는 네트워크의 성능에 대한 전반적인 인식을 제공

Latency (지연 시간)

  • 데이터가 네트워크를 통해 이동하는 데 걸리는 시간

Processing Time(처리 시간)

  • 시스템이 요청을 받은 후, 해당 요청을 처리하는 데 소요되는 시간

  • 일반적으로 응답시간보다 짧은 시간으로 측정

  • 시스템이 얼마나 효율적으로 요청을 처리하고 있는지를 나타내는 중요한 지표

부록 2. Little’s Law : TPS를 통한 동시 접속자 수 예상

Little’s Law

  • 그래서 TPS 가 어떠한 의미가 있는가 → 이를 통해 **동시 접속자 수를 예상**할 수 있음
  • TPS * (Repsonse Time + Think Time) = 600 * (0.08 + 0.5) = 대략 350명
    • Think Time : 요청을 보내고 두번째 요청을 보낼 때 걸리는 시간
