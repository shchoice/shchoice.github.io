CORS

개요

CORS (Cross-Origin Resource Sharing)는 웹 애플리케이션에서 보안 문제를 해결하기 위해 만들어진 정책으로, 서로 다른 출처 간에 자원 요청을 허용하는 방법을 정의합니다. 이 글에서는 CORS의 개념, 동작 방식, 그리고 Spring Boot에서 CORS 설정 방법을 자세히 알아보자

CORS(Cross Origin Resource Sharing) 란

  • CORS는 웹 애플리케이션이 서로 다른 출처의 자원에 접근할 수 있도록 브라우저에 허용하는 체제
  • 예를 들어, http://example.com에서 호스팅되는 웹 애플리케이션이 http://api.example.com의 데이터를 요청하고자 할 때 CORS 정책이 필요
  • 브라우저는 보안상의 이유로 동일 출처 정책(Same-Origin Policy)을 기본적으로 따르며, 이를 통해 다른 출처에서의 리소스 접근을 제한

출처(Origin)란?

  • 출처는 URL의 프로토콜, 호스트, 포트 번호의 조합을 말합니다. 이 세 가지가 모두 동일한 경우 동일 출처라고 하며, 하나라도 다르면 다른 출처로 간주

  • Protocol + Host + Port 3가지가 같으면 동일 출처(Origin)라고 함

동일 출처 예시

예시 설명
http://example.com:80
http://example.com		 같은 포트 번호 사용
http://example.com/app1/index.html
http://example.com/app2/index.html		 Protocol, Host, Port이 같으며, Path부터 다름

다른 출처 예시

예시 설명
http://example.com/app1
https://example.com/app2		 프로토콜이 다름
http://example.com
http://www.example.com
http://myapp.example.com		 호스트가 다름
http://example.com
http://example.com:8080		 포트가 다름

CORS의 필요성

  • 웹 페이지에서 <img>, <script>, <iframe> 등을 사용하여 외부 자원을 불러오는 것은 매우 일반적

  • 그러나, 이러한 요청이 악의적인 목적을 가진 경우 보안 위험이 발생할 수 있음

  • 예를 들어, 사용자가 악성 스크립트를 포함한 페이지를 방문하면 그 페이지가 사용자의 인증 정보를 사용해 다른 출처에 유해한 요청을 보낼 수 있음

  • CORS는 이러한 보안 문제를 해결하기 위해 브라우저가 다른 출처로부터의 요청을 적절하게 제한하고, 허용된 경우에만 요청을 수행할 수 있도록 도와줌

동일 출처 정책(Same-origin policy)

  • 동일 출처 정책(Same-origin policy)는 다른 출처로부터 조회된 자원들의 읽기 접근을 막아 다른 출처 공격을 예방합

  • 그러나, 다른 출처에서 얻은 이미지를 담는 , 외부 주소를 담는 같은 여러 태그들을 허용

  • 동일 출처 정책의 정확한 구현 명세는 없지만 최신의 브라우저들은 일정 규칙을 따름 ( RFC6454 )

동일 출처 요청 vs 다른 출처 요청 그림

  • 요청하는 클라이언트와 요청받는 서버가 같은 출처에 있으면 동일 출처
  • 서로 다른 서버에 있으면 다른 출처 요청
    • domain-a.com 유저가 domain-a.com 서버에 요청하면 동일 정책이기 때문에 아무런 문제가 없음
    • domain-a.com 유저가 domain-b.com 서버에 요청하면 호스트가 다르기 때문에 다른 출처 요청을 함
    • 도메인 이외에, 같은 프로젝트 내에 정의된 css 파일 요청은 동일 출처 요청이고, font 같이 다른 외부 사이트에서 실시간으로 import를 통해 가져온다면 다른 출처 요청 이처럼, 같은 출처가 아닌 외부에 자원을 요청하는 경우가 있는지 잘 확인해보아야 함

기본적으로 동일 출처 요청만 자유롭게 요청이 가능하며 동일 출처 정책(Same-Origin Policy) 이라고 함

하지만 기준을 완화하여 다른 출처 요청도 할 수 있도록 기준을 만든 체제가 다른 출처 정책(Cross-Origin Policy) 임

CORS의 요청 유형

1. 단순 요청 (Simple Request)

  • 사용 시기

    • GET, POST, HEAD와 같은 안전한 HTTP 메서드를 사용하는 경우.

    • 요청 헤더가 CORS 안전 리스트에 있는 헤더들(예: Accept,Accept-Language, Content-Type 등이 일반적인 값으로 설정된 경우)

      • Contet-Type 헤더는 application/x-www-form-urlencoded, multipart/form-data and text/plain 만 가능

    • 보안이 크게 중요하지 않거나, 간단한 데이터 요청을 수행할 때 적합합니다.

  • 장점 : 브라우저가 추가적인 프리플라이트 요청을 보내지 않으므로 요청이 간소화되고 성능이 조금 더 좋을 수 있음

  • 단점 : 특정 메서드와 헤더에만 제한되므로 유연성이 떨어집니다.

  • 특징 : 브라우저는 요청의 출처를 서버에 전달하며, 서버는 Access-Control-Allow-Origin 헤더를 통해 요청을 허용할지를 결정

  • 사용 예시

    • 간단한 GET 요청으로 API 데이터를 받아오는 경우

      • 성공 케이스

      • 실패 케이스

    • 논리 도해

2. 프리 플라이트(Preflight Request)

  • 사용 시기

    • PUT, DELETE와 같은 안전하지 않은 HTTP 메서드를 사용하거나, 사용자 정의 헤더가 있는 요청을 보낼 때
    • 요청이 중요한 데이터를 수정하거나 민감한 작업을 수행하는 경우
    • 보안이 중요한 경우나, 서버가 클라이언트의 요청을 철저히 검토하고 허용할 필요가 있을 때 적합

  • 장점

    • 서버가 OPTIONS 메소드를 통해 요청을 사전에 검사하고 허용할지 여부를 결정할 수 있으므로 보안성이 높음
    • 서버에서 허용된 메서드와 헤더를 명확하게 정의하여 허용되지 않은 요청을 사전에 차단할 수 있음

  • 단점

    • 브라우저가 두 번의 요청(프리플라이트 요청과 실제 요청)을 보내기 때문에 성능에 약간의 영향을 미칠 수 있음

  • 사용 예시

    • 클라이언트에서 데이터를 삭제하거나 업데이트하는 요청(예: REST API에서 DELETE 요청)에서 사용

    • 프리플라이트 요청은 브라우저가 실제 요청을 보내기 전에 서버에게 미리 “이런 요청을 보내도 되겠습니까?”라고 물어보는 사전 요청

    • 요청은 주로 안전하지 않은 HTTP 메서드(예: PUT, DELETE)나, 사용자 정의 헤더가 포함된 요청을 보낼 때 사용됨

    💡 **참고사항**
    • 요청 헤더에는 다음 값이 포함
      • origin : 어디서 요청을 했는지 서버에 알려주는 주소
      • access-control-request-method : 실제 요청이 보낼 HTTP 메서드
      • access-control-request-headers : 실제 요청에 포함된 header
    • 응답 헤더에는 다음 값이 포함
      • access-control-allow-origin : 서버가 허용하는 출처
      • access-control-allow-methods : 서버가 허용하는 HTTP 메서드 리스트
      • access-control-allow-headers : 서버가 허용하는 header 리스트
      • access-control-max-age : 프리 플라이트 요청의 응답을 캐시에 저장하는 시간
      • 플리플라이트 요청은 OPTIONS 를 사용해 자신의 주소 https://www.api.com?q=test 를 보내며, 또한 origin, access-control-request-method, access-control-request-headers를 같이 보냄
      • 정상적인 응답으로 access-control-allow-origin, access-control-allow-method, access-control-allow-headers, access-control-max-age를 응답받음
      • 정상 요청과 응답이 가능하다는 프리 플라이트 덕분에 실제 요청을 함 그리고 정상적인 응답을 받음

3. 신용 요청(Credentialed Request)

  • 사용 시기

    • 인증 정보(쿠키, 세션 정보, 인증 헤더, TLS 클라이언트 인증서 등)를 포함하여 요청을 보낼 때
    • 사용자의 인증 상태를 유지하면서 서버와 통신해야 하는 경우
    • 인증이 중요한 경우, 특히 사용자의 민감한 데이터를 다루는 서비스에서 사용됨

  • 특징

    • CORS 정책상 기본적으로 신용 요청은 차단되지만, 서버가 Access-Control-Allow-Credentials 헤더를 true로 설정한 경우 허용됨

  • 장점

    • 인증 정보를 포함하여 보안된 요청을 보낼 수 있음

  • 단점

    • 서버가 Access-Control-Allow-Credentials: true를 설정해야 하고, Access-Control-Allow-Origin에서 *을 사용할 수 없으므로 설정이 까다로울 수 있음

  • 사용 예시

    • 인증 정보를 포함하여 보내는 요청
    • 사용자가 로그인 상태에서 서버의 개인 데이터를 조회하거나 수정하는 요청

  • 만약 서버 응답에 access-control-allow-credentials 가 true로 설정되지 않았거나 access-control-allow-origin 헤더에 있는 값이 허용된 출처가 아니라면 아래와 같이 오류가 발생

어떤 요청 방식을 사용해야 할까요?

  • 프리플라이트 요청을 사용하는게 좋음
    • 프리 플라이트 요청으로 실제 요청이 실행되기 이전에 검사를 하고 허용할지 안할지를 결정할 수 있기 때문
    • 신용 요청 방식과 동시에 함께 사용하기도 함
