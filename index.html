Jenkins - SVM 연동 > Multibranch Pipeline 설정 방법

개요

  • Jenkins Multibranch Pipeline을 사용하여 여러 브랜치의 빌드 및 테스트를 자동으로 관리하는 방법을 설명
  • Multibranch Pipeline은 각 브랜치를 자동으로 탐색하고, 브랜치나 PR이 변경될 때마다 이를 감지하여 빌드를 실행하는 강력한 기능을 제공
  • Jenkins를 활용한 Multibranch Pipeline 설정 방법을 단계별로 설명하며, 여러 브랜치 관리와 빌드 및 테스트를 자동으로 관리하는 방법을 소개

설정 방법

0. 플러그인 설치

  1. Multibranch Scan Webhook Trigger

    • Dashboard > Jenkins Management > Plugins > Available plugins 탭

1. Jenkins에서 Job 생성

  • Dashboard > New Item > Multibranch project 선택

  • 프로젝트 이름 작성 > OK 버튼 클릭

2. 구성(Configuration) > 일반 설정(General)

  • Optional 로 원하는 형태로 기술

3. 구성(Configuration) > Branch Sources

  • Add source > Git

    • Project Repository : Git 주소 작성

    • Behaviors

      • Discover branches : 리파지토리의 브랜츠들을 자동으로 탐색
      • Filter by name (with wildcards)
        • include : 브랜치들 중에서 원하는 브랜치들만 필터링하여 탐색함
        • exclude : 브랜치들 중에서 특정 브랜치를 제외하고 탐색함

    • Credentials : Git 서버 인증

      • Add Credentials

        • 세부 설정사항
          • Scope
            • Global : 이 자격 증명이 시스템 전체에서 사용될 수 있음
            • System : 특정 시스템 또는 프로젝트에서만 이 자격 증명을 사용할 수 있음
          • Username : 대상 서버 계정
          • Password : 대상 서버 계정의 패스워드
          • ID (선택사항) : 자격 증명에 대해 Jenkins에서 사용할 고유 식별자를 입력하는 옵션, 기본적으로 이 필드를 비워두면 Jenkins가 자동으로 ID를 생성

    • 참고사항 : 시스템에 Git이 설치가 되어져 있어야 함, 그렇지 않으면 에러 발생

4. 구성(Configuration) > Build Configuration

  • Multibranch pipelin은 Jenkinsfile로 브랜치를 관리 > 6번 항목에서 상세 기술

5. 구성(Configuration) > Scan Multibranch Pipeline Triggers

  • 브랜치 및 PR 변화를 감지하여 빌드를 트리거하는 방법을 설정하는 옵션

  • Scan by webhook

    • 외부 서비스(GitHub, GitLab 등)에서 변경 사항을 푸시하거나 PR이 발생할 때 자동으로 Jenkins에서 해당 변경을 스캔하고 빌드를 트리거할 수 있도록 설정하는 옵션
    • GitHub/GitLab Webhook을 통해 Jenkins가 즉각적인 변화를 감지
    • Yona 에서는 MR(PR)에 대해서는 즉각적인 변화를 감지하지 못함
    • Trigger token : Webhook에서 Jenkins로 요청을 보낼 때 인증을 위해 사용되는 토큰, 이 토큰을 사용하여 외부에서 Jenkins로 브랜치 스캔을 요청할 수 있으며 보안 목적으로 잘 관리되어야 함

  • Periodically if not otherwise run

    • 이 옵션을 활성화하면 Jenkins가 주기적으로 브랜치를 스캔하여 변경 사항이 있는지 확인
    • 변경 사항이 발견되면 빌드를 트리거
    • Yona의 경우는 즉각적인 MR(PR)이 되지 않기 때문에 해당 옵션을 활성화하여 주기적으로 변경 감지를 수행

Jenkins Multi-branch Pipeline 설정 가이드

각 Git 브랜치 (feature, develop, stage, master)에 대해 Jenkins에서 수행해야 하는 작업들을 자세히 설명합니다.

1. Feature 브랜치 파이프라인 설정 (Multi-branch Pipeline)

목적: 개별 기능 개발 후 코드 리뷰 및 유닛 테스트를 통해 코드 품질을 검증한 후 develop 브랜치로 병합합니다.

설명:

  • Checkout: feature 브랜치를 Yona에서 체크아웃하여 Jenkins가 해당 브랜치를 기반으로 빌드하고 테스트를 진행할 수 있도록 합니다.
  • Build & Unit Test: mvn clean install로 빌드하고, mvn test로 유닛 테스트를 실행합니다.
  • Code Coverage: Jacoco 플러그인을 사용해 코드 커버리지를 측정합니다.
  • Post Actions: 빌드 성공/실패 결과를 Slack으로 전송해 개발자에게 알립니다.

MR(Merge Request) 연동:

  • Yona에서 Merge Request(MR)를 생성하면 Jenkins가 이를 감지하고 빌드를 자동으로 실행하도록 Webhook을 설정합니다.
  • MR이 승인되면 Jenkins에서 자동으로 develop 브랜치로 병합합니다.

2. Develop 브랜치 파이프라인 설정

목적: 여러 feature 브랜치가 병합된 후 통합 테스트 및 전체 코드 검증을 수행합니다.

설명:

  • Integration Test: 여러 모듈이 정상적으로 상호작용하는지 확인하는 통합 테스트를 수행합니다.
  • Post Actions: 빌드 실패 시 Slack으로 알림을 전송해 문제를 빠르게 해결할 수 있도록 합니다.

3. Stage 브랜치 파이프라인 설정

목적: stage 환경에서 통합 테스트와 성능 테스트를 실행하여 최종 배포 준비 상태를 확인합니다.

설명:

  • Performance Test: 부하 테스트 및 성능 테스트를 통해 배포 전 시스템이 안정적으로 작동하는지 확인합니다.
  • Post Actions: 빌드 실패 시 빠르게 대응할 수 있도록 Slack으로 알림을 전송합니다.

4. Master 브랜치 파이프라인 설정 (프로덕션 배포)

목적: master 브랜치에 병합된 코드를 프로덕션 서버에 배포합니다.

설명:

  • Deploy to Production: 배포 스크립트를 통해 프로덕션 서버에 배포를 실행합니다.
  • Post Actions: 배포 성공/실패에 대한 알림을 Slack으로 전송합니다.

빌드 실패 시 처리 방식에 대한 설명에서 제가 제공한 내용은 기본적인 절차이지만, 실제 전문적인 DevOps 팀에서는 다음과 같은 보다 체계적이고 자동화된 방식으로 빌드 실패를 처리합니다. 여기서는 더 효율적이고 안정적인 실패 처리 프로세스를 살펴보고, 자동화 도구와 전략을 사용해 신속하게 대응하는 방법을 설명드리겠습니다.

전문적인 DevOps 팀의 빌드 실패 처리 방식

1. 자동화된 오류 감지 및 보고

  • 실시간 모니터링: 빌드 실패 시 Slack, 이메일, Microsoft Teams 등을 통해 실시간으로 개발자와 DevOps 팀에게 자동 알림을 보냅니다. 실패한 빌드 로그와 원인을 포함하여 즉각적인 대응이 가능합니다.

  • Jenkins 플러그인

    : Jenkins와 같은 CI/CD 도구는 여러 플러그인을 사용해 빌드 실패 로그를 상세하게 기록하고 분석할 수 있습니다. 이를 통해 문제의 원인을 신속히 파악할 수 있습니다.

    • 예시: Jenkins Email Extension Plugin을 사용해 빌드 실패 시 이메일로 상세 로그를 전송.

2. 자동 롤백 및 재시도

  • 자동화된 롤백

    : 전문적인 DevOps 팀은 배포 실패 시

    자동 롤백

    절차를 구축해 놓습니다. 이는 특히

    Master

    브랜치에서 배포 실패 시, 시스템을 원상 복구하고 문제를 해결한 후 다시 배포하는 과정이 자동으로 이루어집니다.

    • 예시: Kubernetes를 사용하는 경우, Helm을 통해 자동 롤백을 수행하거나, Ansible과 같은 도구를 사용해 스크립트를 통해 이전 버전으로 복구합니다.

  • 자동 재시도

    : 실패한 빌드는 일반적으로 일정 횟수만큼 자동으로 재시도합니다. Jenkins는 기본적으로 빌드 실패 시

    재시도 설정

    을 지원하며, 이는 일시적인 네트워크 문제나 기타 일시적인 이슈를 해결하는 데 유용합니다.

    • 예시: Jenkins에서 retry 블록을 사용해 빌드를 자동으로 재시도할 수 있습니다.

3. Root Cause Analysis (RCA)

  • 빌드 실패 원인 분석

    : DevOps 팀은 빌드 실패가 빈번하게 발생하면 **Root Cause Analysis (RCA)**를 통해 근본적인 원인을 분석합니다. 이를 통해 빌드 프로세스나 코드에서 발생하는 지속적인 문제를 해결하여 시스템을 개선할 수 있습니다.

    • 예시: 특정 단계에서 반복적인 문제가 발생하는 경우, 빌드 스크립트를 최적화하거나 CI 도구에서 해당 단계를 개선하는 조치가 이루어집니다.

4. 블루/그린 배포 또는 카나리 배포 전략 사용

  • 블루/그린 배포: DevOps 팀은 빌드가 성공하더라도 운영 환경에서 문제가 발생할 수 있기 때문에, 블루/그린 배포 전략을 사용해 트래픽을 새로운 버전과 이전 버전 사이에서 스위칭하면서 문제가 없는지 확인합니다.
  • 카나리 배포: 새로운 버전을 모든 사용자에게 배포하지 않고, 일부 사용자에게만 점진적으로 배포해 문제가 발생하면 즉시 롤백하는 방식입니다. 이 방법은 서비스의 안정성을 유지하면서 새로운 기능을 배포하는 데 매우 유용합니다.

5. Pull Request 기반 빌드 검증 (Pre-merge Checks)

  • PR 빌드 검증

    : 전문적인 DevOps 팀은

    브랜치에서 빌드를 실패시키지 않기 위해,

    Pull Request(PR) 빌드 검증

    을 사용합니다. MR/PR이 merge되기 전에 테스트 및 빌드를 사전 검증하여 빌드 실패를 방지합니다.

    • 예시: GitLab의 CI/CD 파이프라인에서 MR이 발생할 때 자동으로 빌드 및 테스트를 실행하여 merge 여부를 결정합니다.

6. 빌드 실패의 책임 분담 (소유자에게 알림)

  • 빌드 소유자 알림

    : DevOps 팀은 빌드 실패 시, 해당 빌드에 책임이 있는 개발자나 팀에게 직접 알림을 보냅니다. 이를 통해 책임 분담이 명확해지고, 더 빠른 대응이 가능합니다.

    • 예시: Git의 blame 기능이나 Jenkins의 GIT_COMMITTER_NAME 변수를 통해 변경한 개발자에게 직접 알림을 보낼 수 있습니다.

DevOps 팀에서 각 브랜치별 빌드 실패 처리 방식 개선

Feature 브랜치 빌드 실패 시 (개발 환경)

  • 빌드 실패 자동 알림: Slack을 통해 개발자에게 실시간 알림을 보내며, 빌드 실패의 구체적인 로그와 실패 원인까지 포함합니다.
  • 자동 재시도: 간헐적인 네트워크 문제 또는 일시적인 장애를 고려하여 3회 정도 자동으로 재시도합니다.
  • PR/MR 자동 검증: Pull Request/Merge Request 시점에서 자동으로 빌드를 검증해, 문제가 있을 경우 merge되지 않도록 방지합니다.

Develop 브랜치 빌드 실패 시 (통합 테스트 실패)

  • 통합 테스트 로깅: 통합 테스트가 실패한 경우, 자동으로 로그를 분석하여 상세한 실패 원인을 Slack 및 이메일로 관련 개발자에게 전달합니다.
  • 자동화된 이슈 생성: 빌드 실패가 발생할 경우, Yona나 GitLab과 연동하여 자동으로 이슈 트래킹 시스템에 이슈가 생성됩니다. 이를 통해 누락되는 문제 없이 추적 가능합니다.

Stage 브랜치 빌드 실패 시 (성능 테스트 실패)

  • 성능 데이터 분석: 성능 테스트 실패 시 자동으로 성능 데이터를 분석하고, 차트나 그래프 형태로 Slack 또는 이슈 트래킹 시스템에 보고합니다. 이를 통해 성능 문제를 시각적으로 빠르게 파악할 수 있습니다.
  • 성능 테스트 재시도: 성능 테스트에 실패한 경우에도 일시적인 이슈일 수 있으므로 자동 재시도 기능을 활용합니다.

Master 브랜치 배포 실패 시 (프로덕션 환경)

  • 자동 롤백: 배포에 실패하면 자동으로 이전 버전으로 롤백하며, 시스템 가동에 문제가 없도록 합니다. 이는 Ansible, Helm, Terraform과 같은 도구를 통해 수행됩니다.
  • 배포 이력 관리: 모든 배포 이력은 자동으로 저장되며, 이를 기반으로 특정 버전에서의 문제를 추적할 수 있습니다.
  • 카나리 배포 전략: 모든 사용자에게 새로운 버전을 배포하기 전에, 일부 사용자에게만 배포하여 문제가 없는지 확인합니다.

결론

전문적인 DevOps 팀에서는 빌드 실패 시 더 자동화되고, 빠른 대응이 가능한 시스템을 구축하여 팀의 생산성을 높입니다. 기본적인 실패 대응 방법 외에도, 자동 롤백, 재시도, 성능 데이터 분석, 블루/그린 배포 전략, PR/MR 자동 검증 등의 고급 DevOps 전략을 통해 더욱 안정적이고 신속하게 문제를 해결합니다. 이를 통해 빌드와 배포에서 발생하는 장애를 최소화하고, 코드 품질을 유지할 수 있습니다.

Jenkins - SVM 연동 > Freestyle Project 설정 방법

개요

  • Jenkins의 Freestyle Project는 소규모 프로젝트나 단일 브랜치를 관리할 때 간단한 빌드 및 배포 작업에 매우 유용
  • Jenkins와 SVM을 연동하여 Freestyle Project로 단일 브랜치 관리하여 자동으로 빌드를 유발하는 방법을 알아보자

설정 방법

1. Jenkins에서 Job 생성

  • Dashboard > New Item > Freestyle project 선택

  • 프로젝트 이름 작성 > OK 버튼 클릭

2. 구성(Configuration) > 일반 설정(General)

  • 옵션설명
    • Github 과 연동일 경우 Github Project 클릭
    • Throttle builds : 빌드를 동시에 실행하는 상황에서 빌드의 병렬 실행 수를 제한하거나 특정 노드에서만 실행하도록 제어하는 기능(일반적으로는 자주 사용되는 옵션은 아님)

3. 구성(Configuration) > 소스 코드 관리(Source Code Management)

  • 옵션 설명

    • Git을 선택하고, Repository URL에 프로젝트의 Git 주소를 입력

      • 참고사항

        시스템에 Git이 설치가 되어져 있어야 함, 안그러면 에러 발생

    • Branches to build에서 빌드할 브랜치를 선택합니다. 단일 브랜치만 관리할 경우, 특정 브랜치(ex: master)만 선택

      • 참고사항

        Add Branch를 통해서 */master 브랜치, */develop 브랜치, */jenkins-test 3개의 브랜치를 만들어 보았으나 마지막에 설정했던 */jenkins-test 브랜치만 동작, 따라서 1개씩 여러 프로젝트를 만들거나, multibranch pipeline 을 사용하자

    • Credentials : Git 서버 인증

      • Add Credentials

        • 세부 설정사항
          • Scope
            • Global : 이 자격 증명이 시스템 전체에서 사용될 수 있음
            • System : 특정 시스템 또는 프로젝트에서만 이 자격 증명을 사용할 수 있음
          • Username : 대상 서버 계정
          • Password : 대상 서버 계정의 패스워드
          • ID (선택사항) : 자격 증명에 대해 Jenkins에서 사용할 고유 식별자를 입력하는 옵션, 기본적으로 이 필드를 비워두면 Jenkins가 자동으로 ID를 생성

4. 구성(Configuration) > 빌드 유발(Build Triggers)

  • 옵션 설명

    • 빌드를 원격으로 유발 (예: 스크립트 사용)

      • 외부 시스템(스크립트, CI/CD 파이프라인 등)에서 Jenkins 빌드를 원격으로 트리거할 수 있음

      • HTTP 요청을 통해 Jenkins API를 호출하여 빌드를 유발할 수 있음

      • 외부 도구나 서비스에서 Jenkins 빌드를 트리거해야 하는 경우 사용됨

        ※ Yona SVM 같은 경우는 마이너한 제품이라 설정 필요

    • Build after other projects are built

      • 다른 프로젝트의 빌드가 완료된 후 현재 프로젝트의 빌드를 트리거하는 옵션
      • 업스트림/다운스트림 빌드 관계를 설정할 수 있음
      • 프로젝트 간 의존성이 있는 경우 사용

    • Build periodically

      • 주기적으로 빌드를 실행하는 옵션입니다. Cron 표현식을 사용

    • GitHub hook trigger for GITScm polling

      • GitHub Webhook을 통해 GitHub에서 커밋, PR(Pull Request) 등의 이벤트가 발생하면 Jenkins가 이를 감지하고 자동으로 빌드를 트리거하는 옵션

    • Poll SCM

      • SCM을 주기적으로 확인하여 변경 사항이 있을 때 빌드를 트리거하는 옵션, Cron 표현식을 사용하여 폴링 주기를 설정

      • Webhook이 지원되지 않는 환경에서 사용됨

※ 위 사진에서 설명이 안되어져 있는 부분은 별도의 플러그인 설치 후 생긴 것

5. 구성(Configuration) > 빌드 환경 (Build Environment)

  • 옵션 설명
    • Delete workspace before build starts
      • 빌드를 시작하기 전에 워크스페이스를 삭제하여 이전 빌드의 잔여물을 제거하고, 깨끗한 환경에서 빌드를 시작
      • 특히 이전 빌드의 결과물이 다음 빌드에 영향을 줄 수 있는 경우 유용
      • 프로젝트가 의존성 문제로 인해 실패할 가능성이 있는 경우, 이 옵션을 활성화하여 클린 상태에서 시작할 수 있음
      • 안정성과 일관성을 위해 자주 사용됨
    • Use secret text(s) or file(s)
      • 빌드 환경에서 비밀 정보(예: API 키, 비밀번호)나 파일을 사용하는 경우, 이를 안전하게 관리하고 사용할 수 있도록 설정
    • Add timestamps to the Console Output
      • 빌드 과정에서 시간 정보를 추적하여 성능 문제를 분석하거나 빌드 과정의 정확한 실행 시간을 확인
      • 빌드 시간을 분석하고 최적화할 때 많이 사용함
    • Inspect build log for published build scans
      • gradle 또는 Maven 프로젝트에서 빌드 스캔 기능을 사용할 때 유용
      • 빌드 성능, 문제 원인, 빌드 과정을 추적하고 분석할 때 유용
    • Terminate a build if it’s stuck
      • 빌드가 특정 시간 이상 실행되지 않고 멈춰 있는 경우 자동으로 빌드를 중단하는 옵션
    • With Ant
      • Apache Ant를 사용하는 프로젝트에서 빌드를 실행할 때 사용

6. 구성(Configuration) > 빌드 단계(Build Steps)

  • Spring 을 사용할 경우 gradlew 가 함께 SVM에 올라가져 있을 경우, 별도로 사용하지 않아도 gradle 빌드가 됨
  • 프로젝트에 맞는 빌드 환경에 맞게 설정

7. 구성(Configuration) > 빌드 후 조치(Post-build Actions)

  • 빌드가 성공적으로 완료된 후 알림이나 배포 작업을 설정할 수 있음

8. SVM에서의 설정 > Web hook

Jenkins에서 다양한 빌드 옵션 선택하기 > Multi-branch Pipeline vs Freestyle Project

개요

  • Jenkins는 소프트웨어 빌드, 테스트, 배포를 자동화하는 데 널리 사용되는 도구
  • Jenkins에서 프로젝트를 자동화하려면 다양한 Item(빌드 유형) 중에서 적합한 것을 선택해야 함
  • Jenkins의 대표적인 두 가지 옵션인 Multi-branch Pipeline과 Freestyle Project를 비교하고, 각각의 장단점을 살펴보자

Jenkins의 주요 Item들

각 Item은 프로젝트의 크기, 복잡성, 그리고 배포 전략에 따라 적합한 경우가 다름

Freestyle Project

  • 적합한 경우
    • Jenkins에서 제공하는 가장 기본적인 작업 유형으로, 소규모 프로젝트나 단순한 빌드/테스트 작업에 적합
    • 복잡한 파이프라인 설정 없이 간단한 빌드 및 테스트 작업을 설정할 수 있음
  • 특징
    • 간단한 설정: 복잡한 설정 없이 빠르게 Jenkins 작업을 설정할 수 있
    • 스크립트 없이도 사용 가능: Jenkins의 기본 GUI로 빌드, 테스트, 배포 작업을 설정할 수 있으며, 추가적인 스크립트가 필요 없
    • 제한된 확장성: 브랜치 전략이 복잡하거나 여러 단계의 파이프라인이 필요한 경우는 적합하지 않으며, 모든 브랜치에서 동일한 빌드 규칙을 적용
  • 장점
    • 사용이 간편하고 빠르게 설정 가능
    • 복잡한 파이프라인 설정이 필요 없는 소규모 프로젝트에 적합
  • 단점
    • 복잡한 브랜치 전략을 지원하지 않음
    • 대규모 프로젝트에서 관리하기 어려움

Pipeline Project

  • 적합한 경우
    • Jenkins 스크립트를 통해 여러 단계의 빌드 및 배포 과정을 자동화할 수 있는 경우 적합
    • Jenkinsfile을 사용해 파이프라인을 정의하고, 이를 통해 다양한 단계를 거치는 빌드를 구성할 수 있음
  • 특징
    • Jenkinsfile 기반: 빌드, 테스트, 배포의 각 단계를 코드(Jenkinsfile)로 관리할 수 있음
    • 유연성: 복잡한 빌드 과정이나 여러 단계를 가진 파이프라인을 쉽게 구성할 수 있음
    • 단일 브랜치: Multi-branch Pipeline과는 달리, 단일 브랜치에만 적용되며 모든 브랜치별로 따로 파이프라인을 작성해야 함
  • 장점
    • 복잡한 파이프라인을 코드로 작성할 수 있어 버전 관리가 용이
    • 단계별 빌드 및 테스트를 구체적으로 정의할 수 있어 유연함
  • 단점
    • Jenkinsfile을 작성해야 하기 때문에 Freestyle Project에 비해 초기 설정이 복잡함
    • 브랜치마다 별도의 Jenkinsfile을 유지 관리해야 할 수 있음

Multi-branch Pipeline Project

  • 적합한 경우
    • 여러 브랜치에 대해 서로 다른 파이프라인을 자동으로 설정하고, 각 브랜치에 맞는 빌드 및 테스트를 관리하고자 할 때 적합
    • Git 브랜치 전략이 복잡하거나, 여러 브랜치에서 독립적인 파이프라인을 실행해야 하는 경우에 매우 유용함
  • 특징
    • 자동 브랜치 감지: Git 저장소에 있는 모든 브랜치를 자동으로 탐색하고, 각 브랜치에 맞는 Jenkinsfile을 실행
    • 브랜치별 독립적 관리: 각 브랜치에 따라 독립적인 파이프라인을 설정할 수 있어 브랜치 전략을 충실히 반영할 수 있음
    • 자동화 수준이 높음: 브랜치가 새로 생성되면 자동으로 파이프라인이 설정되고 빌드가 실행됨
  • 장점
    • 복잡한 브랜치 전략을 효과적으로 관리할 수 있음
    • 브랜치별 파이프라인이 자동화되어 별도로 관리할 필요가 적음
    • GitLab Flow나 GitFlow와 같은 브랜치 전략을 사용하는 경우에 특히 유용함
  • 단점
    • Jenkins 설정이 상대적으로 복잡할 수 있음
    • 브랜치별로 Jenkinsfile을 관리해야 하므로 코드 유지 보수가 필요할 수 있음

Multi-branch Pipeline vs Freestyle Project: 차이점 요약

특성 Multi-branch Pipeline Freestyle Project
브랜치 관리 여러 브랜치를 자동으로 관리 모든 브랜치에 동일한 설정 적용
자동화 수준 각 브랜치에 맞는 파이프라인을 자동 실행 수동으로 설정해야 함
사용 난이도 Jenkins 설정이 복잡할 수 있음 설정이 간단함
확장성 대규모 프로젝트에 적합 소규모 프로젝트에 적합
파이프라인 관리 방법 Jenkinsfile을 통한 코드 관리 Jenkins GUI로 관리

어떤 Item을 선택해야 할까?

  1. 프로젝트의 규모와 복잡도에 따라
    • 소규모 프로젝트: 간단한 빌드와 테스트 과정만 필요한 소규모 프로젝트라면 Freestyle Project를 사용하는 것이 적합
    • 대규모 프로젝트: 복잡한 브랜치 전략을 따르거나, 각 브랜치마다 독립적인 빌드와 테스트가 필요하다면 Multi-branch Pipeline을 사용하는 것이 효율적
  2. 브랜치 전략이 복잡한 경우
    • GitFlow나 GitLab Flow와 같은 브랜치 전략을 사용하는 경우, Multi-branch Pipeline이 적합
    • 모든 브랜치를 자동으로 감지하고, 브랜치마다 독립적인 빌드를 실행할 수 있기 때문
  3. 자동화 요구 사항에 따라
    • 복잡한 파이프라인을 관리하고, 다양한 단계를 통해 빌드 및 배포를 자동화하고자 한다면 Pipeline 또는 Multi-branch Pipeline을 사용하는 것이 적합

Git Commit Rule

개요

  • Git 커밋 메시지는 프로젝트의 히스토리와 팀 간의 협업을 기록하는 중요한 역할을 함
  • 일관된 커밋 메시지 규칙을 사용하여 코드 변경 사항을 체계적으로 관리하고, 코드 리뷰 및 버전 관리 작업을 더욱 쉽게 진행할 수 있음
  • 따라서 Git commit rule을 소개하고, 이를 Java 웹 개발 환경에 맞춰 설명

1. Git Commit Rule이 중요한 이유

Git 커밋 규칙은 코드 변경 사항을 명확히 기록하여 여러 가지 이점을 제공

  • 명확성: 변경 사항을 추적하기 쉽고, 이유를 이해하는 데 도움이 됨
  • 협업 효율성: 팀원들이 커밋 메시지만 보고도 어떤 변경이 이루어졌는지 쉽게 파악할 수 있음
  • 자동화 도구 연계: 커밋 메시지가 표준화되면 CI/CD 파이프라인에서 자동화된 릴리스 관리 등이 가능해짐

2. 전문 개발팀의 Git Commit Rule

2.1 Conventional Commits

Conventional Commits는 대규모 프로젝트에서 많이 사용되는 커밋 규칙입니다. 커밋 메시지 형식을 표준화하여 코드 관리 및 협업을 용이하게 만듭니다.

Conventional Commits 형식

  • 타입: 커밋의 목적을 나타내는 부분

    • feat: 새로운 기능 추가
    • fix : 버그 수정
    • docs : 문서 수정
    • style : 코드 스타일 변경 (코드 formatting, 세미콜론(;) 누락 등, 기능 변화 없음)
    • refactor : 리팩토링 (기능 변경 없음)
    • perf : 성능 개선 (성능 최적화를 위한 코드 변경)
    • build : 빌드 시스템 수정 (예: 패키지 매니저 설정 변경, 의존성 추가/업데이트)
    • test: 테스트 코드 추가/수정
    • anno : 주석 작업
    • exp : 실험적인 기능 추가/변경 (예: 실험적인 코드 또는 기능 테스트)
    • merge : 코드 합칠 경우
    • revert : 이전 커밋 되돌리기
    • chore: 코드의 기능 변화와 무관한 유지보수 작업(예: 로그 정리, 설정 파일 변경 등)
    • etc : 기타

  • 스코프: 선택 사항으로, 변경된 코드의 범위나 모듈을 명시합니다. 예를 들어, Java 웹 개발에서는 controller, service, repository 등의 스코프를 사용할 수 있음

  • 커밋 메시지 요약: 간결한 한 줄로 커밋의 핵심을 설명하며, 50자 이내로 작성하는 것이 좋음

3. 커밋 메시지 작성 방법

3.1 커밋 제목 작성 방법

  • 커밋 제목은 변경 사항을 간결하게 설명하며, 명령형으로 작성하는 것이 좋음

  • 제목은 가능한 한 50자를 넘지 않도록 함

    • feat: 새로운 기능 추가
    • controller: 컨트롤러 레이어에서의 변경 사항
    • add new endpoint for user registration: 사용자 등록을 위한 엔드포인트 추가

3.2 커밋 본문 작성 방법

  • 커밋 본문은 더 복잡한 변경 사항에 대한 설명을 포함하며, 변경된 코드가 어떤 문제를 해결하는지, 또는 어떤 새로운 기능을 추가했는지 설명
  • 본문은 선택 사항이지만, 중요한 변경 사항일 경우 작성하는 것이 좋음

3.3 이슈 번호와 연동 (Closes #)

  • 커밋 메시지에 Closes #<이슈 번호>를 추가하면, 커밋이 머지될 때 자동으로 해당 이슈가 닫힘
  • 이를 통해 커밋과 이슈 트래커를 쉽게 연동할 수 있음

결론

  • 일관된 Git commit rule을 사용하는 것은 코드 관리의 기본
  • Conventional Commits와 같은 표준 규칙을 따름으로써 코드 히스토리를 명확하게 유지할 수 있고, 팀 간의 협업을 원활하게 진행할 수 있음

Git Merge와 Rebae의 차이

개요

  • Git에서 rebase는 두 개의 브랜치를 합칠 때 사용되는 방식 중 하나로, 현재 브랜치의 변경 사항을 기준 브랜치의 가장 최신 커밋 뒤에 덮어씌우는 방식으로 작업을 정리
  • 이는 병합(Merge)과 유사하지만, Git의 커밋 히스토리를 더 간결하고 깔끔하게 유지할 수 있음

1. Git Rebase의 기본 개념

  • rebase는 특정 브랜치에서 작업한 커밋들을 다른 브랜치의 최신 커밋 이후에 다시 적용하는 방식
  • 이를 통해, 마치 모든 작업이 동일한 커밋 흐름 안에서 진행된 것처럼 새로운 커밋 히스토리를 생성

rebase와 merge의 차이

  • Merge
    • 두 브랜치를 병합할 때, 두 브랜치의 공통 조상 이후 각 브랜치의 변경 사항을 결합하여 새로운 병합 커밋(Merge Commit)을 생성
  • Rebase
    • 두 브랜치의 커밋을 일렬로 정렬하여, 마치 한 브랜치에서 계속 작업이 이루어진 것처럼 새로운 히스토리를 만듬
    • 병합 커밋이 생기지 않으며, 기존 커밋을 재작성하는 방식

2. 사용 예시

1) rebase

  • test-01 브랜치 commit 이력

  • test-03 브랜치 commit 이력

  • rebase test-03 onto test-01

    • test-01의 commit 이력이 먼저 commit 되고 그 위에 test-03 의 내용이 commit 되는 것을 볼 수 있음, 시간 순서가 아니라 브랜치별 일렬로 쌓이는 것을 볼 수 있음

    2) merge

  • test-01 브랜치 commit 이력

  • test-02 브랜치 commit 이력

  • merge test-01 into test-02

3. Rebase의 장점

  1. 깔끔한 커밋 히스토리
    • rebase는 병합 커밋 없이 커밋 히스토리를 일렬로 정리
    • 결과적으로, 마치 모든 커밋이 순차적으로 이루어진 것처럼 보임
  2. 간결한 히스토리 관리
    • 복잡한 브랜치 구조를 간소화하고 커밋 히스토리를 간결하게 유지할 수 있음
  3. 최신 코드 기반 작업
    • 작업 중인 브랜치에서 최신 상태로 코드를 동기화하여, 최신 코드 기반에서 작업을 계속할 수 있음

4. Rebase의 주의사항

  • 공개된 브랜치에서 rebase를 피하는 것이 좋음
    • rebase는 기존 커밋을 다시 작성하기 때문에, 이미 다른 사람과 공유한 브랜치에서 rebase를 수행하면 커밋 충돌이 발생할 수 있음
    • 따라서, 공개된 브랜치에서는 rebase 대신 merge를 사용하는 것이 더 안전
  • 충돌 해결: rebase 중 충돌이 발생하면, 충돌을 해결한 뒤 git rebase –continue 명령어를 사용해 작업을 이어가야함

5. Rebase vs. Merge 비교

특성 Rebase Merge
커밋 히스토리 깔끔하고 일관된 히스토리를 유지 병합 커밋이 추가되어 히스토리가 복잡해짐
커밋 충돌 해결 rebase 중 커밋 충돌을 순차적으로 해결 병합 시 한 번에 충돌 해결
병합 커밋 없음 있음
사용 시점 개인 작업을 깔끔하게 정리할 때 협업 브랜치나 공개 브랜치에서 사용

6. Rebase 사용 시의 전략

  • 개인 작업이나 feature 브랜치에서 작업한 후, 최신 상태로 코드를 동기화할 때 rebase를 사용
  • 협업 중인 브랜치에서 커밋 히스토리를 유지해야 하는 경우에는 merge를 사용하는 것이 적합

결론

  • Git rebase는 커밋 히스토리를 깔끔하게 정리하고, 최신 코드 기반에서 작업을 진행할 수 있도록 돕는 강력한 도구
  • 하지만 공개된 브랜치에서 rebase는 주의해야 하며, 커밋 충돌이 발생할 가능성을 염두에 두고 사용해야 함
  • rebase와 merge는 각각의 장단점이 있으므로, 상황에 맞게 적절히 사용하여 Git 히스토리를 효율적으로 관리하는 것이 중요

Git 브랜치 전략 - git/gitlab/github flow

개요

  • Git 브랜치는 코드의 독립적인 작업 공간을 제공하여, 여러 개발자가 동시에 다른 작업을 진행하더라도 코드 충돌 없이 협업할 수 있게 해준다
  • 브랜치 전략을 통해 코드의 안정성, 협업의 효율성, 그리고 버전 관리가 체계적으로 이루어진다.
  • 브랜치 전략 Git Flow, Github Flow, GitLab Flow 에 대해 알아 보도록 하자

Git 브랜치란

  • Git에서 브랜치는 코드의 독립된 작업 공간을 의미하며, 프로젝트의 메인 코드와 분리된 공간에서 코드를 수정하거나 새로운 기능을 추가할 수 있음
  • 각 브랜치는 다른 브랜치와 독립적으로 동작하며, 개발 중인 코드를 안정적인 상태의 코드에 영향을 주지 않고 동시에 여러 작업을 진행할 수 있도록 해줌

브랜치의 핵심 기능

  • 독립성: 브랜치는 다른 작업에 영향을 주지 않으며, 각 작업을 독립적으로 처리할 수 있음
  • 병합(Merge): 작업이 완료되면 브랜치를 메인 브랜치에 병합하여 변경 사항을 반영할 수 있음
  • 버그 수정이나 새로운 기능 추가: 작업 중에 문제가 발생하거나 새로운 기능을 개발할 때, 별도의 브랜치에서 안전하게 작업할 수 있음

Git 브랜치 전략이 필요한 이유

  1. 코드 안정성 보장: 여러 개발자가 동시에 작업하더라도, 메인 브랜치(보통 main 또는 master)는 항상 안정적인 상태를 유지할 수 있음
  2. 효율적인 협업: 각각의 개발자가 독립적인 브랜치에서 작업하면서, 서로의 작업을 방해하지 않고 효율적으로 협업할 수 있음
  3. 버전 관리: 여러 버전의 코드를 동시에 관리할 수 있어, 긴급한 버그 수정이나 새로운 기능 개발을 병행할 수 있음
  4. 문제 발생 시 복구 용이: 작업 중에 문제가 발생하면, 문제가 있는 브랜치를 간단히 롤백하거나 병합하지 않음으로써 쉽게 복구할 수 있음

대표적인 Git 브랜치 전략

1) Git Flow

  • Git Flow는 가장 널리 사용되는 브랜치 전략 중 하나로, 긴 개발 주기를 가진 프로젝트에 적합
  • 다음과 같은 브랜치를 사용
    • main 브랜치: 프로덕션에 배포되는 최종 코드가 있는 브랜치로, 항상 안정적인 상태여야 하며, 버그가 없는 상태만 유지
    • develop 브랜치: 현재 개발 중인 코드가 모여있는 브랜치로, 새로운 기능이나 버그 수정을 위한 작업이 완료되면, develop 브랜치에 병합됨
    • feature 브랜치: 새로운 기능을 개발할 때 사용하는 브랜치로, develop 브랜치에서 분기하여 독립적으로 기능을 추가하고, 작업이 완료되면 다시 develop에 병합됨
    • release 브랜치: 배포를 준비할 때 사용하는 브랜치로, develop 브랜치에서 release 브랜치를 생성한 뒤, 최종 테스트 및 버그 수정 후 main 브랜치에 병합됨
    • hotfix 브랜치: 긴급한 버그가 발생했을 때 사용하는 브랜치로,main 브랜치에서 hotfix 브랜치를 생성하여 문제를 수정한 후, 수정 사항을 main과 develop에 모두 반영

  • Git Flow의 장점

    • 명확한 브랜치 구조: 기능 개발, 버그 수정, 배포 준비, 긴급 수정 등 각 작업이 명확하게 구분됨
    • 안정적인 배포 프로세스: 개발 중인 코드와 프로덕션 배포 코드를 철저히 구분할 수 있어, 배포 준비가 안정적으로 이루어짐

  • Git Flow의 단점

    • 복잡성: 브랜치가 많아져서 구조가 복잡할 수 있음
    • 특히 짧은 개발 주기나 빠른 배포가 요구되는 프로젝트에서는 관리가 까다로울 수 있음

2) GitHub Flow

  • GitHub Flow는 상대적으로 단순한 브랜치 전략으로, 빠른 배포와 짧은 개발 주기를 가진 프로젝트에 적합
  • GitHub Flow는 다음과 같은 브랜치를 사용
    • main 브랜치: 프로덕션에서 사용되는 최종 코드가 위치한 브랜치로, 항상 배포 가능한 안정적인 코드만 포함
    • feature 브랜치: 새로운 기능이나 버그 수정 작업을 할 때 main 브랜치에서 분기하여 작업을 진행하며, 작업이 완료되면 main 브랜치에 병합하고 배포

  • GitHub Flow의 장점

    • 단순한 구조: 브랜치 구조가 간단하여 관리하기 쉽고, 개발 및 배포 주기가 짧은 프로젝트에 적합
    • 빠른 배포: 기능 개발이 완료되면 바로 main 브랜치에 병합하여 빠르게 배포할 수 있음

  • GitHub Flow의 단점

    • 긴 개발 주기에는 부적합: 복잡한 프로젝트에서는 개발 단계와 배포 단계가 명확히 구분되지 않아 관리가 어려움
    • 대규모 프로젝트에는 비효율적: 대규모 협업 프로젝트에서는 브랜치 관리가 불안정해질 수 있음

  • GitHub Flow 사용 예시

3) GitLab Flow

  • GitLab Flow는 GitHub Flow와 Git Flow의 장점을 결합한 브랜치 전략으로, 단순한 구조와 배포 환경을 고려한 흐름을 제공

  • 특히, 배포 환경(프로덕션, 스테이징 등)에 따라 브랜치를 추가하여 운영

  • 다음과 같은 브랜치를 사용

    • main 브랜치: 프로덕션에서 사용되는 최종 코드
    • feature 브랜치: 새로운 기능이나 버그 수정을 위한 작업 브랜치
    • 환경별 브랜치: production(프로덕션 환경), staging(스테이징 환경)과 같은 환경별 브랜치가 추가되어, 코드가 각 환경에서 별도로 테스트되고 배포될 수 있음

  • GitLab Flow의 장점

    • 환경별 배포: 스테이징, 프로덕션 등 환경에 맞춘 배포가 가능하여, 배포 프로세스가 더 체계적
    • 단순성: 구조는 단순하지만, GitHub Flow에 비해 더 많은 배포 환경을 지원할 수 있음

  • GitLab Flow의 단점

    • 환경 설정의 복잡성: 환경별로 브랜치를 관리해야 하므로, 환경 설정이 복잡해질 수 있음

  • GitLab Flow 사용 예시

브랜치 전략 선택 가이드

각 전략은 프로젝트의 규모, 팀 구성, 배포 주기에 따라 선택

  • 대규모 프로젝트나 긴 개발 주기를 가진 프로젝트
    • Git Flow를 추천, 명확한 브랜치 구조와 안정적인 배포 과정을 유지할 수 있음
  • 빠른 개발 및 배포가 필요한 프로젝트
    • GitHub Flow가 적합빠, 르게 새로운 기능을 추가하고, 바로 배포할 수 있음
  • 다양한 배포 환경을 고려하는 프로젝트
    • GitLab Flow를 사용하여 환경별로 브랜치를 관리하고, 각 환경에 맞게 테스트하고 배포할 수 있음

결론

  • Git 브랜치 전략은 협업 개발에서 코드의 안정성, 배포 관리, 효율적인 협업을 가능하게 하는 중요한 전략
  • 개발자는 프로젝트의 요구 사항에 맞는 적절한 브랜치 전략을 선택하여, 코드 품질을 유지하고 원활한 배포를 관리해야 함
  • 각 브랜치 전략의 장단점을 이해하고, 팀과 프로젝트 상황에 맞게 유연하게 적용하는 것이 성공적인 개발의 중요한 요소

OSI 7계층과 TCP/IP 4계층의 차이

개요

  • OSI 7계층 모델과 TCP/IP 4계층 모델은 둘 다 네트워크 통신을 설명하는 참조 모델이지만, 그들의 목적과 구체적인 사용 방법이 다름
  • 이 두 모델을 구분하는 관점은 추상화 수준과 구체적인 사용 사례
  • 이 차이를 명확하게 이해해보자

1. 추상화 수준

  • OSI 7계층 모델

    • 이론적이고 세밀하게 통신 과정을 7단계로 나눠 설명하는 참조 모델

    • 각 계층은 특정 기능을 담당하고 있으며, 데이터의 흐름이 어떻게 일어나는지를 구조화된 방식으로 설명

    • 7계층 구조: 물리 계층, 데이터 링크 계층, 네트워크 계층, 전송 계층, 세션 계층, 표현 계층, 애플리케이션 계층

    • 이 모델은 네트워크의 모든 기능을 세부적으로 나누어 설명하는 데 유용

  • TCP/IP 4계층 모델

    • 실제 구현에 중점을 둔 실용적인 모델로, 인터넷 프로토콜 스위트를 기반으로 한 더 단순한 계층 구조
    • 실제 네트워크에서 동작하는 프로토콜들을 중심으로 만들어졌으며, TCP/IP 프로토콜 스택에 맞게 더 간단한 구조

    를 제공

    • 4계층 구조: 네트워크 인터페이스 계층, 인터넷 계층, 전송 계층, 애플리케이션 계층
    • TCP/IP 모델은 인터넷과 실제 네트워크 구현에 사용되는 프로토콜을 중심으로 설계됨

2. 사용 사례와 목적

  • OSI 모델
    • 이론적 교육이나 네트워크 문제를 분석할 때 많이 사용됨
    • 각 계층의 역할이 명확히 정의되어 있어, 특정 문제가 어느 계층에서 발생했는지 추적하거나 네트워크 설계를 설명할 때 유용
    • 각 계층을 나누어 생각함으로써 각 프로토콜이나 장치가 어떤 계층에 속하는지 분석할 수 있음
  • TCP/IP 모델
    • 실제 네트워크 프로토콜 구현에 사용됨 TCP/IP 프로토콜 스택에 의해 인터넷과 대부분의 네트워크가 실제로 작동하며, 실제 네트워크 장비(라우터, 스위치)에서 이 모델을 기반으로 네트워크가 구성됨
    • **인터넷 프로토콜(IP)**과 **전송 프로토콜(TCP/UDP)**를 설명하는 데에 집중되어 있어 실무적인 네트워크 구현과 설정에 더 적합

3. 계층 간의 기능 비교

OSI 7계층 TCP/IP 4계층 기능 설명
1. 물리 계층 물리적인 전송 매체(케이블, 신호)
2. 데이터 링크 계층 1. 네트워크 인터페이스 계층 네트워크 인터페이스를 통한 데이터 전송(이더넷, MAC 주소)
3. 네트워크 계층 2. 인터넷 계층 IP 주소 기반 패킷 전송 및 라우팅 (IP, ICMP 프로토콜)
4. 전송 계층 3. 전송 계층 데이터의 신뢰성, 흐름 제어 (TCP/UDP 프로토콜)
5. 세션 계층 연결 세션을 관리하는 역할, 오늘날에는 거의 사용되지 않음
6. 표현 계층 데이터 포맷, 인코딩/디코딩, 암호화 처리
7. 애플리케이션 계층 4. 애플리케이션 계층 애플리케이션에 데이터를 전달하는 최종 계층 (HTTP, FTP, SMTP 등)
  • 네트워크 인터페이스 계층(TCP/IP)은 물리 계층과 데이터 링크 계층(OSI 모델)을 포함하며, TCP/IP의 인터넷 계층은 OSI의 네트워크 계층과 유사
  • 전송 계층은 두 모델에서 모두 데이터 전송의 신뢰성과 연결을 처리하는데, TCP/IP에서는 주로 TCP와 UDP 프로토콜을 다룸
  • 애플리케이션 계층은 두 모델에서 모두 애플리케이션에서 사용되는 프로토콜을 처리하지만, OSI는 이 계층을 좀 더 세분화하여 세션 계층과 표현 계층을 별도로 구분

4. 구현 vs 이론의 차이

  • OSI 7계층 모델은 이론적이며, 실제 네트워크 프로토콜 구현에 있어서 참조 모델로 사용됨, 각각의 계층에서 발생하는 기능을 독립적으로 분리해 이해하는 데 유용하며, 교육 목적이나 문제 해결 시 도움을 줌
  • TCP/IP 4계층 모델은 인터넷을 포함한 실제 네트워크에서 사용되는 프로토콜들을 설명하기 위해 설계되었으며, 실제 네트워크 통신을 더 직접적으로 반영

결론: 어떤 관점에서 구분해야 할까?

  • 이론적 분석 및 교육

    • 네트워크의 각각의 계층을 세부적으로 이해하고, 네트워크 문제를 구조화된 방식으로 분석하고 싶을 때는 OSI 7계층 모델을 사용
    • 이는 학습과 네트워크 설계, 문제 해결 시에 유용함

  • 실제 네트워크 구현 및 운영

    • TCP/IP 4계층 모델은 실제 네트워크 프로토콜(예: IP, TCP, HTTP)을 기반으로 동작하며, 실제 네트워크 설정이나 장비 구성 시 많이 사용됨
    • 이는 인터넷과 네트워크 장비에서 직접 사용되므로, 실무적인 구현에 더 적합함

두 모델을 사용하는 목적에 따라 OSI 모델을 이론적으로 사용하거나, TCP/IP 모델을 실제 네트워크 구현에서 사용하는 것이 일반적

L4 스위치의 로드 밸런싱 과 웹 서버의 로드 밸런싱 차이

개요

  • 리버스 프록시의 로드 밸런싱과 L4 스위치를 통한 로드 밸런싱은 서로 다른 계층에서 동작하며, 각각 장단점이 있음
  • L4 스위치는 네트워크 계층에서 트래픽을 분산하는 반면,
  • 리버스 프록시는 애플리케이션 계층에서 트래픽을 분산
  • 그러면 왜 웹 서버를 통한 로드 밸런싱을 사용하는지, 그 이유를 비교하며 살펴보자

1. L4 스위치 로드 밸런싱 (전송 계층, 4계층)

  • L4 스위치는 OSI 모델의 4계층(전송 계층)에서 IP 주소와 포트 정보를 기반으로 트래픽을 분산
  • 이 방식은 TCP/UDP 연결 정보를 바탕으로 트래픽을 전달하기 때문에, 애플리케이션의 동작 방식에 대한 이해 없이 순수하게 네트워크 정보만을 가지고 로드 밸런싱을 수행

장점

  • 성능
    • L4 스위치는 네트워크 계층에서 동작하기 때문에 트래픽 처리 속도가 매우 빠름
    • 애플리케이션 계층까지 파악하지 않기 때문에 부하가 적음
  • 프로토콜 독립성: TCP/UDP와 같은 전송 계층 프로토콜만 처리하므로, HTTP 외에도 다양한 프로토콜을 지원
  • 안정성: 네트워크 장비의 특성상 안정적으로 트래픽을 분산할 수 있음

단점

  • 세부적인 트래픽 제어가 불가능
    • L4 스위치는 HTTP 요청의 헤더나 쿠키 등의 세부 정보에 접근할 수 없기 때문에, 애플리케이션 레벨에서 필요한 세밀한 로드 밸런싱(예: 세션 스티키네스, 특정 경로 기반 로드 밸런싱 등)이 불가능
  • 상태 기반 로드 밸런싱 불가능
    • L4 스위치에서는 사용자 세션이나 쿠키 같은 애플리케이션 상태를 인식할 수 없으므로, 세션 기반의 분산 처리가 어려움

2. 웹 서버를 통한 리버스 프록시 로드 밸런싱 (애플리케이션 계층, 7계층)

  • Nginx나 Apache 같은 웹 서버는 리버스 프록시로서 **애플리케이션 계층(7계층)**에서 트래픽을 분산함
  • HTTP 및 HTTPS 요청을 직접 처리하고, HTTP 헤더나 쿠키와 같은 애플리케이션 관련 데이터를 기반으로 로드 밸런싱을 수행할 수 있음

장점

  • 세부적인 트래픽 제어 가능

    • 웹 서버는 HTTP 요청의 세부적인 정보(예: URL 경로, 헤더, 쿠키, 파라미터)를 분석하여, 요청을 세밀하게 분산할 수 있음
    • 이를 통해 경로 기반(path-based) 또는 도메인 기반 로드 밸런싱이 가능합니다.

  • 세션 관리

    • 웹 서버는 세션 스티키네스(Session Stickiness)를 설정하여, 사용자의 요청을 같은 서버로 보내는 방식을 지원
    • 이를 통해 로그인 상태나 장바구니 같은 사용자 상태를 유지할 수 있음

  • SSL/TLS 처리

    • 웹 서버는 SSL/TLS 암호화를 처리할 수 있음
    • 웹 서버가 SSL 종료를 처리하고, 내부로는 HTTP 요청을 전달할 수 있음

  • 다양한 부하 분산 방식 라운드 로빈, 최소 연결 방식, IP 해싱 등 다양한 로드 밸런싱 전략을 적용할 수 있음

단점

  • 성능

    • L4 스위치보다 **처리 속도가 느릴 수 있음
    • 특히 대규모 트래픽 환경에서는 L4 스위치만큼의 성능을 보장하기 어려울 수 있음

  • 복잡성

    • L4 스위치에 비해 웹 서버는 더 많은 구성 요소와 설정을 요구
    • SSL 처리, 상태 관리 등의 기능을 위해 설정이 복잡할 수 있음

3. 웹 서버를 통한 로드 밸런싱을 사용하는 이유

1) 애플리케이션 레벨의 세밀한 제어

  • L4 스위치에서는 네트워크 트래픽만을 기반으로 단순하게 분산하지만, 웹 서버를 통한 리버스 프록시에서는 HTTP 헤더, 쿠키, URL 경로와 같은 정보를 기반으로 세밀한 트래픽 관리가 가능
  • 예를 들어, 웹 서버는 사용자의 세션을 유지하거나, 특정 API 경로에 따라 다른 서버로 트래픽을 분산하는 등의 작업을 수행할 수 있음

2) SSL/TLS 처리

  • 웹 서버는 SSL 종료를 처리할 수 있음
  • L4 스위치는 단순히 트래픽을 분산하지만, Nginx/Apache와 같은 웹 서버는 SSL/TLS 암호화를 처리한 후, 내부 네트워크에서는 HTTP로 통신할 수 있음
  • 이 방식은 보안과 성능을 모두 고려한 설계입니다.

3) 복잡한 로드 밸런싱 전략 지원

  • 웹 서버는 세션 기반 로드 밸런싱, 라운드 로빈이나 최소 연결 방식 등의 다양한 로드 밸런싱 알고리즘을 지원
  • 이를 통해 웹 애플리케이션에서 요구하는 다양한 로드 분산 정책을 구현할 수 있음

4) 유연성

  • 웹 서버는 모니터링 및 로깅 기능도 함께 제공하여, 애플리케이션 상태를 추적하고 트래픽 분산을 유연하게 조정할 수 있음
  • 또한, 새로운 서버 추가 시 더 간단하게 확장성을 제공할 수 있음

4. 결론

  • L4 스위치는 성능과 네트워크 기반의 단순한 트래픽 분산에 강점이 있음
  • 반면에 **리버스 프록시 웹 서버(Nginx, Apache)**는 애플리케이션 계층에서 HTTP 요청을 세밀하게 처리하고, SSL 종료와 세션 관리를 지원할 수 있는 강력한 기능을 제공
  • 따라서, 단순한 네트워크 트래픽 분산이 필요한 경우에는 L4 스위치를, HTTP 요청 기반의 세밀한 제어와 SSL 처리가 필요한 경우에는 리버스 프록시 웹 서버를 사용하는 것이 일반적
