Jenkins - SVM 연동 > Multibranch Pipeline 설정 방법

개요

  • Jenkins Multibranch Pipeline을 사용하여 여러 브랜치의 빌드 및 테스트를 자동으로 관리하는 방법을 설명
  • Multibranch Pipeline은 각 브랜치를 자동으로 탐색하고, 브랜치나 PR이 변경될 때마다 이를 감지하여 빌드를 실행하는 강력한 기능을 제공
  • Jenkins를 활용한 Multibranch Pipeline 설정 방법을 단계별로 설명하며, 여러 브랜치 관리와 빌드 및 테스트를 자동으로 관리하는 방법을 소개

설정 방법

0. 플러그인 설치

  1. Multibranch Scan Webhook Trigger

    • Dashboard > Jenkins Management > Plugins > Available plugins 탭

1. Jenkins에서 Job 생성

  • Dashboard > New Item > Multibranch project 선택

  • 프로젝트 이름 작성 > OK 버튼 클릭

2. 구성(Configuration) > 일반 설정(General)

  • Optional 로 원하는 형태로 기술

3. 구성(Configuration) > Branch Sources

  • Add source > Git

    • Project Repository : Git 주소 작성

    • Behaviors

      • Discover branches : 리파지토리의 브랜츠들을 자동으로 탐색
      • Filter by name (with wildcards)
        • include : 브랜치들 중에서 원하는 브랜치들만 필터링하여 탐색함
        • exclude : 브랜치들 중에서 특정 브랜치를 제외하고 탐색함

    • Credentials : Git 서버 인증

      • Add Credentials

        • 세부 설정사항
          • Scope
            • Global : 이 자격 증명이 시스템 전체에서 사용될 수 있음
            • System : 특정 시스템 또는 프로젝트에서만 이 자격 증명을 사용할 수 있음
          • Username : 대상 서버 계정
          • Password : 대상 서버 계정의 패스워드
          • ID (선택사항) : 자격 증명에 대해 Jenkins에서 사용할 고유 식별자를 입력하는 옵션, 기본적으로 이 필드를 비워두면 Jenkins가 자동으로 ID를 생성

    • 참고사항 : 시스템에 Git이 설치가 되어져 있어야 함, 그렇지 않으면 에러 발생

4. 구성(Configuration) > Build Configuration

  • Multibranch pipelin은 Jenkinsfile로 브랜치를 관리 > 6번 항목에서 상세 기술

5. 구성(Configuration) > Scan Multibranch Pipeline Triggers

  • 브랜치 및 PR 변화를 감지하여 빌드를 트리거하는 방법을 설정하는 옵션

  • Scan by webhook

    • 외부 서비스(GitHub, GitLab 등)에서 변경 사항을 푸시하거나 PR이 발생할 때 자동으로 Jenkins에서 해당 변경을 스캔하고 빌드를 트리거할 수 있도록 설정하는 옵션
    • GitHub/GitLab Webhook을 통해 Jenkins가 즉각적인 변화를 감지
    • Yona 에서는 MR(PR)에 대해서는 즉각적인 변화를 감지하지 못함
    • Trigger token : Webhook에서 Jenkins로 요청을 보낼 때 인증을 위해 사용되는 토큰, 이 토큰을 사용하여 외부에서 Jenkins로 브랜치 스캔을 요청할 수 있으며 보안 목적으로 잘 관리되어야 함

  • Periodically if not otherwise run

    • 이 옵션을 활성화하면 Jenkins가 주기적으로 브랜치를 스캔하여 변경 사항이 있는지 확인
    • 변경 사항이 발견되면 빌드를 트리거
    • Yona의 경우는 즉각적인 MR(PR)이 되지 않기 때문에 해당 옵션을 활성화하여 주기적으로 변경 감지를 수행

결론

  • Jenkins Multibranch Pipeline은 대규모 프로젝트에서 여러 브랜치를 효과적으로 관리하는 데 매우 유용
  • 각 브랜치마다 별도의 파이프라인을 설정하지 않아도 Jenkins가 자동으로 브랜치를 탐색하고 빌드를 실행할 수 있음
  • 특히 SVM 연동을 통해 자동화된 Webhook 설정으로 브랜치 변경 사항을 즉시 감지하고, 빠르고 효율적으로 빌드와 배포를 진행할 수 있음
  • Jenkins의 Multibranch Pipeline을 활용하면 복잡한 브랜치 전략을 더 쉽게 관리할 수 있으며, CI/CD 파이프라인을 통해 개발 속도를 높이고, 협업 효율성 또한 크게 향상시킬 수 있음

Jenkins Multi-branch Pipeline 설정 가이드

각 Git 브랜치 (feature, develop, stage, master)에 대해 Jenkins에서 수행해야 하는 작업들을 자세히 설명합니다.

1. Feature 브랜치 파이프라인 설정 (Multi-branch Pipeline)

목적: 개별 기능 개발 후 코드 리뷰 및 유닛 테스트를 통해 코드 품질을 검증한 후 develop 브랜치로 병합합니다.

설명:

  • Checkout: feature 브랜치를 Yona에서 체크아웃하여 Jenkins가 해당 브랜치를 기반으로 빌드하고 테스트를 진행할 수 있도록 합니다.
  • Build & Unit Test: mvn clean install로 빌드하고, mvn test로 유닛 테스트를 실행합니다.
  • Code Coverage: Jacoco 플러그인을 사용해 코드 커버리지를 측정합니다.
  • Post Actions: 빌드 성공/실패 결과를 Slack으로 전송해 개발자에게 알립니다.

MR(Merge Request) 연동:

  • Yona에서 Merge Request(MR)를 생성하면 Jenkins가 이를 감지하고 빌드를 자동으로 실행하도록 Webhook을 설정합니다.
  • MR이 승인되면 Jenkins에서 자동으로 develop 브랜치로 병합합니다.

2. Develop 브랜치 파이프라인 설정

목적: 여러 feature 브랜치가 병합된 후 통합 테스트 및 전체 코드 검증을 수행합니다.

설명:

  • Integration Test: 여러 모듈이 정상적으로 상호작용하는지 확인하는 통합 테스트를 수행합니다.
  • Post Actions: 빌드 실패 시 Slack으로 알림을 전송해 문제를 빠르게 해결할 수 있도록 합니다.

3. Stage 브랜치 파이프라인 설정

목적: stage 환경에서 통합 테스트와 성능 테스트를 실행하여 최종 배포 준비 상태를 확인합니다.

설명:

  • Performance Test: 부하 테스트 및 성능 테스트를 통해 배포 전 시스템이 안정적으로 작동하는지 확인합니다.
  • Post Actions: 빌드 실패 시 빠르게 대응할 수 있도록 Slack으로 알림을 전송합니다.

4. Master 브랜치 파이프라인 설정 (프로덕션 배포)

목적: master 브랜치에 병합된 코드를 프로덕션 서버에 배포합니다.

설명:

  • Deploy to Production: 배포 스크립트를 통해 프로덕션 서버에 배포를 실행합니다.
  • Post Actions: 배포 성공/실패에 대한 알림을 Slack으로 전송합니다.

빌드 실패 시 처리 방법

  1. Feature 브랜치 빌드 실패

    :

    • Jenkins는 실패를 감지하고 Slack으로 알림을 보냅니다.
    • 개발자는 실패 원인을 확인한 후 수정 작업을 진행하고, 브랜치를 다시 푸시하여 빌드를 재실행합니다.

  • 빌드 실패 시: 실패한 원인을 확인한 후 개발자는 로컬에서 문제를 수정하고, 다시 푸시합니다.

빌드 실패 시 처리 방식에 대한 설명에서 제가 제공한 내용은 기본적인 절차이지만, 실제 전문적인 DevOps 팀에서는 다음과 같은 보다 체계적이고 자동화된 방식으로 빌드 실패를 처리합니다. 여기서는 더 효율적이고 안정적인 실패 처리 프로세스를 살펴보고, 자동화 도구와 전략을 사용해 신속하게 대응하는 방법을 설명드리겠습니다.

전문적인 DevOps 팀의 빌드 실패 처리 방식

1. 자동화된 오류 감지 및 보고

  • 실시간 모니터링: 빌드 실패 시 Slack, 이메일, Microsoft Teams 등을 통해 실시간으로 개발자와 DevOps 팀에게 자동 알림을 보냅니다. 실패한 빌드 로그와 원인을 포함하여 즉각적인 대응이 가능합니다.

  • Jenkins 플러그인

    : Jenkins와 같은 CI/CD 도구는 여러 플러그인을 사용해 빌드 실패 로그를 상세하게 기록하고 분석할 수 있습니다. 이를 통해 문제의 원인을 신속히 파악할 수 있습니다.

    • 예시: Jenkins Email Extension Plugin을 사용해 빌드 실패 시 이메일로 상세 로그를 전송.

2. 자동 롤백 및 재시도

  • 자동화된 롤백

    : 전문적인 DevOps 팀은 배포 실패 시

    자동 롤백

    절차를 구축해 놓습니다. 이는 특히

    Master

    브랜치에서 배포 실패 시, 시스템을 원상 복구하고 문제를 해결한 후 다시 배포하는 과정이 자동으로 이루어집니다.

    • 예시: Kubernetes를 사용하는 경우, Helm을 통해 자동 롤백을 수행하거나, Ansible과 같은 도구를 사용해 스크립트를 통해 이전 버전으로 복구합니다.

  • 자동 재시도

    : 실패한 빌드는 일반적으로 일정 횟수만큼 자동으로 재시도합니다. Jenkins는 기본적으로 빌드 실패 시

    재시도 설정

    을 지원하며, 이는 일시적인 네트워크 문제나 기타 일시적인 이슈를 해결하는 데 유용합니다.

    • 예시: Jenkins에서 retry 블록을 사용해 빌드를 자동으로 재시도할 수 있습니다.

3. Root Cause Analysis (RCA)

  • 빌드 실패 원인 분석

    : DevOps 팀은 빌드 실패가 빈번하게 발생하면 **Root Cause Analysis (RCA)**를 통해 근본적인 원인을 분석합니다. 이를 통해 빌드 프로세스나 코드에서 발생하는 지속적인 문제를 해결하여 시스템을 개선할 수 있습니다.

    • 예시: 특정 단계에서 반복적인 문제가 발생하는 경우, 빌드 스크립트를 최적화하거나 CI 도구에서 해당 단계를 개선하는 조치가 이루어집니다.

4. 블루/그린 배포 또는 카나리 배포 전략 사용

  • 블루/그린 배포: DevOps 팀은 빌드가 성공하더라도 운영 환경에서 문제가 발생할 수 있기 때문에, 블루/그린 배포 전략을 사용해 트래픽을 새로운 버전과 이전 버전 사이에서 스위칭하면서 문제가 없는지 확인합니다.
  • 카나리 배포: 새로운 버전을 모든 사용자에게 배포하지 않고, 일부 사용자에게만 점진적으로 배포해 문제가 발생하면 즉시 롤백하는 방식입니다. 이 방법은 서비스의 안정성을 유지하면서 새로운 기능을 배포하는 데 매우 유용합니다.

5. Pull Request 기반 빌드 검증 (Pre-merge Checks)

  • PR 빌드 검증

    : 전문적인 DevOps 팀은

    브랜치에서 빌드를 실패시키지 않기 위해,

    Pull Request(PR) 빌드 검증

    을 사용합니다. MR/PR이 merge되기 전에 테스트 및 빌드를 사전 검증하여 빌드 실패를 방지합니다.

    • 예시: GitLab의 CI/CD 파이프라인에서 MR이 발생할 때 자동으로 빌드 및 테스트를 실행하여 merge 여부를 결정합니다.

6. 빌드 실패의 책임 분담 (소유자에게 알림)

  • 빌드 소유자 알림

    : DevOps 팀은 빌드 실패 시, 해당 빌드에 책임이 있는 개발자나 팀에게 직접 알림을 보냅니다. 이를 통해 책임 분담이 명확해지고, 더 빠른 대응이 가능합니다.

    • 예시: Git의 blame 기능이나 Jenkins의 GIT_COMMITTER_NAME 변수를 통해 변경한 개발자에게 직접 알림을 보낼 수 있습니다.

DevOps 팀에서 각 브랜치별 빌드 실패 처리 방식 개선

Feature 브랜치 빌드 실패 시 (개발 환경)

  • 빌드 실패 자동 알림: Slack을 통해 개발자에게 실시간 알림을 보내며, 빌드 실패의 구체적인 로그와 실패 원인까지 포함합니다.
  • 자동 재시도: 간헐적인 네트워크 문제 또는 일시적인 장애를 고려하여 3회 정도 자동으로 재시도합니다.
  • PR/MR 자동 검증: Pull Request/Merge Request 시점에서 자동으로 빌드를 검증해, 문제가 있을 경우 merge되지 않도록 방지합니다.

Develop 브랜치 빌드 실패 시 (통합 테스트 실패)

  • 통합 테스트 로깅: 통합 테스트가 실패한 경우, 자동으로 로그를 분석하여 상세한 실패 원인을 Slack 및 이메일로 관련 개발자에게 전달합니다.
  • 자동화된 이슈 생성: 빌드 실패가 발생할 경우, Yona나 GitLab과 연동하여 자동으로 이슈 트래킹 시스템에 이슈가 생성됩니다. 이를 통해 누락되는 문제 없이 추적 가능합니다.

Stage 브랜치 빌드 실패 시 (성능 테스트 실패)

  • 성능 데이터 분석: 성능 테스트 실패 시 자동으로 성능 데이터를 분석하고, 차트나 그래프 형태로 Slack 또는 이슈 트래킹 시스템에 보고합니다. 이를 통해 성능 문제를 시각적으로 빠르게 파악할 수 있습니다.
  • 성능 테스트 재시도: 성능 테스트에 실패한 경우에도 일시적인 이슈일 수 있으므로 자동 재시도 기능을 활용합니다.

Master 브랜치 배포 실패 시 (프로덕션 환경)

  • 자동 롤백: 배포에 실패하면 자동으로 이전 버전으로 롤백하며, 시스템 가동에 문제가 없도록 합니다. 이는 Ansible, Helm, Terraform과 같은 도구를 통해 수행됩니다.
  • 배포 이력 관리: 모든 배포 이력은 자동으로 저장되며, 이를 기반으로 특정 버전에서의 문제를 추적할 수 있습니다.
  • 카나리 배포 전략: 모든 사용자에게 새로운 버전을 배포하기 전에, 일부 사용자에게만 배포하여 문제가 없는지 확인합니다.

결론

전문적인 DevOps 팀에서는 빌드 실패 시 더 자동화되고, 빠른 대응이 가능한 시스템을 구축하여 팀의 생산성을 높입니다. 기본적인 실패 대응 방법 외에도, 자동 롤백, 재시도, 성능 데이터 분석, 블루/그린 배포 전략, PR/MR 자동 검증 등의 고급 DevOps 전략을 통해 더욱 안정적이고 신속하게 문제를 해결합니다. 이를 통해 빌드와 배포에서 발생하는 장애를 최소화하고, 코드 품질을 유지할 수 있습니다.
